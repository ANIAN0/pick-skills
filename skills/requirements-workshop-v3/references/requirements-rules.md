# 需求细则

## 文档和支撑材料

支撑材料放在入口旁的同名目录中，按需创建，不预建空目录或空文件。

```text
workplace/3/requirements/2026-06-21-order-import.md
workplace/3/requirements/2026-06-21-order-import/
├── research/RT-001-duplicate-handling.md
├── evidence/EVD-001-current-flow.md
├── decisions/DEC-001-duplicate-strategy.md
├── ui/UI-001-import-page.md
└── review/REV-001-requirements-review.md
```

编号使用类型加递增数字：`RT-*` 调研、`EVD-*` 证据汇总、`DEC-*` 需求决策、`UI-*` UI 需求、`REV-*` 审查。项目证据能直接链接真实代码或已有文档时，不复制证据文件；只有多个来源需要汇总结论时才创建 `EVD-*`。

只有需求决策同时满足以下条件才创建 `DEC-*`：以后改变代价明显；只看最终结论无法理解原因；讨论中存在真实取舍。普通结论直接压缩进入口；架构或实现取舍留给 `tech-design-v3`。

## 推导深度

简单改动只做足以排除歧义和遗漏的分析。中大型需求先用业务语言说清核心机制：关键对象如何流转，谁触发谁，哪些规则决定结果，哪些失败影响主流程。

以下任一情况视为中大型需求：新模块或子系统、跨模块协作、事实源变化、持久化数据或权限变化、AI/Agent、异步流程、兼容或迁移、三条以上用户路径。

功能点推导时检查：

- 是否直接服务用户故事，是否遗漏完成目标所需能力。
- 功能之间是否重复、互相依赖或边界冲突。
- 用户可见功能是否同时考虑界面和后台行为。
- 本期做和明确不做是否已达成一致。
- AI/Agent 的任务、角色边界、上下文、工具、自主范围和错误后果是否清楚。
- 用户强调的限制、风险和偏好是否进入功能点或验收要求。

## 用户故事、功能点和验收

用户故事使用四段式，逐项分行列出：作为 `[谁]`，当 `[场景]` 时，我需要 `[功能]`，以便 `[目的]`。

- 谁：具体角色，包含关键属性。
- 场景：触发条件或上下文。
- 功能：动词驱动，不写实现细节。
- 目的：业务价值或用户可见结果。

功能点只写用户可理解的能力和结果，不写模块设计、API、数据库、文件路径或实现步骤。验收要求优先使用最高层可观察行为，只有最高层证据不足以证明关键风险时才补充较低层要求。

## UI 需求

需求阶段不生成原型。涉及 UI 时创建或链接 `ui/UI-*`，写清页面、状态、交互、数据、失败反馈、可访问性和用户确认结论。入口只保留摘要和链接；无 UI 时写明“不涉及 UI”。

## 按需调研

缺少事实会导致用户故事、功能点或验收要求无法确定时，在受影响内容下登记 `RT-*`，写清调研问题、范围和完成条件，然后调用 `project-research-v3`。调研报告链接回 `RT-*`，入口链接调研报告；结论未回写或仍 blocked 时不得确认需求。

## 回退

下游证明已确认需求需要改变时，把原入口 `status` 改回 `discussing`、`review` 重置为 `pending`，并说明哪些方案、计划、执行和验收确认失效。修改后重新自检、审查和用户确认，不能复用原确认。
