# 需求工作坊 v3

需求阶段可以经过多轮对话。每轮只吃透当前最重要的问题，结论形成后立即更新同一份需求入口文档，不要求一次完成全文。

未完成独立审查和用户确认前，不进入技术方案、任务或实现阶段。

## 第0步：确认文档路径（必须最先执行）

开始任何工作前，执行以下步骤：

1. 读取项目根目录，找到 `workplace/` 目录
2. 确认当前活跃版本号（必须是整数或两段数字，如 `3` 或 `3.1`）；无法唯一确认时询问用户
3. 检查当前事项是否已有需求入口文档：
   - 用户给出文档路径时，使用该文件
   - `workplace/<版本>/requirements/` 下已有相关文件时，继续更新原文件
   - 两者都没有时，从用户描述的功能名称推导文件名，格式为 `YYYY-MM-DD-功能名.md`
4. 向用户输出将要使用的完整路径，例如：
   `workplace/3/requirements/2026-06-23-order-import.md`
   说明"如需修改路径请告知，否则直接继续"，然后继续执行，不等待用户回复

**硬性禁止：**
- 禁止在 `workplace/` 以外创建需求文档
- 禁止将需求、方案、任务内容写入同一文件
- 禁止在路径中使用 `<version>`、`x.x` 等占位符

## 文档

一个研发事项只能有一份需求入口文档，路径由第0步确认。

创建新文档时使用 [需求入口模板](assets/requirements-entry.md)。入口文档只保留用户会检查的四项内容：用户故事、功能点、验收要求、原型。

完整调研、项目证据、讨论记录、UI 需求和审查报告保存为支撑材料。入口文档只写结论和链接，支撑材料必须链接回入口文档对应内容。

## 支撑材料

支撑材料放在入口文档旁边的同名目录中。例如入口文档是：

```text
workplace/3/requirements/2026-06-21-order-import.md
```

对应支撑目录是：

```text
workplace/3/requirements/2026-06-21-order-import/
├── research/RT-001-duplicate-handling.md
├── evidence/EVD-001-current-import-flow.md
├── decisions/DEC-001-duplicate-strategy.md
├── ui/UI-001-import-page.md
├── prototype/PROTOTYPE-001-import-page.html
└── review/REV-001-requirements-review.md
```

命名使用“类型编号 + 简短英文 slug”：调研 `RT-*`、证据汇总 `EVD-*`、讨论形成的决策 `DEC-*`、UI 需求 `UI-*`、原型 `PROTOTYPE-*`、审查 `REV-*`。同一类型从 `001` 递增。

不预建空目录或空文档，实际产生材料时再创建。项目证据能直接链接真实代码或已有文档时，不复制证据文件；只有多个来源需要汇总结论时才创建 `evidence/EVD-*`。

不要为每次讨论都创建 `DEC-*`。只有一个需求决策同时满足以下条件时才创建：以后改变代价明显；只看最终结论无法理解原因；讨论中存在真实取舍。普通结论直接压缩进入口文档；架构或实现取舍留给 `tech-design-v3`。

入口文档在相关用户故事、功能点、验收要求或原型处链接支撑材料。每份 Markdown 支撑材料在开头链接回入口文档的对应标题或编号；非 Markdown 原型必须在页面内或自身元数据中提供需求入口链接。

入口文档只有两个状态：

- `discussing`：需求仍在讨论、调研、制作原型或审查。
- `confirmed`：审查通过，并且用户明确确认四项内容。

## 逐项追问

沿需求的决策依赖逐项追问，直到双方理解一致。一次只问一个问题，并给出推荐答案及理由。答案会改变后续问题时，先等待用户回答，不提前展开后续分支。

能从项目、已有文档或可靠来源确认的事实先自行调查，不把调查工作推给用户。发现用户描述与项目事实冲突时，明确指出冲突并询问以哪一个为准。

不要把“支持相关能力”“完善流程”“合理处理”“后续再看”“视情况”等表述写成结论。把它们转换成具体角色、场景、行为和结果继续讨论。

用户使用模糊、重载或与项目既有语言冲突的术语时，立即指出差异并提出明确的统一称呼。讨论业务对象、归属、状态或关系时，主动构造具体正常、边界和失败场景，借场景暴露尚未说清的规则，不停留在抽象表述。

每解决一个问题，立即更新入口文档或相关支撑材料。已经确认且未被新证据推翻的内容不重复询问。

## 提问门槛

只有同时满足以下条件才向用户提问：

- 答案会实质改变用户故事、功能点、验收要求、原型或本期范围。
- 无法从项目、已有文档、历史对话或调研中获得答案。
- 错误假设会导致明显返工、范围错误、用户可见问题或风险。

不满足时不要提问：能调查就调查；能安全推断时，在受影响内容下写明假设、依据和影响，再继续推进。

需要提问时，把问题放在回复最后的 `## 需要你确认` 区块。区块只包含一个问题、推荐答案和不同选择的影响，后面不再追加总结、说明或下一步。

每轮开始先综合已有对话、入口文档和项目事实，能直接形成草稿就先更新文档。写作过程中不重新访谈已经解决的问题，只对通过上述门槛的真实缺口提问。

## 深入推导

用户最终只检查四项，但 Agent 在形成四项之前必须完成足够的需求推导，不能只整理用户原话。

先检查与当前需求直接相关的项目文档、代码、页面、接口、数据、配置和相似功能，确认已有能力、缺口、可复用内容和冲突。项目存在 glossary、`CONTEXT.md`、上下文映射或相关 ADR 时，读取与当前需求有关的部分，沿用已确认业务术语并把 ADR 作为现有约束；需求阶段不新建或改写技术 ADR。

讨论用户故事前，必须理解：

- 谁在什么场景遇到什么问题。
- 现有做法为什么不够，不做会有什么后果。
- 完成后用户或系统得到什么结果。

中大型需求在列功能点前，必须用业务语言说清核心机制：哪些关键对象或信息在流转，谁触发谁，哪些规则决定结果，哪些失败会影响主流程。以下任一情况视为中大型需求：新模块或子系统、跨模块协作、事实源变化、持久化数据或权限变化、AI/Agent、异步流程、兼容或迁移、三条以上用户路径。

推导功能点时检查：

- 每个功能是否直接服务用户故事，是否遗漏完成目标所需的关键能力。
- 功能之间是否重复、互相依赖或存在顺序与边界冲突。
- 用户可见功能是否同时考虑界面和后台行为，不能只留下其中一侧。
- 本期功能和明确不做内容是否已经与用户达成一致。
- 涉及 AI 时，任务、角色边界、所需上下文或工具、可自主决定的范围和错误后果是否清楚。
- 用户明确强调的限制、风险和偏好是否进入相关功能点或验收要求，没有在压缩文档时丢失。

这些分析不要求全部写进入口文档。复杂论证和证据放入双向链接的支撑材料，最终结论压缩为一至两句话的功能点和对应验收要求。

简单改动只做足以排除歧义和遗漏的分析，不为了流程完整感制造额外问题或文档。

## 形成四项内容

先把用户故事讨论准确。Agent 应能用自己的话复述角色、问题和目标结果，并得到用户认可。

再形成完整功能点。每个功能点只用一至两句话说明它是什么、做什么；不写模块设计、API、数据库、文件路径或实现步骤。

然后为每个功能点形成验收要求。优先从现有用户入口或公开接口等最高层可观察行为验收，避免把验收拆成内部模块检查；只有最高层证据不足以证明关键风险时才补充较低层要求。验收要求只描述用户如何确认该功能正确完成，并覆盖真正影响判断的关键失败或边界，不堆砌通用质量清单。

涉及 UI 时，优先使用当前环境已经可用的 UI 需求或原型 skill；能力未安装或不可调用时，直接在 `ui/UI-*` 中写清页面、状态、交互、数据、失败反馈和可访问性要求，并在 `prototype/PROTOTYPE-*` 中创建可审查的代码原型。不能把不存在或尚不可用的 skill 当作前置条件。入口文档只保留原型说明、链接和用户确认结论。无 UI 时写明“不涉及 UI”。

## 按需调研

如果缺少事实会导致用户故事、功能点或验收要求无法确定：

1. 在受影响内容下面登记 `RT-*`，写清调研问题、范围和完成条件。
2. 请用户审核调研任务。
3. 审核通过后调用 `project-research-v3`。
4. 调研报告链接回 `RT-*`，入口文档链接调研报告。
5. 根据报告更新原内容，再继续逐项追问。

调研未完成、结论未回写或用户仍有疑问时，不关闭 `RT-*`。

## 审查与确认

下游阶段回退并证明已确认需求需要改变时，先把原入口 `status` 改回 `discussing`、把 `review` 重置为 `pending`，并明确使相关方案、计划、执行和验收确认失效。修改四项内容后重新完成独立审查和用户确认，原确认不能复用。

四项内容形成后，先逐项自检；发现问题直接修正，直到自检通过，再调用 `project-development-review-v3` 独立审查。审查应检查项目事实是否正确、核心机制是否理解、功能是否遗漏或越界、验收是否足以证明用户故事、用户强调的约束是否保留、UI 文档和原型是否一致。发现问题时继续修改同一份入口文档并重新自检、复审。

审查通过后，请用户依次确认用户故事、功能点、验收要求和原型。用户对任何一项有疑问时继续讨论；全部确认后把状态改为 `confirmed`，再交给 `tech-design-v3`。

## 完成条件

- 用户故事准确，Agent 与用户能够一致复述。
- 功能点完整，每项均用一至两句话说清是什么、做什么。
- 每个功能点都有用户认可的验收要求。
- 涉及 UI 时，原型已生成、已链接并经用户确认；无 UI 时已明确声明。
- 所有调研任务已完成，报告与入口文档双向链接，结论已经回写。
- 独立审查通过。
- 用户明确确认四项内容。

向用户报告需求阶段完成前，必须重新对照本节自检。任何一项不成立都先修正，不把检查工作留给用户发现。

未满足任何一项时保持 `discussing`。

## 边界

- 不设计架构、数据库、API 或代码修改方案。
- 不为单个用户故事、功能点、验收要求或调研任务创建入口文档。
- 不把支撑材料正文复制进入口文档。
- 不自动替用户确认需求完成。
