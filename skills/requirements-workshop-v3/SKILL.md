# 需求工作坊 v3

目标是形成足够清楚、足够短的需求入口：用户故事、功能点、验收要求；涉及 UI 时补充 UI 需求链接。不要进入技术方案、任务或实现阶段。

## 默认流程

1. 确认入口路径：只在 `workplace/<版本>/requirements/` 下创建或更新一份需求入口。优先使用用户给出的路径，其次复用已有相关入口，否则按 `YYYY-MM-DD-功能名.md` 命名。输出完整路径并说明“如需修改请告知，否则继续”，然后继续执行。
2. 读取上下文：查看相关项目文档、代码、页面、接口、数据、配置、glossary、`CONTEXT.md`、ADR 和相似功能。能查到的事实自己查，不问用户。
3. 写入或更新入口：新建时使用 [需求入口模板](assets/requirements-entry.md)。入口只写结论和必要链接；调研、证据、决策、UI 细节和审查报告放到同名支撑目录。
4. 讨论缺口：只问会改变范围、用户故事、功能点、验收要求或 UI 需求的问题。相关问题一次成组提问，通常 2-5 个；每组都给推荐答案和影响。不要为了流程完整感拆成多轮单问。
5. 形成三项核心内容：用户故事讲清“谁 / 何时 / 要什么 / 为什么”；功能点一至两句话说明用户可见能力；验收要求用可观察行为覆盖关键边界和失败。
6. 自检、独立审查、用户确认：自检通过后调用 `project-development-review-v3`。审查通过后请用户确认三项核心内容和 UI 需求（如有），全部确认后把 `status` 改为 `confirmed`。

## 提问规则

满足全部条件才问用户：

- 答案会实质改变需求结论或本期范围。
- 无法从项目、已有文档、历史对话或调研获得答案。
- 错误假设会导致明显返工、范围错误、用户可见问题或风险。

提问时放在回复最后的 `## 需要你确认`，后面不再追加总结。能安全假设时，直接写明假设、依据和影响并继续推进。

## 下沉材料

按需读取 [需求细则](references/requirements-rules.md)：

- 需要创建支撑材料、调研项、证据汇总、需求决策或 UI 文档时。
- 需求中等以上复杂，涉及跨模块、权限、持久化、异步、兼容、迁移、AI/Agent 或三条以上用户路径时。
- 需要判断用户故事、功能点、验收要求是否写到位时。
- 下游回退导致已确认需求需要重开时。

## 完成条件

- 入口路径真实，位于 `workplace/`，且一个事项只有一份需求入口。
- 用户故事、功能点、验收要求完整、简洁、互相对应。
- 涉及 UI 时已有 UI 需求文档或入口中的 UI 需求链接；不生成 `prototype/PROTOTYPE-*`。
- 必要调研已完成并回写，支撑材料与入口双向链接。
- 独立审查通过，用户明确确认需求内容。

任一项不成立时保持 `status: discussing`。

## 边界

- 不设计架构、数据库、API 或代码修改方案。
- 不把支撑材料正文复制进入口。
- 不自动替用户确认需求完成。
