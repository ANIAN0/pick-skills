# 任务规划细则

## 文档结构

按需创建支撑目录，不预建空文件。

```text
workplace/3/implementation-planning/2026-06-21-order-import.md
workplace/3/implementation-planning/2026-06-21-order-import/
├── tasks/T-001-import-happy-path.md
├── tasks/T-002-duplicate-handling.md
├── evidence/EVD-001-test-baseline.md
├── research/RT-001-unknown-constraint.md
└── review/REV-001-plan-review.md
```

入口链接每个 `T-*`，每个任务文件链接回入口及对应方案内容。`T-*`、`EVD-*`、`RT-*`、`REV-*` 编号一旦被引用，不得重排或复用；删除、合并或替换时保留历史说明并让新编号递增。

## 缺口分流

不能在任务文档中自行补设计：

- 方案契约缺口返回 `tech-design-v3`：真实变更位置、接口字段、状态、失败行为、数据归属、验证定义或技术决策不清。
- 需求语义缺口返回 `requirements-workshop-v3`：用户目标、验收边界、特别关注点或不做范围不清。
- 执行事实缺口留在规划阶段自修或登记 `RT-*`：测试入口、fixture、命令、相似模式、既有失败或任务顺序需要核验。

开放问题会影响任务边界、接口契约、字段、状态、失败行为、验证方式、依赖顺序或最终闭环时，不得创建对应 `T-*` 的详细任务契约。

## 覆盖审计

在支撑材料或规划工作区逐项核对，不把完整矩阵堆进入口：

- `F-*`、验收要求、`D-*`、`C-*`、`V-*` 是否全部进入任务。
- 覆盖完整性：所有追踪项有任务落点。
- 可执行完整性：真实路径、测试入口、命令、fixture、失败判定、执行断言、闭环和证据要求明确。
- 服务边界、API、数据实体、迁移、配置、页面、状态、错误反馈和运行链路是否有落点。
- UI 需求是否有前端任务；无 UI 时是否明确。
- 阻塞审查发现是否已关闭，调研结论是否已吸收。

入口只展示总数、覆盖完整性、可执行完整性、缺口和处理方式。

## 测试先行策略

把验收要求和 `V-*` 转成按价值和风险排序的可观察行为。优先选择现有用户入口、公共 API 或稳定契约；优先真实代码路径的集成测试；纯逻辑用单元测试；页面和关键路径用组件、浏览器或 smoke。

mock 只允许在第三方 API、时间、随机性、文件系统或无法提供测试实例的外部边界；不 mock 自有模块。第一个功能任务优先使用最窄但完整的 tracer bullet。

规划阶段必须尽量运行相关且不修改项目状态的现有基线和命令试验。既有失败只有在使 RED/GREEN、验证命令、最终闭环或任务边界不可判定时才阻塞；未阻塞的既有失败必须在任务中写明如何区分。

## 任务拆分

任务按最小可验证纵向价值拆分。一个切片贯穿该行为实际需要的层，完成后可以独立演示和验证。任务太大或包含多个不能独立演示的行为时拆分；拆分后只能验证内部结构或无法产生完整结果时合并。

只有真实依赖要求时才提前创建迁移、共享契约、基础设施或行为保持的 prefactor 任务。禁止把同一行为的测试、实现、回归或断言拆成相互独立的横向任务。

入口表格顺序就是默认执行顺序。可并行或可乱序必须在入口明确标注；任何改变风险收敛、tracer bullet 或闭环责任的重排都必须返回规划阶段。

## T-* 契约

每个 `T-*` 必须包含：

- 追踪和目标：关联 `F/D/C/V`、纵向交付行为、独立演示结果。
- 上下文和边界：当前状态、目标状态、相关 `D-*` 摘要、特别关注点、不做范围、禁止捷径；BUGFIX 写根因、风险和回退。
- 文件和契约：测试与实现文件、上游依赖、本任务提供、下游消费、相邻接口、字段、事件、状态和错误。
- 测试设计：公共测试入口、mock 边界、场景的输入/动作/预期、类型、资产路径和具体测试或检查内容。
- 实施步骤：一个或多个有序的测试/验证先行循环。
- 执行断言：必须消失、必须出现、必须传递、必须落盘，以及检查方式和失败判定。
- 完成和证据要求：可观察结果、独立演示、完成证据、相关 `V-*`、回归范围和执行阶段必须留下的证据。

任务文件不能用“参见入口”“参见其他任务”省略执行所需信息。可以链接方案和证据作为来源，但必须内联当前任务需要的实际契约和约束。

## 测试或验证先行循环

业务行为使用 `RED -> GREEN -> REFACTOR`：

- RED：一个可观察行为、测试资产路径、具体测试/断言内容、精确命令和因为行为缺失而失败的预期。
- GREEN：通过当前 RED 所需的最小文件、接口、字段、状态或交互变更，以及同一命令的预期通过结果。
- REFACTOR：只在 GREEN 后允许的整理范围和回归命令；没有必要时写“无”。

迁移、配置、构建和基础设施任务使用“失败检查 -> 最小变更 -> 通过检查 -> 整理回归”。一个任务可包含多个循环，但顺序必须是 `RED1 -> GREEN1 -> RED2 -> GREEN2`，禁止批量先写所有测试再实现。

## 执行断言和闭环

从每个任务的方案变更和步骤提取：

- 必须消失：旧文件、调用、字段、分支、fallback 或行为及残留搜索。
- 必须出现：新文件、函数、参数、日志、事件、配置或测试资产。
- 必须传递：字段、标识、状态或 metadata 经过的完整调用链。
- 必须落盘：测试、fixture、review 清单、smoke、迁移和文档的精确路径。

每条断言写检查方式和失败判定，不能用 build、lint 或类型检查替代行为与链路证据。闭环必须证明真实入口、关键链路、可回查结果、失败行为和无模拟完成。

## 回退

执行或验收回退并证明规划契约需要修改时，把原入口 `status` 改回 `discussing`，把 `plan_review`、`execution_status` 和 `execution_review` 重置为 `pending`，并将受影响任务恢复为 `pending`。修订后重新完成计划审查和用户确认。
