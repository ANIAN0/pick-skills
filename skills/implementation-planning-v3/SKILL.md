# 实施任务规划 v3

任务阶段产生两层文档：用户只审查简洁入口，执行者按完整任务文档实施。不能把实现细节塞进入口，也不能为了入口简洁而省略执行契约。

## 文档结构

一个研发事项只有一份任务入口文档。用户给出路径时使用该文件；当前事项已有入口时继续更新；都没有时询问保存位置，不猜测版本目录。

创建新产物时使用 [任务入口模板](assets/planning-entry.md) 和 [详细任务模板](assets/task-detail.md)：

```text
workplace/3/implementation-planning/2026-06-21-order-import.md
└── 2026-06-21-order-import/
    ├── tasks/T-001-import-happy-path.md
    ├── tasks/T-002-duplicate-handling.md
    ├── evidence/EVD-001-test-baseline.md
    ├── research/RT-001-unknown-constraint.md
    └── review/REV-001-plan-review.md
```

按需创建目录和材料，不预建空文件。入口链接每个 `T-*`，每个任务文件链接回入口及对应方案内容。

入口状态只使用：

- `discussing`：仍在覆盖审计、测试设计、任务拆分、修改或审查。
- `confirmed`：详细任务全部完成、独立审查通过且用户确认入口摘要。

## 两层职责

任务入口只展示用户判断计划是否可靠所需的信息：

- 交付范围和覆盖审计摘要。
- 测试先行策略摘要。
- 任务编号、纵向交付结果、覆盖项、依赖、顺序和详情链接。
- 最终闭环、主要风险、阻塞和审查结论。

入口不展示文件清单、接口字段、实现步骤、测试代码、精确命令、RED/GREEN/REFACTOR 细节或执行断言。

每个 `T-*` 任务文件必须完全自含，使执行者不读取历史对话、不重新设计方案也能完成任务。规划阶段写入不可擅自改动的实施契约，并初始化空的执行状态和执行记录；`plan-execution-v3` 直接在同一任务文件的专用执行区写入实际结果，不能改写规划契约。

## 输入和交接契约

只接受状态为 `confirmed` 的技术方案入口。读取已确认需求、方案、相关调研和审查报告，完整提取：

- `F-*` 及其验收要求：任务最终要交付和证明的行为。
- `D-*`：所有任务必须遵守的技术决策、取舍和边界。
- `C-*`：分类、类型、真实位置、当前状态、目标修改及关联 `F/V`。
- `V-*`：需要证明的行为、验证方式和可观察证据。
- 用户特别关注的限制、风险、兼容、UI 和不做范围。

建立覆盖审计后才能拆任务。每个 `C-*` 必须有实现落点，每个 `V-*` 必须有测试或最终验证落点，每个 `D-*` 必须被相关任务继承。方案缺少真实位置、接口、字段、状态、失败行为或验证定义时返回 `tech-design-v3`，不能在任务文档中自行补设计。

## 项目和测试基线

检查方案涉及的真实代码、目录、接口、数据、页面、配置、测试框架、fixtures、运行命令、glossary、`CONTEXT.md` 和相关 ADR。确认：

- 方案位置存在或明确为新增，路径不含版本占位符。
- 现有模块责任、接口和项目惯例与方案一致。
- 可复用的公共测试入口、测试资产和相似任务模式。
- 相关基线命令的当前结果及既有失败。

规划阶段只运行不修改项目状态的现有基线和命令试验；详细输出放 `evidence/EVD-*`，入口只写结论。事实不清且会改变任务边界或验证时，在入口受影响内容下登记 `RT-*`，用户审核后调用 `project-research-v3`。

## 覆盖审计

在支撑材料或规划工作区逐项核对，不把完整矩阵堆进入口：

- `F-*`、验收要求、`D-*`、`C-*`、`V-*` 是否全部进入任务。
- 服务边界、API、数据实体、迁移、配置、页面、状态、错误反馈和运行链路是否有落点。
- UI 需求是否有前端任务；无 UI 时是否明确。
- 跨服务、前后端、异步、日志、历史、外部工具或模型调用是否有最终闭环。
- 阻塞审查发现是否已关闭，调研结论是否已吸收。

入口只展示各类总数、覆盖结果、缺口和处理方式。存在未解决缺口时保持 `discussing`，不生成看似完整的粗任务。

## 测试先行策略

先把验收要求和 `V-*` 转成按价值和风险排序的可观察行为：

- 优先选择现有用户入口、公共 API 或稳定契约，测试外部行为而非内部实现。
- 优先真实代码路径的集成测试；纯逻辑使用单元测试，页面和关键用户路径使用组件、浏览器或 smoke。
- mock 只允许在第三方 API、时间、随机性、文件系统或无法提供测试实例的外部边界；不 mock 自有模块，数据库优先测试实例。
- 第一个功能任务优先使用最窄但完整的 tracer bullet，先证明关键链路和测试入口可用。
- 测试资产使用项目真实约定；没有约定时在任务中明确统一位置、fixtures 和运行命令。

入口只显示测试入口、优先行为、基线结论、mock 边界和 tracer bullet 摘要。具体场景、资产、命令和断言写入对应 `T-*`。

## 任务拆分

默认按可独立观察的纵向行为拆分 `T-*`，不是按数据库、后端、前端和测试横向拆层。一个切片贯穿该行为实际需要的层，完成后可以独立演示和验证。

只有真实依赖要求时才提前创建迁移、共享契约、基础设施或行为保持的 prefactor 任务。任务太大、包含多个不能独立演示的行为时拆分；拆分后只能验证内部结构或无法产生完整结果时合并。

确定任务后，先在入口展示编号化摘要：标题、交付结果、覆盖 `F/C/V`、依赖和任务链接。与用户讨论切片粒度、顺序、依赖、合并或拆分；确认摘要后再完成所有详细任务文档。

## 详细任务契约

每个 `T-*` 必须独立写入 `tasks/T-编号-简短英文-slug.md`，并完整包含：

- 追踪和目标：关联 `F/D/C/V`、纵向交付行为、独立演示结果。
- 上下文和边界：当前状态、目标状态、特别关注点、不做范围、禁止捷径；BUGFIX 时增加根因、风险和回退。
- 文件和契约：测试与实现文件、上游依赖、本任务提供、下游消费、相邻接口、字段、事件、状态和错误。
- 测试设计：公共测试入口、mock 边界、场景的输入/动作/预期、类型、资产路径和具体测试或检查内容。
- 实施步骤：一个或多个有序的测试/验证先行循环。
- 执行断言：必须消失、必须出现、必须传递、必须落盘，以及检查方式和失败判定。
- 完成和证据要求：任务完成条件、独立演示、相关 `V-*`、回归范围和执行阶段必须留下的证据。

任务文件不能用“参见入口”“参见其他任务”省略执行所需信息。可以链接方案和证据作为来源，但必须内联当前任务需要的实际契约和约束。

## 测试或验证先行循环

任务文档规划执行阶段必须遵守的循环，不在规划阶段创建测试或修改实现。

业务行为使用 `RED -> GREEN -> REFACTOR`：

- RED 计划：一个可观察行为、测试资产路径、具体测试/断言内容、精确命令和因为行为缺失而失败的预期；不得把语法、导入、环境或 fixture 错误当作 RED。
- GREEN 计划：通过当前 RED 所需的最小文件、接口、字段、状态或交互变更，以及同一命令的预期通过结果；不提前实现后续循环。
- REFACTOR 计划：只在 GREEN 后允许的整理范围和回归命令；没有必要时写“无”。

迁移、配置、构建和基础设施任务使用“失败检查 -> 最小变更 -> 通过检查 -> 整理回归”，检查可以是契约、schema、smoke、文件或可重复运行验证。

一个任务可包含多个循环，但顺序必须是 `RED1 -> GREEN1 -> RED2 -> GREEN2`，禁止批量先写所有测试再实现。

## 执行断言和最终闭环

从每个任务的方案变更和步骤提取：

- 必须消失：旧文件、调用、字段、分支、fallback 或行为及残留搜索。
- 必须出现：新文件、函数、参数、日志、事件、配置或测试资产。
- 必须传递：字段、标识、状态或 metadata 经过的完整调用链。
- 必须落盘：测试、fixture、review 清单、smoke、迁移和文档的精确路径。

每条断言写检查方式和失败判定，不能用 build、lint 或类型检查替代行为与链路证据。

涉及前后端、多服务、异步、历史、日志、外部工具或模型调用时，创建最终闭环任务；简单单文件或纯文档任务可声明不适用并说明理由。闭环必须证明真实入口、关键链路、可回查结果、失败行为和无模拟完成。

## 审查与确认

先自检并修正：入口是否简洁且覆盖完整；所有 `D/C/V` 是否进入任务；任务是否自含；接口、特别关注点和不做范围是否保留；测试场景和内容是否具体；RED/GREEN 顺序是否成立；测试资产和执行断言是否精确；依赖无环；最终闭环和交接是否完整。

然后调用 `project-development-review-v3`，要求审查入口、全部 `T-*`、技术方案和真实项目路径，并把报告写入入口的 `plan_review`。发现问题时修改原入口和任务文件并复审。审查通过后，请用户确认入口中的覆盖摘要、测试策略、任务顺序、依赖、最终闭环和风险；用户无需逐项阅读实现细节。

确认后把入口 `status` 改为 `confirmed`，保持 `execution_status: pending`，交给 `plan-execution-v3`。交接必须包含任务入口、需求、技术方案、任务数量、`F/D/C/V` 覆盖、详细任务目录、测试资产、最终闭环、风险和特别关注点。

## 完成条件

- 入口简洁，足以让用户判断覆盖、测试策略、顺序、依赖、闭环和风险。
- 每个 `F/D/C/V` 有明确任务落点，没有额外范围或未处理方案缺口。
- 每个 `T-*` 文件完整自含，执行者无需历史对话或重新设计。
- 所有行为测试或验证先行，测试内容、真实路径、命令、接口契约、执行断言和完成证据要求明确。
- 最终闭环、BUGFIX 回归、UI 覆盖及特别关注点按适用情况完整。
- 所有调研已回写，独立审查通过，用户明确确认入口摘要。

任一项不成立时保持 `discussing`，不得交给执行阶段。

## 边界

- 不修改生产代码、测试、配置或数据；只允许安全基线和命令试验。
- 不重新决定需求或方案，发现缺口时返回责任阶段。
- 不把详细任务内容塞进入口；规划阶段不得填写执行状态和执行记录。
- 不发布 issue tracker，不用脚本生成任务替代覆盖审计、测试设计和任务拆分。
