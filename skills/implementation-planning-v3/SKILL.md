# 实施任务规划 v3

目标是让不合格计划停在规划阶段：缺覆盖、缺真实路径、缺测试入口、缺执行断言或缺最终闭环时保持 `discussing`。

## 默认流程

1. 确认入口路径：只在 `workplace/<版本>/implementation-planning/` 下创建或更新一份任务入口。优先使用用户路径，其次复用已有相关入口，否则按 `YYYY-MM-DD-功能名.md` 命名。输出完整路径并继续。
2. 校验输入：只接受 `status: confirmed` 的技术方案入口。读取已确认需求、方案、相关调研和审查报告；缺少确认方案时返回 `tech-design-v3`。
3. 建立覆盖审计：提取全部 `F-*`、验收要求、`D-*`、`C-*`、`V-*`、用户特别关注点和不做范围。发现方案或需求缺口时返回责任阶段，不在任务里补设计。
4. 调查执行事实：核验真实代码路径、测试框架、fixtures、运行命令、相似任务和既有失败。必要时运行不修改项目状态的基线命令；详细输出放 `EVD-*`。
5. 写入入口和任务：新建时使用 [任务入口模板](assets/planning-entry.md) 和 [详细任务模板](assets/task-detail.md)。入口只给用户审查摘要；每个 `T-*` 必须自含可执行契约。
6. 讨论任务切片：首次从方案拆任务时，一次性讨论切片粒度、顺序、依赖、合并/拆分、最终闭环和主要风险。相关问题成组提问，不逐项拉长轮次。
7. 自检、独立审查、用户确认：自检通过后调用 `project-development-review-v3` 审查入口和全部 `T-*`。审查通过后请用户确认入口摘要，确认后把 `status` 改为 `confirmed`，保持 `execution_status: pending`。

## 规划原则

- 入口简洁：只展示覆盖、测试策略、任务顺序、依赖、闭环、风险和审查结论；实现细节写入 `T-*`。
- 任务纵向：默认按最小可验证用户价值拆分，不按后端、前端、数据库、测试横向拆层。
- 测试先行：每个行为任务包含 RED/GREEN/REFACTOR 或等价失败检查循环；规划阶段只写契约，不创建测试或改实现。
- 自含契约：执行者不读历史对话、不重新设计方案也能完成任务。
- 最终闭环：除非纯文档、纯内部重构或简单单文件且无外部可观察行为，否则必须由某个 `T-*` 承担。

## 下沉材料

按需读取 [任务规划细则](references/planning-rules.md)：

- 需要创建支撑材料、证据、调研或审查链接时。
- 需要判断覆盖完整性、可执行完整性、任务切片或依赖顺序时。
- 需要编写完整 `T-*` 契约、测试先行循环、执行断言或最终闭环时。
- 执行或验收回退导致计划需要重开时。

## 完成条件

- 入口路径真实，位于 `workplace/`，且一个事项只有一份任务入口。
- 每个 `F/D/C/V` 有明确任务落点，没有额外范围或未处理方案缺口。
- 覆盖完整性和可执行完整性分别成立。
- 每个 `T-*` 文件完整自含，测试内容、真实路径、命令、接口契约、执行断言和证据要求明确。
- 最终闭环、BUGFIX 回归、UI 覆盖及特别关注点按适用情况完整。
- 独立审查通过，用户明确确认入口摘要。

任一项不成立时保持 `status: discussing`，不得交给执行阶段。

## 边界

- 不修改生产代码、测试、配置或数据；只允许安全基线和命令试验。
- 不重新决定需求或方案，发现缺口时返回责任阶段。
- 不填写实际执行状态、执行结果或验证记录，只初始化 `pending` 和 `<待执行>`。
