# 项目验收 v3

验收不是复述执行报告，而是从已确认需求出发独立证明项目结果。只有真实验收、独立审查和用户确认全部完成，标准研发流程才结束。

## 第0步：确认文档路径（必须最先执行）

开始任何工作前，执行以下步骤：

1. 读取项目根目录，找到 `workplace/` 目录
2. 确认当前活跃版本号（必须是整数或两段数字，如 `3` 或 `3.1`）
3. 检查当前事项是否已有验收入口文档：
   - 用户给出文档路径时，使用该文件
   - `workplace/<版本>/acceptance/` 下已有相关文件时，继续更新原文件
   - 两者都没有时，从任务入口标题推导文件名，格式为 `YYYY-MM-DD-功能名.md`，输出完整路径后继续（如需修改请告知）
4. 输出将要使用的完整路径，例如：
   `workplace/3/acceptance/2026-06-23-order-import.md`
   说明"如需修改路径请告知，否则直接继续"，然后继续执行，不等待用户回复

**硬性禁止：**
- 禁止在 `workplace/` 以外创建验收文档
- 禁止将验收内容与需求、方案或任务写入同一文件
- 禁止在路径中使用 `<version>`、`x.x` 等占位符

## 文档

一个研发事项只能有一份验收入口文档，路径由第0步确认。

创建新文档时使用 [验收入口模板](assets/acceptance-entry.md)。入口只保留用户判断最终交付所需的验收结论、功能验收、原型与用户路径、回归与剩余风险。状态只使用：

- `accepting`：仍在验收、修复回退或审查。
- `confirmed`：全部验收通过、独立审查通过且用户明确接受结果与剩余风险。

测试计划、自动化资产、截图、日志、测试报告和缺陷证据放在入口旁同名目录的 `test/`、`evidence/EVD-*`、`logs/LOG-*`、`defects/DEF-*` 和 `review/REV-*`，按需创建并双向链接。入口不堆积原始输出。

## 输入门槛

只接受 `status: confirmed` 且 `execution_status: confirmed` 的任务入口。执行阶段没有独立入口。开始前读取已确认需求、方案、复用任务入口、全部 `T-*` 执行记录和 `execution_review`，建立 `F-*`、验收要求、`C-*`、`T-*`、`V-*` 和执行证据的追踪关系。

执行仍有 `not_verified`、`blocked`、未解决高风险发现或未解释偏差时，不开始最终验收，返回执行阶段处理。

## 验收设计

根据每个 `F-*` 的验收要求确定最强可行证据，不固定测试数量：

- 独立业务逻辑使用单元测试或直接可重复检查。
- API、数据库、服务边界和共享契约使用集成或契约测试。
- 用户可见功能使用真实页面、浏览器交互或端到端路径。
- 持久化、异步、外部工具、权限、超时和失败隔离验证真实行为及可回查结果。
- 兼容、迁移和共享代码变更运行相关回归。
- 自动化不可行时才使用手动步骤，并写清前置条件、动作、期望结果和证据；不能把“人工检查”当作空白证明。

可以复用执行阶段的测试资产，但必须独立核对适用范围和当前结果。不能因为任务入口或 `T-*` 写了 `done` 就直接判定验收通过。

## 执行验收

1. 按 `F-*` 逐项执行成功、关键失败和边界验收，记录使用的 `V-*`、实际结果和证据链接。
2. 验证关键用户路径从真实入口到最终可见结果，涉及 UI 时对照已确认原型检查页面、状态、反馈和交互。
3. 运行受影响模块、共享契约、数据迁移和兼容范围的回归检查。
4. 对证据与预期不一致的情况登记 `DEF-*`，记录复现步骤、期望、实际、影响、证据和责任回退阶段。
5. 修复必须回到执行阶段；缺陷暴露需求、方案或任务错误时继续回退到相应阶段。修复并重新确认后，重新运行受影响验收和回归。

每个 `DEF-*` 必须记录状态（`open` / `resolved`）、复现步骤、期望、实际、影响、证据、责任阶段、受影响 `T/F/V` 和恢复条件，并与验收项及责任入口双向链接。回退规则：

- 仅实现偏差且规划契约仍正确：验收保持 `accepting`；执行阶段按开放缺陷把 `execution_status` 从 `confirmed` 重开为 `executing`，重置执行审查，修复后重新审查并请求用户确认。
- 任务契约错误：返回规划阶段，把计划改回 `discussing`，修订后重新审查和确认；方案或需求错误继续逐级回退。
- 只有修复证据、重新确认的执行结果以及受影响验收和回归全部通过后，验收阶段才能把缺陷标记为 `resolved`。历史确认不能替代重新确认。

验收 skill 不直接修改生产代码，也不通过降低验收要求来关闭缺陷。

## 结果判断

每项验收使用：

- `passed`：真实执行且结果符合验收要求，证据可回查。
- `failed`：结果不符合要求，已关联缺陷。
- `blocked`：缺少环境、权限、数据或外部条件，无法形成结论。

只有所有必须验收项为 `passed`，才能把总体结论写为通过。`blocked` 不能包装成通过；剩余风险必须具体说明影响和接受条件，并由用户明确决定是否接受。

## 调研与提问

验收事实不明确时可在受影响项下登记 `RT-*`，经用户审核后调用 `project-research-v3`。调研不能代替实际测试。

能从项目、环境、入口文档和证据获得的信息自行检查。只有缺少用户才能提供的环境、账号、数据或风险接受决定，并且会阻止验收结论时才提问。

问题一次只问一个，提供推荐处理和不同选择的影响，放在回复最后的 `## 需要你确认` 区块，后面不追加内容。

## 自检、审查与确认

验收内容形成后先自检并修正：每个 `F-*` 和验收要求有结果；成功、失败、UI、兼容和回归覆盖与风险相称；证据来自真实执行；缺陷均有状态和回退；没有把执行自检当作独立验收。

然后调用 `project-development-review-v3`。发现问题时继续验收或回退修复并复审。审查通过后，请用户确认验收结论、功能结果、原型与用户路径、回归和剩余风险；全部确认后把状态改为 `confirmed`。

## 完成条件

- 每个 `F-*` 及其验收要求均有 `passed` 结果和可回查证据。
- 关键成功、失败、边界、UI/原型、兼容和回归已经真实验证。
- 所有缺陷已修复并复验，没有 `failed` 或 `blocked` 必须项。
- 独立审查通过，用户明确确认最终结果和剩余风险。

报告项目完成前重新对照本节自检。任一项不成立时保持 `accepting`，不得结束流程。

## 边界

- 不修改生产代码或替执行阶段关闭缺陷。
- 不降低已确认需求和验收要求，不默认接受剩余风险。
- 不为每个测试或验收项创建入口文档。
- 不以文档状态、测试名称或历史结果代替当前真实验收。
