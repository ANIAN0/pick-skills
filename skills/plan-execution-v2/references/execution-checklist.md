# 执行检查清单

每个 `T-*` 任务都使用这份清单。

## 编辑前

- 读取 `index.md`。
- 读取当前任务文件。
- 检查关联的 `R-*`、`A-*`、`C-*` 和 `V-*`。
- 检查相关现有代码。
- 确认任务中没有未解决阻塞。
- 提取本任务的真实依赖要求：数据库、MCP、tool、daemon、调度、共享契约、UI、日志、diagnostics、外部服务。
- 提取本任务的负向验收要求：失败、超时、无效引用、禁用、隔离、回滚、来源追溯。
- 提取本任务的测试资产目录和关联 `V-*`。
- 提取本任务的执行断言：必须消失、必须出现、必须传递、必须落盘。
- 检查任务文件中没有未替换的 `<version>`、`1.X` 或其他占位路径。
- 提取特别关注点；没有则记录“无”。

## 编辑中

- 保持在任务范围内。
- 除非任务明确要求改变，否则沿用现有代码模式。
- 不引入 mock、fallback 或 stub 完成。
- 不用内存 `Map`、模块级 dict、数组替代计划要求的持久化；如果只是缓存，必须有真实事实源。
- 不用固定返回、回显输入、空校验、恒真/恒假函数替代真实集成。
- 新增测试、fixtures、review 清单和 smoke 脚本放到计划指定的 `workplace/<version>/test/` 子目录。
- 执行过程中不提交 git，除非用户明确要求。
- 如果执行细节有小变化，更新计划。
- 只有范围、数据、兼容性、安全或 UX 决策才询问用户。

## 验证

- 运行任务定义的验证。
- 变更触及共享行为时，运行邻近测试。
- UI 任务可行时，验证真实渲染行为。
- API/数据任务可行时，验证真实本地契约或集成行为。
- build/typecheck/lint 只能证明基础质量；除非任务只要求构建，否则不能单独作为 `done` 证据。
- 持久化任务必须验证写入后可查询，必要时验证重启或重新初始化后数据仍存在。
- MCP/tool/调度任务必须验证真实 discover/list/resolve/call，不得只验证固定响应。
- 引用校验任务必须验证有效引用和无效引用两类路径。
- diagnostics/log/event 任务必须验证成功和失败记录都可查询。
- 计划或验收包含失败、超时、隔离时，必须验证负向路径。
- 同一验证连续失败最多修复三轮；三轮后停止并记录为 `blocked` 或 `not_verified`。

## 执行断言检查

标记完成前逐条检查任务文件的 `执行断言`：

- 必须消失：运行计划指定搜索，确认旧循环、旧过滤、旧字段、旧调用或旧 fallback 无残留。
- 必须出现：确认新参数、函数、调用、字段、metadata、日志、配置或测试文件存在。
- 必须传递：确认字段沿计划指定调用链贯通，例如入口参数 -> 内部函数 -> LLM/tool 调用 -> side-event metadata -> 日志/响应。
- 必须落盘：确认计划指定的测试资产精确文件存在，并运行对应命令。
- 如果检查失败，回到实现；不能用 build/typecheck/lint 或结构检查替代。

证据日志必须记录每条断言的检查命令或证据。任务没有执行断言但包含删除、替换、传递或测试落盘要求时，先补任务文件再继续。

## 反模拟扫描

标记完成前，在任务修改范围内搜索并处理：

- `mock`, `stub`, `fake`, `dummy`, `placeholder`, `fallback`, `hardcoded`
- `模拟`, `占位`, `临时`, `固定返回`, `回显`
- `TODO`, `return []`, `return true`, `return false`
- `new Map`, 模块级 dict、数组或全局变量存储

证据日志必须记录：

- 扫描方式。
- 命中项。
- 每个命中项为什么不是完成捷径，或已如何修复。

## 测试资产合规

- 新增测试文件在 `workplace/<version>/test/backend/unit/`、`backend/integration/`、`frontend/unit/`、`frontend/component/`、`contract/`、`smoke/` 或 `review/` 下。
- fixtures 在 `workplace/<version>/test/fixtures/` 下。
- `README.md` 包含运行命令矩阵、依赖前置条件和覆盖的 `V-*`。
- 源码旁新增测试必须有计划例外和理由。
- 每个相关 `V-*` 能追踪到测试文件、脚本、UI smoke 或可观察步骤。

## 独立任务审核

标记 `done` 前独立复核：

- 功能覆盖：当前任务关联的 `R-* / A-* / C-* / V-*` 均有实现和证据。
- 验收执行：验证命令或步骤已真实运行，失败路径已覆盖。
- 反模拟：无 mock/fallback/stub/固定返回/空校验冒充完成。
- 边界契约：输入、输出、字段、状态和下游消费一致。
- 测试目录：新增测试资产位置合规。
- 特别关注点：相关关注点已落实。

## 证据日志格式

```markdown
## 证据日志
- 修改文件：
  - <路径>
- 执行命令：
  - `<命令>` -> <结果>
- UI/API 证据：
  - <观察结果>
- done 前硬门禁：
  - 反模拟扫描：<命中项和处理结论>
  - 真实依赖接入：<数据库/MCP/tool/UI/diagnostics 等证据>
  - 负向路径验证：<失败/超时/无效引用/隔离等证据或不适用理由>
  - 执行断言检查：<ASSERT-* 的命令、结果和处理结论>
  - 测试资产合规：<新增测试文件、fixtures、README 更新和例外说明>
  - 独立审核：<通过/问题，审核方式和结论>
- 与计划的偏差：
  - <无或详情>
- 验证状态：
  - done | not_verified | blocked
```

## 标记完成前

- 所有计划步骤已完成。
- 验证已通过。
- 证据已记录。
- 没有用捷径替代计划行为。
- 执行断言全部通过；旧代码无残留，新增/传递/落盘项有证据。
- 反模拟扫描已执行，命中项已解释或修复。
- 计划要求的真实依赖已经接入并有证据。
- 任务要求的失败、超时、无效引用、诊断或隔离路径已经验证；不适用时有明确理由。
- 测试资产目录合规，或计划中有明确例外。
- 独立任务审核通过。
- 如果只能通过 build/typecheck/lint，任务本身必须确实只要求构建，否则不得标记 `done`。
- 剩余风险已明确。

## 审查后状态回滚

如果审查发现已完成任务不满足计划、验证或反模拟要求：

- 把任务状态从 `done` 改为 `in_progress`、`not_verified` 或 `blocked`。
- 在证据日志追加审查发现编号和原因。
- 在 `index.md` 同步更新总览。
- 阻塞级发现解决前，不得把相关最终闭环或整体计划标记完成。

## 整体完成前

- 所有非跳过任务都有状态和证据。
- 最终闭环/放行任务已执行；若不适用，计划中有明确理由。
- 全局验证已运行。
- 最终审核已执行并通过。
- 每个 `done` 任务的执行断言均有证据；计划指定测试资产文件全部存在。
- 测试资产目录完整，README 命令矩阵已更新。
- 无 mock、固定回复、硬编码测试 ID、测试旁路或降级实现冒充完成。
- `not_verified` 或 `blocked` 项不得被包装成整体完成。
