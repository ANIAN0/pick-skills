# 实施计划创建

将技术方案拆分为模块任务清单，通过链接引用已有文档。

<HARD-GATE>
- 技术方案必须已存在
- 计划不重复技术细节，只编排任务顺序
- 必须**同时**覆盖技术方案的后端与前端模块——遗漏前端模块即不合格
</HARD-GATE>

## 版本号约定

`1.X` 为占位符，详见 `requirements-workshop/SKILL.md`。真实路径必须替换为具体数字（如 `workplace/1/plan/`）。

## 文档结构原则

产出文档为**目录结构**，解决上下文占用和状态追踪：

```
workplace/1.X/plan/YYYY-MM-DD-xxx/
├── index.md               # 执行索引 + 元数据 + 特别关注点 + 映射表
├── M1-数据库迁移与模型.md   # 完全自含的模块详情
├── M2-创建工单API.md
├── M3-工单创建页.md
└── ...
```

**为什么用目录而非单文件**：plan-execution 执行时需要"只看当前模块"，单文件内无法物理隔离——agent 读文件时总会看到全部内容。拆成独立文件后，读 `index.md` 就只知道索引，读 `M1-xxx.md` 就只有 M1 的上下文，天然隔离。

**两层读取**：
- **index.md**：紧凑任务表 + 元数据 + 特别关注点。plan-execution 初始化时只读此文件
- **模块文件**：每个模块独立一个 `.md`，完全自含。执行某模块时只读对应文件

**模块文件自含原则**：模块文件不得用"参考 index.md"来省略信息——subagent 只能看到这一个文件，必须包含完成该模块所需的全部上下文（包括与该模块相关的特别关注点、相邻模块的接口契约摘要）。

**模块级拆分原则**：
- 任务按**技术模块/业务模块**分组，单模块预计 8-16 小时
- 模块边界 = 上下文切换点，模块内连续实现
- **前端必须独立成模块**（不与后端混在同一模块）
- **不创建端到端联调模块**：本套 skill 默认只到模块级集成测试为止

## 第三步：确定模块顺序

基于第二步的拆分结果，确定执行顺序。

**默认顺序**：数据层 → 服务层 → 接口层 → 前端基础 → 前端页面

**排序原则**：
- 被依赖的模块先执行
- 同层内按业务重要性排序
- 前端模块在后端接口完成后执行

## 第五步：关联验收标准

基于技术方案的功能点清单（FP-N）和测试点清单（TC-N），为每个模块创建验收标准。

**核心原则**：
- 每个任务模块对应一组功能点
- 每个功能点有且只有一种验收方式：后端py脚本 或 前端subagent审查清单
- 计划阶段必须编写具体的验收脚本或审查清单内容
- 基础设施模块（无对应功能点，如数据库迁移、项目骨架）使用 TC-0 编号，验收脚本验证基础结构而非业务功能

**关联流程**：

1. **按模块映射功能点**：从技术方案的功能点清单中，将每个 FP-N 分配到对应模块
2. **解析测试点路径占位符**：将技术方案测试点清单中的 `{module}` 占位符替换为当前模块编号（如 M1、M2），确保验收脚本路径与模块对应。注意：目录中的 `{module}` 保持模块编号大小写（如 M1、M2），文件名中的 `{module}` 转为小写（如 m1、m2）
3. **逐模块编写验收标准与接口契约**：
   - **后端模块**：编写 py 验收脚本（含 assert 语句），覆盖该模块所有 FP-N 对应的 TC-N。脚本须可直接运行、输出 `M{N} ALL PASS`
   - **前端模块**：编写审查清单（含逐项勾选框），覆盖该模块所有 FP-N 对应的 TC-N。每项须描述具体操作与预期结果；此外标注前端测试文件目录（单元/组件测试至少一类，前端模块必备）
   - **接口契约**：为每个模块定义"提供"（本模块暴露给下游的接口/数据）和"依赖"（本模块消费的上游接口/数据），来源为技术方案的 API 设计、数据模型、前端设计章节。计划执行阶段 implementer 和 module-reviewer 均需要此信息
4. **确认覆盖完整性**：核对功能点-模块映射表，确保每个 FP-N 和 TC-N 都有对应的验收标准

## 特殊情况

- **模块过大**：超过 16 小时则拆为子模块（M1-a, M1-b），保持接口契约
- **无测试命令**：后端模块仍须写 py 验收脚本（可简化为仅调用接口验证返回值）；前端模块写审查清单替代
- **简单项目（<3 个模块）**：跳过里程碑，精简清单
- **跨模块协调**：放入前置模块末尾或单独作微型模块
- **跳过模块**：状态可标 `跳过`，注明原因和受影响下游；下游若依赖被跳过模块的输出，必须更新前置依赖或加替代实现
- **纯后端服务（无 UI）**：可不含前端模块，但需在文档开头声明"本项目无前端"，否则审核默认要求前端模块
- **BUGFIX 计划**（来自 issue-troubleshooting 联动）：每个模块名加 `[BUGFIX]` 前缀；每个模块详情额外包含"关联根因"、"风险点"、"回退方案"三个字段；最后一个模块为"回归验证"（运行全量测试+验证原bug已修复+检查关联功能）
