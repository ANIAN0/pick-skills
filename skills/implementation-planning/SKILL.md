# 实施计划创建

将技术方案拆分为任务清单，通过链接引用已有文档。

<HARD-GATE>
技术方案必须已存在。实施计划不重复技术细节，只编排任务顺序。
</HARD-GATE>

## 核心设计

输出文档分两层，解决**上下文占用**和**状态追踪**两个问题：

- **执行索引**：紧凑任务表，含状态列。每次执行只需读这一节，找到下一个"待执行"任务
- **模块详情**：每个**模块**独立成节。执行某模块时只读对应章节，不加载其他模块内容

**关键原则：模块级拆分**
- 任务按**技术模块/业务模块**分组，而非按单个操作或文件拆分
- 单个模块预计8-16小时完成，包含内部子步骤但不做独立审核
- 模块边界=上下文切换点，模块内连续实现，减少审核开销

## 工作流程

1. 从 `workplace/1.0/tech-design/` 读取技术方案
2. 提取技术方案中的架构设计、数据模型、API 设计，自主拆分阶段和任务
3. 确认执行顺序：数据层 → 服务层 → 接口层 → 前端
4. 关联验收标准（来自技术方案的测试策略、约束定义）
5. 生成下方模板格式的文档，提交用户确认

**检查**：若无技术方案，提示用户先完成技术方案设计。

## 设计原则

- **聚焦编排**：不重复技术细节，通过链接引用技术方案章节
- **按需加载**：索引紧凑（O(n)行），详情按章节独立读取（O(1)模块）
- **状态持久**：执行后立即更新索引状态并提交，跨会话可接续
- **验收可执行**：每个模块有整体验收标准（测试命令或人工检查点）
- **模块优先**：按技术层次/业务模块拆分，单个模块8-16小时完成。模块内部子步骤不独立审核
- **边界清晰**：模块之间通过接口契约衔接，模块内连续实现减少上下文切换

## 特殊情况

- **模块过大**：若单个模块预估超过16小时，拆分为子模块（M1-a, M1-b），保持接口契约
- **无测试命令**：改为"检查：[具体人工验收步骤]"
- **简单项目（<3个模块）**：跳过里程碑，直接产出精简清单
- **混合任务**：若某步骤需要跨模块协调（如全局配置），放入前置模块的末尾或单独作为一个微型模块
