# 实施计划创建

将技术方案拆分为任务清单，通过链接引用已有文档。

<HARD-GATE>
技术方案必须已存在。实施计划不重复技术细节，只编排任务顺序。
计划必须**同时**覆盖技术方案中的后端模块与前端模块——遗漏前端模块即不合格。
</HARD-GATE>

## 版本号约定

`1.X` 为当前工作版本目录占位符（X 为整数）。详见 `requirements-workshop/SKILL.md`。
真实路径必须替换为具体数字，例如 `workplace/1/plan/`。

## 核心设计

输出文档分两层，解决**上下文占用**和**状态追踪**两个问题：

- **执行索引**：紧凑任务表，含状态列。每次执行只需读这一节，找到下一个"待执行"任务
- **模块详情**：每个**模块**独立成节。执行某模块时只读对应章节，不加载其他模块内容

**关键原则：模块级拆分**
- 任务按**技术模块/业务模块**分组，而非按单个操作或文件拆分
- 单个模块预计 8-16 小时完成，包含内部子步骤但不做独立审核
- 模块边界=上下文切换点，模块内连续实现，减少审核开销
- **前端必须独立成模块**（页面/组件/状态），不与后端混在同一模块，便于并行实现与审核

## 工作流程

1. 从 `workplace/1.X/tech-design/` 读取技术方案（含前端设计章节）
2. 提取技术方案中的架构设计、数据模型、API 设计、**前端设计（路由/页面/组件）**，自主拆分阶段和任务
3. 确认执行顺序的默认建议：**数据层 → 服务层 → 接口层 → 前端基础（路由/状态/API 客户端） → 前端页面 → 端到端联调**
4. 关联验收标准（来自技术方案的测试策略、约束定义）；测试文件位置统一指向 `workplace/1.X/test/` 下对应子目录
5. 生成下方模板格式的文档，提交用户确认
6. 派发 plan-document-reviewer subagent 审查

**检查**：
- 若无技术方案，提示用户先完成技术方案设计
- 若技术方案缺少"前端设计"章节，提示回到 tech-design 补全后再来
- 计划完成后必须有：至少一个前端模块（除非该项目本身无 UI）

## 设计原则

- **聚焦编排**：不重复技术细节，通过链接引用技术方案章节
- **按需加载**：索引紧凑（O(n)行），详情按章节独立读取（O(1)模块）
- **状态持久**：执行后立即更新索引状态并保存文件，跨会话可接续
- **验收可执行**：每个模块有整体验收标准（测试命令或人工检查点）
- **模块优先**：按技术层次/业务模块拆分，单个模块 8-16 小时完成。模块内部子步骤不独立审核
- **边界清晰**：模块之间通过接口契约衔接，模块内连续实现减少上下文切换
- **前后端均覆盖**：前端必须以独立模块出现（页面/组件/状态/路由），不允许"顺带做"
- **测试集中**：所有测试文件统一落到 `workplace/1.X/test/` 下，按 backend/frontend/e2e/fixtures 分类

## 特殊情况

- **模块过大**：若单个模块预估超过 16 小时，拆分为子模块（M1-a, M1-b），保持接口契约
- **无测试命令**：改为"检查：[具体人工验收步骤]"，并在 `workplace/1.X/test/` 下放置一份手测脚本/checklist
- **简单项目（<3 个模块）**：跳过里程碑，直接产出精简清单
- **混合任务**：若某步骤需要跨模块协调（如全局配置），放入前置模块的末尾或单独作为一个微型模块
- **跳过模块**：状态可标 `跳过`，但必须在该模块章节末尾注明跳过原因和受影响的下游模块；下游模块若依赖被跳过模块的输出，必须更新前置依赖或添加替代实现
- **纯后端服务（无 UI）**：可不含前端模块，但需在文档开头声明"本项目无前端"，否则审核默认要求前端模块
