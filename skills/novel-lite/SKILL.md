# 小说单章创作

## 定位

三件套分工：
- **novel-setup**：宏观层。项目启动、大纲、主角档案、故事状态、伏笔追踪。
- **novel-lite（本skill）**：中观层。单章怎么写、怎么改、怎么定稿。
- **novel-review**：中观层。单章怎么审、怎么循环。

本 skill 是**创作核心**：每次开写新章节前读 setup 的简报，写完定稿后回调 setup 回写状态。

## 实体设定文件（角色/物品/技能）

每个实体单独一个 md，只存核心信息：

```markdown
# [实体名称]

## 主角知晓状态
- [主角已知 / 主角未知]
- 知晓方式: [直接经历 / 他人告知 / 传闻 / 未知]
- 当前认知: [完全了解 / 部分了解 / 误解 / 毫无认知]

## 基本信息
[关键属性]

## 核心特征
[最重要的特征]

## 相关关系
[与其他实体的关联]

## 出场记录
- 第X章：[做了什么 / 状态变化]
```

### 主角知晓状态说明

**核心原则**：读者与主角同步探索，禁止"读者知道但主角不知道"的信息不对称。

**知晓方式**：
- 直接经历：主角亲身经历
- 他人告知：被其他角色告诉
- 传闻：听到的传说（可能真假混杂）
- 未知：完全不了解

**当前认知**：
- 完全了解：掌握完整真相
- 部分了解：存在认知缺口
- 误解：与真相有偏差（适合制造冲突）
- 毫无认知：完全无知

## 章节三件套

- **事件摘要.md**：本章主要事件（3-5 条）
- **正文.md**：章节内容
- **相关实体.md**：本章涉及的角色/物品/技能及其行为/变化/状态迁移

## 创作流程

### 写新章节前：读开写前简报

开写第 N 章前，必须按以下顺序读取（或请求 novel-setup 产出简报）：

1. `相关设定/故事大纲.md` — 本章在整体中的定位
2. `相关设定/故事状态.md` — 当前主角/地点/时间/活跃支线
3. `相关设定/伏笔追踪.md` — 待回收伏笔与计划新埋点
4. `相关设定/主角档案.md` — 主角当前弧光状态
5. `相关设定/写作注意事项.md` — 硬性约束
6. `相关设定/创作风格.md` — 文风基调
7. `相关设定/角色/`中本章可能出场的角色
8. `创作标准/好的示例.md` 与 `差的示例.md`
9. `正文/第N-1章/` — 上一章结尾，用于衔接

### 三种创作路径（灵活选择）

- **摘要 → 正文**：用户给摘要 → 生成正文 → 提取实体
- **正文 → 摘要**：用户给正文 → 提取摘要 → 提取实体
- **修改章节**：修改正文 → 同步更新摘要和实体

### 主角视角约束

写正文前，先检查相关角色/物品/技能设定的"主角知晓状态"。正文**只能**展示主角已知或正在探索的信息。主角未知的真相只能通过暗示、悬念或他人视角呈现。

## 三选项机制

每次写正文或重大修改前必须给出 3 个不同选项：

```
**选项A**: [描述] — 特点: [差异点]
**选项B**: [描述] — 特点: [差异点]
**选项C**: [描述] — 特点: [差异点]

**推荐**: 选项[X] — 原因: [理由]
```

差异点应体现在：叙事节奏、视角选择、情绪基调、冲突强度、信息释放方式之一，而非仅遣词造句。

## 版本管理

### 工作区探索

创作时在 **工作区/第X章VX/** 下探索不同版本：
- 目录命名：第X章V1、第X章V2
- 每个版本目录包含完整三件套
- 用于探索不同叙事风格、不同内容走向、不同节奏

### 审查集成

审查报告由 novel-review 产出，固定写入：
```
工作区/第X章VX/审查历史/review_YYYYMMDD_HHMM.md
```

修订本章节正文时，在正文.md 的顶部注释块中追加本次修订依据的审查报告与问题编号：

```markdown
<!--
修订记录：
- 2026-04-22 14:30 基于 review_20260422_1400.md 修订 Q1、Q3、Q5（P0）
- 2026-04-22 16:10 基于用户反馈补充 Q7 相关伏笔
-->
```

### 定稿流程

用户确定某 V 版本定稿后：

1. **定稿移入**：将 `工作区/第X章VX/` 下的 `事件摘要.md`、`正文.md`、`相关实体.md` 移动到 `正文/第X章/`
2. **其他版本归档**：
   - 工作区中其他版本的 `正文.md` 移到 `版本存档/第X章/`，重命名为 `正文_v1.md`、`正文_v2.md`...
   - 事件摘要.md 和 相关实体.md 删除（只存档正文）
3. **审查历史保留**：定稿版本的 `审查历史/` 整体移入 `正文/第X章/审查历史/`
4. **设定更新**：根据定稿版本的 `相关实体.md` 更新 `相关设定/角色/*` 等实体文件，补「出场记录」
5. **状态回写（必须触发 novel-setup）**：
   - 更新 `故事状态.md`（当前章、主角状态、下一章定位）
   - 更新 `伏笔追踪.md`（本章新埋 / 回收 / 强化的伏笔）
   - 如主角弧光推进，更新 `主角档案.md` 的"当前状态"
   - 如偏离大纲，提示用户选择：更新大纲 / 回改章节
6. **清理工作区**：删除 `工作区/第X章/` 下所有版本目录

### 定稿前自检清单

移入正文前，逐项确认：
- [ ] 至少跑过一轮 novel-review 审查循环
- [ ] P0 级问题全部处理或有明确理由保留
- [ ] 主角知晓状态无违规
- [ ] 本章涉及的实体设定已同步更新
- [ ] 至少明确了 1 个本章埋设/回收的伏笔
- [ ] 章末有钩子或明确的情绪落点

## 伏笔登记与回收（重要）

写作中发现"这里埋了个东西"或"这里回收了之前某个线"，**立即登记到本章的 `相关实体.md` 底部**：

```markdown
## 本章伏笔动作
- 新埋 F00X: [简称] — 位置：[段落] — 计划回收：[第Y章左右]
- 回收 F00X: [简称] — 回收方式：[简述]
- 强化 F00X: [简称] — 强化内容：
```

定稿时这些记录由状态回写流程同步到 `伏笔追踪.md`。

## 创作标准

### 标准文件结构

**好的示例.md**：
```markdown
# 好的示例积累

## 对话描写
- 示例: [引用正文片段]
- 原因: [为什么好]
- 适用场景: [什么情况下适用]

## 情绪渲染
...
```

**差的示例.md**：
```markdown
# 需避免的示例

## 信息堆砌
- 示例: [引用需改进内容]
- 问题: [具体问题]
- 改进方向: [如何避免]
```

### 标准积累方式

- 用户指出好的内容 → 提取到 `好的示例.md`
- 用户指出问题 → 提取到 `差的示例.md`
- 每次创作前查阅创作标准
- 每次讨论后更新创作标准

## 设定更新检查

每次对话后思考：

1. **实体内容更新**：用户反馈中有无可更新的角色/物品/技能设定？
2. **知晓状态更新**：主角在本次剧情中是否获得了新认知？
   - "主角未知" → "主角已知"（记录知晓方式）
3. **信息对称检查**：是否存在读者知道但主角不知道的情况？
4. **状态回写提醒**：本次对话若产生剧情推进，提醒定稿时触发 novel-setup 回写

## 边界

- 不做项目级大纲规划（交给 novel-setup）
- 不做章节级审查（交给 novel-review）
- 不擅自修改 `故事大纲.md` / `主角档案.md` / `伏笔追踪.md`（由 setup 负责；本skill只在定稿回写时触发）
- 定稿前不得跳过审查循环（除非用户明确跳过）
