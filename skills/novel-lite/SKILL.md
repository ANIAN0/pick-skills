# 小说单章创作

## 定位

三件套分工：
- **novel-setup**：宏观层。项目启动、大纲、主角档案、故事状态、伏笔追踪。
- **novel-lite（本skill）**：中观层。单章怎么写、怎么改、怎么定稿。
- **novel-review**：中观层。单章怎么审、怎么循环。

本 skill 是**创作核心**：每次开写新章节前读 setup 的简报，写完定稿后回调 setup 回写状态。

## 实体设定文件（角色/物品/技能）

每个实体单独一个 md，只存核心信息：

```markdown
# [实体名称]

## 主角知晓状态
- [主角已知 / 主角未知]
- 知晓方式: [直接经历 / 他人告知 / 传闻 / 未知]
- 当前认知: [完全了解 / 部分了解 / 误解 / 毫无认知]

## 基本信息
[关键属性]

## 核心特征
[最重要的特征]

## 相关关系
[与其他实体的关联]

## 出场记录
- 第X章：[做了什么 / 状态变化]
```

### 主角知晓状态说明

**核心原则**：读者与主角同步探索，禁止"读者知道但主角不知道"的信息不对称。

**知晓方式**：
- 直接经历：主角亲身经历
- 他人告知：被其他角色告诉
- 传闻：听到的传说（可能真假混杂）
- 未知：完全不了解

**当前认知**：
- 完全了解：掌握完整真相
- 部分了解：存在认知缺口
- 误解：与真相有偏差（适合制造冲突）
- 毫无认知：完全无知

## 待定设定（工作区/待定设定/）⭐ 新增

### 定位

创作过程中常会**临时确定**一些设定细节（某 NPC 的具体名字、某条规则的边界、某个物品的副作用、某个组织的内部分工……）。这些设定：
- 与正在写的章节强相关，但**还没经过审查/审视**；
- 不应直接污染 `相关设定/`（那里只放定稿设定）；
- 又必须被记录与跨章传递，否则下章就会忘掉、自相矛盾。

`工作区/待定设定/` 是**项目级共享池**，跨章节共用，定稿章节时**不会**被清理（章节工作区清理只清 `工作区/第X章VX/`）。

### 何时产出待定设定（触发时机）

写正文 / 提取实体 / 修订时，遇到以下情况，**立即**在 `工作区/待定设定/` 创建或更新对应文件：

1. 正文出现了 `相关设定/` 中没有的命名实体（人/物/地/组织）
2. 正文为已有实体扩展了属性（新能力、新关系、新背景）但该扩展尚未审定
3. 正文为世界规则给出了具体参数/边界（数值、范围、代价、例外）
4. 临时编了一个解释来圆场，需要被后续审视决定是否正式化
5. 用户在对话中给出新设定但说"先这样定，回头再说"

### 待定设定文件格式

每条待定设定是 `工作区/待定设定/<类别>/<名称>.md`，模板：

```markdown
# [名称]（待定）

## 元信息
- 类别：角色 / 物品 / 技能 / 规则
- 状态：🟡 草案 / 🟠 待审 / 🟢 拟晋升 / ⚫ 拟废弃
- 起源章节：第X章VX（首次出现位置 / 段落）
- 关联章节：第X章, 第Y章 ...
- 关联既有设定：[相关设定/角色/林轲.md] 等
- 是否影响主线：是 / 否

## 内容
[与正式设定模板对齐：实体走主角知晓状态+基本信息+核心特征+相关关系+出场记录；规则走规则名+触发条件+作用范围+边界/例外+代价]

## 待审视的疑点（创作时记下，留给 review）
- [ ] 与既有设定 X 是否冲突？
- [ ] 数值/边界是否合理？
- [ ] 是否有更克制的写法（不必正式化为设定）？

## 决策日志
- YYYY-MM-DD：创建于第X章VX
- YYYY-MM-DD：经 review_xxx 审查，调整 ...
- YYYY-MM-DD：晋升至 `相关设定/角色/XX.md` / 废弃，理由 ...
```

### 待定设定索引

`工作区/待定设定/_索引.md` 是总览，每次新增/更新待定项时同步：

```markdown
# 待定设定索引

| ID | 名称 | 类别 | 状态 | 起源 | 关联章节 | 拟处置 |
|----|------|------|------|------|---------|--------|
| P001 | 老槐树下的铁盒 | 物品 | 🟠 待审 | 第3章V2 | 第3章, 第5章 | 待 review |
| P002 | 失心者僵直窗口 | 规则 | 🟢 拟晋升 | 第4章V1 | 第4章 | 晋升至 写作注意事项.md |
| P003 | 守门人老周 | 角色 | 🟡 草案 | 第2章V1 | 第2章 | 暂存 |
```

### 待定设定的生命周期

```
🟡 草案（创作中临时记） 
   ↓（用户认为可以审视，或定稿前必须处理）
🟠 待审（交给 novel-review 的"新设定审查"模式）
   ↓（review 后用户决策）
🟢 拟晋升 → 由 novel-lite 移入 `相关设定/<类别>/`，从待定池删除，更新决策日志
   或
⚫ 拟废弃 → 从待定池删除，并检查正文是否需要回改以撤销该设定的暗示
```

### 与章节定稿的关系

章节定稿（移入 `正文/第X章/`）时**不强制**清空相关待定项，但必须：
1. 列出本章涉及的所有待定项；
2. 对每条做明确处置：晋升 / 保留待定（继续观察）/ 废弃；
3. 状态为 🟡 草案的项目，至少升级为 🟠 待审，避免被遗忘；
4. 更新 `_索引.md`。

### 与 novel-review 的衔接

- 创作时一旦累积 ≥3 条 🟠 待审 项，提示用户："建议跑一轮 novel-review 的新设定审查"
- review 报告会针对待定项给出晋升/调整/废弃建议，**不直接改文件**，由本 skill 落地

### 与 novel-setup 的衔接

- 待定项若涉及世界观底层规则、主线大纲、伏笔体系，晋升时必须由 novel-setup 接手（更新 `世界设定.md` / `故事大纲.md` / `伏笔追踪.md`），而非简单写入 `相关设定/` 子目录。
- 本 skill 在晋升此类项时**不擅自改宏观文件**，只在决策日志中标注"待 novel-setup 接手"。

## 章节三件套

- **事件摘要.md**：本章主要事件（3-5 条）
- **正文.md**：章节内容
- **相关实体.md**：本章涉及的角色/物品/技能及其行为/变化/状态迁移
- **未在 `相关设定/` 中的实体或规则，必须同时在 `工作区/待定设定/` 中有对应条目**

## 创作流程

### 写新章节前：读开写前简报

开写第 N 章前，必须按以下顺序读取（或请求 novel-setup 产出简报）：

1. `相关设定/故事大纲.md` — 本章在整体中的定位
2. `相关设定/当前冲突.md` ⭐ — 短期靶心：本章必须啃哪一块、推进哪一节拍
3. `相关设定/故事状态.md` — 当前主角/地点/时间/活跃支线
4. `相关设定/伏笔追踪.md` — 待回收伏笔与计划新埋点
5. `相关设定/主角档案.md` — 主角当前弧光状态
6. `相关设定/写作注意事项.md` — 硬性约束
7. `相关设定/创作风格.md` — 文风基调
8. `相关设定/角色/`中本章可能出场的角色
9. `创作标准/好的示例.md` 与 `差的示例.md`
10. `正文/第N-1章/` — 上一章结尾，用于衔接

### 三种创作路径（灵活选择）

- **摘要 → 正文**：用户给摘要 → 生成正文 → 提取实体
- **正文 → 摘要**：用户给正文 → 提取摘要 → 提取实体
- **修改章节**：修改正文 → 同步更新摘要和实体

### 主角视角约束

写正文前，先检查相关角色/物品/技能设定的"主角知晓状态"。正文**只能**展示主角已知或正在探索的信息。主角未知的真相只能通过暗示、悬念或他人视角呈现。

## 三选项机制

每次写正文或重大修改前必须给出 3 个不同选项：

```
**选项A**: [描述] — 特点: [差异点]
**选项B**: [描述] — 特点: [差异点]
**选项C**: [描述] — 特点: [差异点]

**推荐**: 选项[X] — 原因: [理由]
```

差异点应体现在：叙事节奏、视角选择、情绪基调、冲突强度、信息释放方式之一，而非仅遣词造句。

## 版本管理

### 工作区探索

创作时在 **工作区/第X章VX/** 下探索不同版本：
- 目录命名：第X章V1、第X章V2
- 每个版本目录包含完整三件套
- 用于探索不同叙事风格、不同内容走向、不同节奏

### 审查集成

审查报告由 novel-review 产出，固定写入：
```
工作区/第X章VX/审查历史/review_YYYYMMDD_HHMM.md
```

修订本章节正文时，在正文.md 的顶部注释块中追加本次修订依据的审查报告与问题编号：

```markdown
<!--
修订记录：
- 2026-04-22 14:30 基于 review_20260422_1400.md 修订 Q1、Q3、Q5（P0）
- 2026-04-22 16:10 基于用户反馈补充 Q7 相关伏笔
-->
```

### 定稿流程

用户确定某 V 版本定稿后：

1. **定稿移入**：将 `工作区/第X章VX/` 下的 `事件摘要.md`、`正文.md`、`相关实体.md` 移动到 `正文/第X章/`
2. **其他版本归档**：
   - 工作区中其他版本的 `正文.md` 移到 `版本存档/第X章/`，重命名为 `正文_v1.md`、`正文_v2.md`...
   - 事件摘要.md 和 相关实体.md 删除（只存档正文）
3. **审查历史保留**：定稿版本的 `审查历史/` 整体移入 `正文/第X章/审查历史/`
4. **设定更新**：根据定稿版本的 `相关实体.md` 更新 `相关设定/角色/*` 等实体文件，补「出场记录」
5. **待定设定处置**：列出本章关联的所有待定项，逐条决策：
   - 🟢 拟晋升 → 移入 `相关设定/<类别>/`，从 `工作区/待定设定/` 删除，更新 `_索引.md`；若涉及世界规则/伏笔/大纲，标注"由 novel-setup 接手"
   - 🟠 待审 → 保留在池中，索引备注下次必须处理
   - ⚫ 拟废弃 → 删除文件，检查正文是否仍残留对应暗示，必要时回改
   - 🟡 草案 → **必须升级**为 🟠 或处置，禁止以 🟡 状态进入下一章
6. **状态回写（必须触发 novel-setup）**：
   - 更新 `故事状态.md`（当前章、主角状态、下一章定位）
   - 更新 `当前冲突.md` ⭐（勾选本章推进的节拍；冲突演化时归档历史并询问下一段）
   - 更新 `伏笔追踪.md`（本章新埋 / 回收 / 强化的伏笔）
   - 如主角弧光推进，更新 `主角档案.md` 的"当前状态"
   - 如偏离大纲，提示用户选择：更新大纲 / 回改章节
   - **接手 novel-lite 标注的"待 setup 处理"的待定项晋升**
7. **阶段回顾触发判断** ⭐：
   - 计算定稿章号 N，若 `N % 5 == 0`（即第 5/10/15/20... 章），**必须**主动提示用户：
     > "已完成第 N 章定稿，建议立即跑一轮 novel-review 的 **阶段回顾（review-arc）**——它会回看最近 5 章正文，评估相关设定是否需要更新、当前冲突是否需要演化、有无被剧情发展悄悄修改的设定。"
   - 用户同意 → 调用 novel-review 的模式 D（review-arc）
   - 用户跳过 → 在 `相关设定/_阶段回顾历史/skipped.md`（不存在则创建）登记跳过章号与时间，避免遗忘
   - 阶段回顾输出的"设定更新建议"由本 skill（针对实体设定）和 novel-setup（针对宏观文件）分别落地
8. **清理工作区**：删除 `工作区/第X章/` 下所有版本目录（**不动 `工作区/待定设定/`**）

### 定稿前自检清单

移入正文前，逐项确认：
- [ ] 至少跑过一轮 novel-review 审查循环
- [ ] P0 级问题全部处理或有明确理由保留
- [ ] 主角知晓状态无违规
- [ ] 本章涉及的实体设定已同步更新
- [ ] **本章涉及的待定设定已逐条处置**（晋升/保留/废弃），无 🟡 草案残留
- [ ] **`工作区/待定设定/_索引.md` 已同步**
- [ ] 至少明确了 1 个本章埋设/回收的伏笔
- [ ] 章末有钩子或明确的情绪落点

## 伏笔登记与回收（重要）

写作中发现"这里埋了个东西"或"这里回收了之前某个线"，**立即登记到本章的 `相关实体.md` 底部**：

```markdown
## 本章伏笔动作
- 新埋 F00X: [简称] — 位置：[段落] — 计划回收：[第Y章左右]
- 回收 F00X: [简称] — 回收方式：[简述]
- 强化 F00X: [简称] — 强化内容：
```

定稿时这些记录由状态回写流程同步到 `伏笔追踪.md`。

## 创作标准

### 标准文件结构

**好的示例.md**：
```markdown
# 好的示例积累

## 对话描写
- 示例: [引用正文片段]
- 原因: [为什么好]
- 适用场景: [什么情况下适用]

## 情绪渲染
...
```

**差的示例.md**：
```markdown
# 需避免的示例

## 信息堆砌
- 示例: [引用需改进内容]
- 问题: [具体问题]
- 改进方向: [如何避免]
```

### 标准积累方式

- 用户指出好的内容 → 提取到 `好的示例.md`
- 用户指出问题 → 提取到 `差的示例.md`
- 每次创作前查阅创作标准
- 每次讨论后更新创作标准

## 阶段回顾触发（每5章必触发） ⭐ 新增

### 定位

单章审查（review-text）只看本章；项目体检（review-canon）频率太低。两者之间需要一个**中频回看**机制：每写完 5 章，强制回看一次最近 5 章的正文与相关设定，判断是否有被剧情悄悄改写的设定、是否要演化当前冲突、是否要补登未登记的伏笔。这就是**阶段回顾**。

### 触发规则

- 定稿第 N 章后，若 `N % 5 == 0`，本 skill 必须主动提示触发阶段回顾
- 用户也可以主动请求"阶段回顾 / 回顾最近5章 / 跑一次 review-arc"
- 已跳过的章号登记在 `相关设定/_阶段回顾历史/skipped.md`
- 已完成的阶段回顾报告由 novel-review 写入 `相关设定/_阶段回顾历史/arc_review_第X-Y章_YYYYMMDD.md`

### 触发后本 skill 的职责

1. 收到 review-arc 报告后，**只落地实体设定相关的整改项**（`相关设定/角色|物品|技能/`）
2. 报告中标注"由 novel-setup 接手"的项（涉及大纲、当前冲突、主角档案、伏笔追踪、世界设定、写作注意事项）转给 novel-setup
3. 报告中"建议回改正文"的项需用户确认后再走单章修改流程
4. 全部落地后，在报告底部追加"## 落地记录"，标注每条整改项的处置结果

### 不强制阻塞下一章

阶段回顾建议触发但用户选择跳过时，下一章可继续创作；但**已跳过 ≥2 次**（即累积 10 章未做阶段回顾）时，本 skill 必须升级提示语调："强烈建议先做阶段回顾，否则设定与正文的同步差距会持续放大。"

## 设定更新检查

每次对话后思考：

1. **实体内容更新**：用户反馈中有无可更新的角色/物品/技能设定？
2. **知晓状态更新**：主角在本次剧情中是否获得了新认知？
   - "主角未知" → "主角已知"（记录知晓方式）
3. **信息对称检查**：是否存在读者知道但主角不知道的情况？
4. **待定设定登记**：本次对话是否产生了新的待定设定（命名/规则/边界/扩展）？是否需要新建或更新 `工作区/待定设定/` 下的条目？同步更新 `_索引.md`。
5. **待定设定晋升信号**：是否有 🟡 草案的项可升 🟠 待审？是否有 🟠 待审已被讨论清楚可直接晋升 🟢？
6. **当前冲突推进检查** ⭐：本次对话/章节是否推进了 `当前冲突.md` 中的某个节拍？是否触发了冲突演化？是否出现"3 章未推进任何节拍"的停滞信号？
7. **状态回写提醒**：本次对话若产生剧情推进，提醒定稿时触发 novel-setup 回写
8. **阶段回顾倒计时** ⭐：当前章号距离下一个 5 的倍数还差几章？接近时（剩 1 章）提前提醒用户准备阶段回顾

## 边界

- 不做项目级大纲规划（交给 novel-setup）
- 不做章节级审查（交给 novel-review）
- 不擅自修改 `故事大纲.md` / `主角档案.md` / `伏笔追踪.md` / `当前冲突.md`（由 setup 负责；本skill只在定稿回写时触发，且修改须基于审查/回顾报告或用户明确指令）
- 定稿前不得跳过审查循环（除非用户明确跳过）
- **不擅自把待定设定写入 `相关设定/`**：必须先经 novel-review 的「新设定审查」并由用户决策
- 不擅自跨界：待定项若属世界规则/伏笔/大纲范畴，晋升交给 novel-setup
