# 创作标准更新规范

## 更新触发时机

- 每次用户讨论后（见下"创作标准更新评估"）
- novel-review 审查循环结束后
- 阶段回顾（review-arc）结束后
- 用户主动要求"把这个加入好/差示例"

## 存放位置与边界

| 内容类型 | 存放文件 | 维护方 |
|---------|---------|--------|
| 跨场景通用写作优秀示范 | `创作标准/好的示例.md` | novel-lite / novel-review |
| 跨场景通用写作反面案例 | `创作标准/差的示例.md` | novel-lite / novel-review |
| 特定场景类型的风格正面示例 | `相关设定/创作风格/场景描写/<分片>.md` 的"正面示例"段 | novel-review 建议 → novel-setup / novel-style-extract 落地 |
| 特定场景类型的风格反面示例 | `相关设定/创作风格/场景描写/<分片>.md` 的"反面示例"段 | novel-review 建议 → novel-setup / novel-style-extract 落地 |
| 新的硬性写作约束（视角/信息对称/篇幅等） | `相关设定/写作注意事项.md` | novel-setup |

## 条目格式规范

**好的示例.md 新增条目**：
```markdown
## [分类标签，如：对话节奏 / 信息释放 / 情绪落点]

- 来源：第X章VX / 审查报告 review_xxx / 用户反馈
- 示例：[引用原文片段，≤200字]
- 原因：[为什么好，2-3句话，指出具体效果]
- 适用场景：[什么情况下适用，什么情况下不适用]
- 类似变体：[如有]
```

**差的示例.md 新增条目**：
```markdown
## [分类标签]

- 来源：第X章VX / 审查报告 review_xxx / 用户反馈
- 示例：[引用问题原文或重构反例，≤200字]
- 问题：[具体问题，指向可操作的改进方向]
- 改进方向：[如何避免，给出1-2个具体写法]
- 是否已回改：是 / 否（如已回改，标注回改版本）
```

## 更新流程

1. **识别**：从用户反馈或审查报告中识别可沉淀的内容
2. **分类**：判断属于"通用写作"还是"特定场景风格" → 决定写入 `创作标准/` 还是 `相关设定/创作风格/`
3. **去重**：查阅现有条目，避免重复；如为已有条目的变体，在"类似变体"中补充
4. **写入**：按条目格式规范追加到目标文件底部（或按分类重组）
5. **通知**：如写入内容涉及 `相关设定/创作风格/` 分片，提示 novel-setup 更新 `_索引.md`

## 审计与清理

- 每完成 10 条新增后，建议对 `创作标准/` 做一次轻量重组：合并同类条目、删除已回改且不再具警示意义的差示例
- 重组时保留原始来源和创建时间，便于追溯

## 创作标准更新评估（每次对话后必执行）

**每次与用户讨论后，无论本次讨论主题为何，必须执行以下评估：**

1. **用户是否指出了好的表达或示范？** → 提取到 `创作标准/好的示例.md`
2. **用户是否指出了写作问题、风格缺陷或反面案例？** → 提取到 `创作标准/差的示例.md`
3. **novel-review 发现的风格类问题是否在本次讨论中被确认？** → 按[反例存放边界]规则沉淀
4. **是否有新的跨场景通用写作约束浮现？** → 更新 `创作标准/好的示例.md` 或 `差的示例.md`
5. **本次讨论是否涉及创作标准本身的适用性争议？** → 在对应标准条目下追加"适用场景"或"例外说明"

> 评估结果若需写入创作标准，遵循上述条目格式规范和更新流程。
