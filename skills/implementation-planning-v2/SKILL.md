# 实施计划 v2

创建执行阶段真正会使用的计划。计划是本流程中最详细的产物：每个任务都必须小、具体、有顺序，并且能用真实证据独立验证。

## 先读取

计划前读取这些引用：

- `../project-development-v2-common/references/delivery-contract.md`
- `../project-development-v2-common/references/no-simulation-policy.md`
- `references/implementation-planning-confirmation-gates.md`
- `references/plan-quality-gates.md`
- `references/implementation-plan-template.md`
- 如用户要求评估、审查、进入执行前把关，使用 `project-development-review-v2`。

## 核心规则

- 必须从已确认的 `tech-design-v2` 文档开始。
- 保持可追踪链路：`R-* -> A-* -> C-* -> T-* -> V-*`。
- 计划阶段必须经过固定确认闸门。闸门确认优先级高于“少问问题”策略。
- 当垂直用户路径任务能降低集成风险时，优先按用户路径拆分。只有依赖关系要求时才按技术层拆分。
- 把隐藏的集成工作显式写进计划。默认不要排除 UI smoke、API 契约或构建验证。
- 计划必须先做覆盖审计，再拆任务。发现需求、验收、页面、服务边界或验证缺口时，先补方案或标记阻塞，不要产出粗计划。
- 计划必须先确认技术方案没有阻塞级审查发现；若技术方案缺少审查记录，先按 `project-development-review-v2` 的技术方案审查口径做最小审查。
- 涉及多服务、前后端、运行链路或数据追踪的需求，必须有最终闭环任务作为放行门槛。
- 每个任务必须包含明确的文件或目录、非目标和验证命令/步骤。
- 本阶段不写生产代码。

## 流程

1. 在 `workplace/<version>/tech-design/` 找到当前技术方案，或使用用户给定路径。
2. 提取所有 `R-*`、`A-*`、`C-*` 和 `V-*`。
3. 按 `references/plan-quality-gates.md` 做覆盖审计，列出需求、验收、页面/API/数据/服务边界、验证方式和缺口。
4. 执行确认闸门 G1：输入理解和覆盖审计确认。
5. 检查代码库，确认任务边界和依赖关系可执行。
6. 创建按顺序执行的 `T-*` 任务：
   - 一个任务通常对应一次聚焦实现。
   - 混合无关领域、无法清晰验证的任务必须拆分。
   - 无法单独完成或验证的任务应合并。
   - 前端页面、后端服务、数据迁移、运行链路、日志/追踪、最终闭环要拆成可验证任务，不要合并成一个粗任务。
7. 执行确认闸门 G2：任务拆分、顺序和依赖确认。
8. 每个任务写清：
   - 目标。
   - 所需上下文。
   - 文件/目录。
   - 步骤。
   - 服务/API/数据/UI 契约和上下游依赖。
   - 禁止的捷径。
   - 验证方式。
   - 完成证据。
9. 执行确认闸门 G3：验证方案、最终闭环和执行风险确认。
10. 按 `project-development-review-v2` 的计划审查口径做交付前自检，确认没有粗任务、漏任务或 mock/fallback 完成路径。
11. 保存计划目录：
   - `workplace/<version>/plan/YYYY-MM-DD-<slug>/index.md`
   - `workplace/<version>/plan/YYYY-MM-DD-<slug>/T-001-<name>.md`
   - 每个 `T-*` 一个任务文件。
12. 最终只请求用户确认是否可进入 `plan-execution-v2`。

## 任务质量标准

任务必须让另一个 agent 不依赖完整对话也能完成。任务文件要自含，但不能臃肿。

必填字段：

- `追踪`：关联的 `R-*`、`A-*`、`C-*`、`V-*`。
- `目标`：一句话目标。
- `文件`：具体路径或路径模式。
- `步骤`：有顺序的实现动作。
- `契约和依赖`：本任务提供什么、依赖什么、下游如何消费。
- `不要做`：捷径和范围边界。
- `验证`：命令和预期证据。
- `完成标准`：精确完成标准。
- `证据日志`：留给执行阶段填写。

## 反模拟计划规则

相关时要显式写入禁止项：

- 不得用静态假数据替代真实 API 响应。
- 不得把 fallback 路径当作主实现。
- 未检查真实渲染路径，不得标记 UI 完成。
- 未运行相关命令或说明 `NOT VERIFIED`，不得标记后端完成。

## 输出标准

使用 `references/implementation-plan-template.md`。

`index.md` 必须包含：

- 来源需求和技术方案路径。
- 确认记录。
- 覆盖审计表。
- 执行顺序。
- 带状态的任务表。
- 可追踪矩阵。
- 最终闭环/放行门槛。
- 全局验证命令。
- 未解决风险或阻塞。

每个 `T-*` 文件都要足够小，可直接用于执行。

## 交付前自检

检查：

- G1 到 G3 均有确认记录，或明确记录用户要求跳过哪一项及其风险。
- 每个 `C-*` 至少出现在一个任务中。
- 每个 `V-*` 至少出现在一个任务或全局验证中。
- 每个需求页面、API、数据实体、服务边界和用户路径都在任务中有落点。
- 多服务或前后端需求有最终闭环任务；没有时必须写明为什么不需要。
- 每个任务都有真实验证。
- 除非明确交付物就是文档、mock、stub 或清单，否则任务不能只靠这些内容完成。
- 计划足够具体，执行时不需要重新发明需求。
- 审查记录中没有未处理的阻塞级发现。
