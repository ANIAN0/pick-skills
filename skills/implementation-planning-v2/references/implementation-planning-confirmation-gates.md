# 实施计划确认闸门

计划阶段必须固定确认关键环节。计划一旦进入执行，返工成本会明显上升，因此不能完全由 agent 自主拆分。

## 总规则

- 每个闸门先给出候选结论，再请求用户确认。
- 用户确认后才能进入下一闸门。
- 如果用户要求跳过确认，必须记录“已跳过”和风险。
- 每个闸门最多列 3 个需要用户判断的点；其余作为假设或风险记录。

## G1：输入理解和覆盖审计确认

在拆任务前执行。

必须确认：

- 需求、验收、变更和验证 ID 是否提取完整。
- 覆盖审计是否发现缺口。
- 缺口处理方式是补方案、标记阻塞还是明确不做。

输出格式：

```markdown
### G1 覆盖审计确认
我提取到：
- 需求：R-...
- 验收：A-...
- 变更：C-...
- 验证：V-...

覆盖缺口：
- ...

请确认：覆盖审计是否准确？是否允许按这些缺口处理方式继续拆任务？
```

## G2：任务拆分、顺序和依赖确认

在写任务文件前执行。

必须确认：

- 任务边界是否合理。
- 执行顺序是否符合依赖。
- 是否需要垂直用户路径任务或最终闭环任务。
- 是否有任务过大或过粗。
- 每个任务是否有足够具体的执行断言，能检查执行后是否真的按任务完成。

输出格式：

```markdown
### G2 任务拆分确认
建议任务拆分：

| 任务 | 目标 | 依赖 | 覆盖 |
|------|------|------|------|

执行断言摘要：
- <T-001: 必须消失/出现/传递/落盘>

请确认：任务边界、顺序、依赖和执行断言是否可以进入详细计划？
```

## G3：验证方案、最终闭环和执行风险确认

在保存计划前执行。

必须确认：

- 每个任务的验证方式是否真实可执行。
- 每个任务的执行断言是否可以真实检查。
- 最终闭环/放行任务是否足够证明需求。
- `not_verified` 风险、`blocked` 或人工检查是否可接受。
- 是否存在执行风险。

输出格式：

```markdown
### G3 验证和风险确认
验证方案：
- ...

最终闭环：
- ...

执行风险：
- ...

请确认：这份计划是否可以写入正式任务清单并进入执行？
```

## 跳过规则

用户可以要求跳过某个闸门，但文档必须记录：

- 跳过哪个闸门。
- 跳过原因。
- 可能导致的风险。
- 后续哪个阶段补确认。
