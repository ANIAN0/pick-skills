# 计划质量门槛

实施计划必须比需求和技术方案更可执行。不要只生成模块名和粗略目标。

## 1. 先做覆盖审计

拆任务前，先列出覆盖审计表。至少检查：

- 每个 `R-*` 是否有任务落点。
- 每个 `A-*` 是否有验证落点。
- 每个 `C-*` 是否有实现任务。
- 每个 `V-*` 是否有命令、脚本、UI smoke 或可观察步骤。
- 需求中的每个页面、入口、状态、错误反馈是否有前端任务。
- 技术方案中的每个 API、数据实体、服务边界、迁移是否有任务。
- 涉及运行链路、历史事实源、日志、Memory/Knowledge、Tool、Agent 配置等跨模块能力时，是否有最终闭环任务。
- 技术方案中的每个阻塞级审查发现是否已解决或被用户接受。
- `V-*` 引用的 API、字段、页面、事件和命令是否存在，且有任务覆盖。

发现缺口时，不要用空泛任务填补。应先：

- 回到技术方案补 `C-*` 或 `V-*`；或
- 在计划中标记 `blocked`，说明缺少什么输入；或
- 明确写入不做范围和影响。

## 2. 任务拆分深度

任务不能只写“实现 X 服务”或“实现前端页面”。必须写清：

- 代码物理落点。
- 主要文件或目录。
- 上下游依赖。
- 对外提供的接口、数据或事件。
- 下游如何消费本任务产物。
- 验证命令或可观察证据。

如果一个任务覆盖多个用户路径、多个服务边界或多个页面，优先拆分。

## 3. 前端覆盖门槛

前端不能只写“页面可访问”。每个页面任务至少写：

- 路由。
- 入口。
- 关键组件。
- 数据来源。
- loading、空、错误、成功状态。
- 与后端 API 的字段契约。
- UI smoke 或自动化测试建议。

如果项目已有 Playwright、Vitest、组件测试或浏览器自动化能力，优先写可运行验证；subagent 审查只能作为补充。

## 4. 最终闭环任务

满足任一条件时，必须添加最终闭环任务：

- 涉及前后端。
- 涉及两个及以上服务。
- 涉及异步运行、日志、事件、会话、历史记录或状态追踪。
- 涉及 Tool、Agent 配置、Memory/Knowledge、外部服务或模型调用。

最终闭环任务要证明关键路径真实串联，而不是重复单模块测试。它至少检查：

- 同一业务 ID、run ID、session ID 或 trace ID 能贯穿关键模块。
- 前端能看到后端真实产生的数据或状态。
- 日志、事件、历史或审计记录可回查。
- 生产路径没有 mock、固定回复、硬编码测试 ID、测试旁路或降级实现冒充完成。

简单纯后端工具、纯文档或单文件改动可以不加最终闭环任务，但必须写明“不适用”的理由。

## 5. 状态和链接

- 新生成的计划默认状态应为 `todo`，除非已经有真实执行证据。
- 如果迁移已有计划，已完成任务必须链接到证据日志或说明完成来源。
- 所有需求、技术方案和任务文件链接必须按计划目录真实位置校验。

## 6. 计划交付前检查

交付前逐项回答：

- 计划是否覆盖所有需求和验收？
- 是否有任何任务需要执行者重新猜需求？
- 是否有前端页面或状态遗漏？
- 是否有跨服务闭环遗漏？
- 是否有只靠 mock、清单或手测口头描述就能“完成”的任务？
- 是否有链接或状态会误导执行者？
- 是否存在技术方案缺口被计划用“实现时再确认”掩盖？
- 是否每个任务都能由执行者按文档直接落地，而不是重新设计 API、数据或 UI？
