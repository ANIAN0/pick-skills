# 技术方案确认闸门

技术设计阶段必须固定确认关键环节。不要因为“少问问题”而跳过这些闸门。

## 总规则

- 每个闸门先给出 agent 的候选结论，再请求用户确认。
- 用户确认后才能进入下一闸门。
- 如果用户要求跳过确认，必须记录“已跳过”和风险。
- 闸门问题要聚焦决策，不要变成大段开放式提问。
- 每个闸门最多列 3 个需要用户判断的点；其余作为假设或风险记录。

## G1：需求理解和验收口径确认

在读取需求简报后执行。

必须确认：

- 本期范围是否准确。
- 验收标准是否代表用户真实想验证的结果。
- 不做范围和高风险假设是否可接受。

输出格式：

```markdown
### G1 需求理解确认
我理解本期目标是：...

关键验收口径：
- A-001: ...
- A-002: ...

我会按以下不做范围处理：
- ...

请确认：这个理解是否准确？是否有必须调整的范围或验收口径？
```

## G2：现状证据、差距和不变量确认

在代码探索后执行。

必须确认：

- 当前代码证据是否找对。
- 现状差距是否真实。
- 架构不变量是否准确。

输出格式：

```markdown
### G2 现状和不变量确认
我看到的关键现状：
- <文件/模块>: <当前行为> -> <问题>

我会把这些作为不可破坏的不变量：
- ...

请确认：现状判断和不变量是否准确？有没有我漏掉的关键文件或约束？
```

## G3：关键技术决策确认

在写变更契约前执行。

必须确认所有会影响后续计划的决策：

- 服务边界和部署形态。
- 数据归属和事实源。
- API 风格。
- 兼容策略。
- 失败隔离策略。

输出格式：

```markdown
### G3 技术决策确认
我建议采用以下决策：

| 决策 | 推荐方案 | 放弃方案 | 理由 | 影响 |
|------|----------|----------|------|------|

请确认这些决策是否成立。特别是：<列 1-3 个高影响决策>。
```

## G4：实现表面和契约确认

在服务/API/数据/UI 契约成形后执行。

必须确认：

- 每个服务的职责和禁止承担的责任。
- API 风格和关键接口是否完整。
- 数据事实源和写入方是否一致。
- 前端页面和状态是否覆盖需求。

输出格式：

```markdown
### G4 实现契约确认
服务/API/数据/UI 落点如下：
- 服务：...
- API：...
- 数据：...
- 页面：...

请确认：这些落点是否完整？是否有 API、数据、页面或服务边界遗漏？
```

## G5：验证方案和矛盾审计确认

保存技术方案前执行。

必须确认：

- 每个验收标准都有验证契约。
- 最终闭环验证能证明真实串联。
- 矛盾审计没有未处理的“不一致”。
- `NOT VERIFIED` 或 `blocked` 是否可接受。

输出格式：

```markdown
### G5 验证和矛盾审计确认
验证方案：
- V-001: ...
- V-999: ...

矛盾审计结果：
- 服务隔离 vs 部署形态：一致/不一致，处理方式...
- API 风格：一致/不一致，处理方式...
- 事实源：一致/不一致，处理方式...

请确认：这份方案是否可以写入正式技术文档并进入实施计划？
```

## 跳过规则

用户可以要求跳过某个闸门，但文档必须记录：

- 跳过哪个闸门。
- 跳过原因。
- 可能导致的风险。
- 后续哪个阶段补确认。

不能静默跳过。
