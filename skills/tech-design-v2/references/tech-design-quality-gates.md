# 技术设计质量门槛

技术设计阶段不是把需求翻译成接口和表，而是基于现有代码和约束做实现决策。

## 1. 代码证据先行

设计前必须给出代码证据表：

- 当前相关文件或模块在哪里。
- 当前行为是什么。
- 为什么不满足需求。
- 哪个需求或验收标准受影响。

如果目标模块是全新目录，也要说明旧代码中可参考或必须隔离的部分。

## 2. 架构不变量

中大型设计必须写出不变量。不变量是后续计划和执行不能破坏的规则。

示例：

- AISDK Service 不依赖 Workflow SDK。
- Session Service 是唯一历史事实源。
- Message Router 不执行 tool。
- 记忆抽取不得阻塞主回复链路。

不变量必须来自需求、风险或现有代码冲突，不要写泛泛原则。

## 3. 核心设计问题

识别会影响实现路径的问题，不要只罗列服务：

- 服务边界如何划分，为什么这样划？
- 谁拥有数据，谁只消费数据？
- 同一业务 ID 如何跨服务追踪？
- 失败时隔离还是阻断？
- 旧项目是迁移、复用、只读参考还是完全保留？
- API 风格是否需要统一，和现有代码是否冲突？

每个问题必须包含：

- 可选方案。
- 选择。
- 取舍理由。
- 影响。
- 需要用户确认时的待决策项。

## 4. 服务边界不是目录名

每个服务边界必须写清：

- 拥有的数据。
- 提供的能力。
- 消费的上游。
- 禁止承担的责任。
- 失败边界。
- 日志或事件要求。

如果服务没有独立数据或独立失败边界，要说明为什么仍然需要独立服务。

## 5. 数据和事实源

涉及会话、运行、日志、记忆、知识、配置等历史或状态时，必须说明：

- 谁是事实源。
- 谁能写入。
- 谁只能读取。
- 来源如何追踪。
- 分支、压缩、撤回或重试如何影响数据。

不要只写表字段。

## 6. 闭环验证设计

多服务、前后端或异步链路必须设计闭环验证：

- 从用户入口或 API 入口开始。
- 经过关键服务。
- 产生可查询数据、日志或事件。
- 前端或查询接口能看到结果。
- 能证明没有 mock、固定回复、硬编码测试 ID 或测试旁路。

闭环验证不能只重复单模块测试。

## 7. 矛盾审计

技术方案交付前必须做矛盾审计。不要让后文技术选择静默推翻前文需求约束。

至少检查：

- **服务隔离 vs 部署形态**：如果需求要求独立服务和故障隔离，而方案选择物理合并部署，必须说明隔离级别到底是进程级、模块级还是逻辑级；如果达不到需求，列为待决策。
- **事实源 vs 数据表设计**：如果需求要求统一事实源，不得保留多套并行历史表作为最终模型，除非有明确统一视图、迁移映射和写入规则。
- **API 风格一致性**：如果选择“统一 POST”或“RESTful”，全篇 API 列表必须一致；不能前面说统一 POST，后面大量使用 `GET /:id`、`DELETE /:id`。
- **能力覆盖一致性**：需求中的创建、查询、更新、禁用、启用、绑定、fork、压缩、抽取、来源追溯等动词，必须在 API、数据和验证中都有落点。
- **验证引用一致性**：验证步骤不能引用方案中没有定义的 API、表、字段、事件或任务。
- **删除 vs 禁用**：需求写“禁用”时，不能只设计 `DELETE`；必须说明禁用语义、状态字段和恢复/过滤规则。
- **异步能力**：如果方案要求异步抽取或后台任务，必须设计触发方式、任务状态、失败记录和查询入口。
- **兼容接口**：如果保留 OpenAI chat 兼容接口，必须说明它与内部 run/event/log 接口的关系，不能只列一个兼容入口。

审计结果必须写入技术方案。任何“不一致”只能有三种处理：

- 修改方案使其一致。
- 明确列为待用户确认的设计变更。
- 标记 `NOT VERIFIED` 或 `blocked`，说明缺少的输入。

不得把不一致留给实施计划阶段处理。

## 8. 契约完整性检查

对每条需求做动词提取，检查是否有完整契约。

示例：

- “查看、启用/禁用、绑定 Tool”需要列表 API、状态字段、启用/禁用 API、绑定关系、验证。
- “Agent 创建、更新、禁用”需要创建 API、更新 API、禁用状态字段、读取配置 API、验证。
- “撤回和压缩自动 fork”需要 branch 表/字段、fork API 或操作语义、压缩记录、查询 API、验证。
- “记忆异步抽取并追溯来源”需要抽取触发、任务状态、source conversation/branch/message 字段、查询 API、验证。

如果只能设计出一部分，必须把缺失部分写成风险或待决策，不得假装覆盖。
