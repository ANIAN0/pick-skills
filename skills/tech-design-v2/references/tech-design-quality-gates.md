# 技术设计质量门槛

技术设计阶段不是把需求翻译成接口和表，而是基于现有代码和约束做实现决策。

## 1. 代码证据先行

设计前必须给出代码证据表：

- 当前相关文件或模块在哪里。
- 当前行为是什么。
- 为什么不满足需求。
- 哪个需求或验收标准受影响。

如果目标模块是全新目录，也要说明旧代码中可参考或必须隔离的部分。

## 2. 技术依据清单

设计不能凭空指定库、协议、框架或旧模块扩展方式。以下情况必须写技术依据：

- 新技术、新依赖或新协议。
- 新服务、新模块或新目录。
- 旧功能扩展、复用、迁移或替换。
- 外部系统、MCP、队列、定时任务、存储、鉴权或兼容接口。

技术依据必须包含：

- 对象：技术、模块、服务或旧代码路径。
- 依据来源：官方文档、现有代码、已有接口、用户提供资料或明确假设。
- 与本需求的关系。
- 缺口：未知版本、未知能力、未验证兼容性或缺少环境。

缺少依据时，不得直接把它写成确定方案；只能列为待决策、风险或 `not_verified` 风险。

## 3. 架构不变量

中大型设计必须写出不变量。不变量是后续计划和执行不能破坏的规则。

示例：

- AISDK Service 不依赖 Workflow SDK。
- Session Service 是唯一历史事实源。
- Message Router 不执行 tool。
- 记忆抽取不得阻塞主回复链路。

不变量必须来自需求、风险或现有代码冲突，不要写泛泛原则。

## 4. 核心设计问题

识别会影响实现路径的问题，不要只罗列服务：

- 服务边界如何划分，为什么这样划？
- 谁拥有数据，谁只消费数据？
- 同一业务 ID 如何跨服务追踪？
- 失败时隔离还是阻断？
- 旧项目是迁移、复用、只读参考还是完全保留？
- API 风格是否需要统一，和现有代码是否冲突？

每个问题必须包含：

- 可选方案。
- 选择。
- 取舍理由。
- 影响。
- 需要用户确认时的待决策项。

## 5. 服务边界不是目录名

每个服务边界必须写清：

- 拥有的数据。
- 提供的能力。
- 消费的上游。
- 禁止承担的责任。
- 失败边界。
- 日志或事件要求。

如果服务没有独立数据或独立失败边界，要说明为什么仍然需要独立服务。

## 6. 数据和事实源

涉及会话、运行、日志、记忆、知识、配置等历史或状态时，必须说明：

- 谁是事实源。
- 谁能写入。
- 谁只能读取。
- 来源如何追踪。
- 分支、压缩、撤回或重试如何影响数据。

不要只写表字段。

## 7. Delta 变更契约

`C-*` 不是泛化 TODO，必须是计划阶段可以直接拆任务的变更契约。

每个 `C-*` 必须包含：

- 分类：后端、前端、数据、配置、迁移、测试支撑或运行链路。
- 类型：新增、修改、删除、重构、配置或迁移。
- 目标文件、模块、路由、组件、表、服务或命令。
- 当前状态：来自代码证据或明确说明为新增。
- 目标状态。
- 具体修改：粒度到字段、函数、接口、事件、页面状态、配置项或错误码。
- 关联 `R-*`、`A-*`、`V-*`。

必须提供 Delta 分类汇总，至少按后端、前端、数据、配置、迁移、运行链路分组。没有某类变更时写“无”，不要留空。

阻塞问题示例：

- `C-*` 只写“实现 Tool MCP Service”，没有文件、接口、字段、调用路径。
- 后端、前端、数据混在一个 `C-*`，导致计划阶段无法独立验证。
- 具体修改写“优化体验”“完善校验”，没有字段、状态、错误码或交互落点。

## 8. 前端覆盖

需求含页面、界面、日志、状态、用户反馈、配置页、列表页、详情页或操作入口时，技术方案必须有 UI 契约。

每个页面或视图必须写：

- 路由或入口。
- 对应 `R-*`、`A-*`、`C-*`、`V-*`。
- 页面状态：初始、loading、empty、error、success、disabled 等适用状态。
- 用户反馈：提示、错误、跳转、刷新、局部更新。
- API 使用：调用哪个接口、关键请求/响应字段、失败处理。
- 权限或不可用状态，如适用。

必须提供页面覆盖表：需求页面或用户入口 -> 技术方案 UI 契约 -> 验证契约。需求没有 UI 时，写“无 UI 变更”，并说明依据。

## 9. 验收场景和测试资产

每个 `A-*` 必须拆成一个或多个可验证场景。场景必须描述输入/前置条件、动作、预期结果和观测证据。

每个关键 `V-*` 必须写测试资产建议位置，默认交接给：

```text
workplace/<version>/test/
├── backend/unit/
├── backend/integration/
├── frontend/unit/
├── frontend/component/
├── contract/
├── smoke/
├── review/
└── fixtures/
```

技术方案阶段只规划资产，不创建测试文件。计划阶段必须细化为真实文件或脚本。

测试资产可以是：

- 单元测试。
- 集成或契约测试。
- UI smoke。
- 前端组件/页面测试。
- 审查清单，限于自动化不现实且必须有可观察条目。
- 手动步骤，限于有明确输入、预期证据和 `not_verified` 风险说明。

## 10. 闭环验证设计

多服务、前后端或异步链路必须设计闭环验证：

- 从用户入口或 API 入口开始。
- 经过关键服务。
- 产生可查询数据、日志或事件。
- 前端或查询接口能看到结果。
- 能证明没有 mock、固定回复、硬编码测试 ID 或测试旁路。

闭环验证不能只重复单模块测试。

## 11. 矛盾审计

技术方案交付前必须做矛盾审计。不要让后文技术选择静默推翻前文需求约束。

至少检查：

- **服务隔离 vs 部署形态**：如果需求要求独立服务和故障隔离，而方案选择物理合并部署，必须说明隔离级别到底是进程级、模块级还是逻辑级；如果达不到需求，列为待决策。
- **事实源 vs 数据表设计**：如果需求要求统一事实源，不得保留多套并行历史表作为最终模型，除非有明确统一视图、迁移映射和写入规则。
- **API 风格一致性**：如果选择“统一 POST”或“RESTful”，全篇 API 列表必须一致；不能前面说统一 POST，后面大量使用 `GET /:id`、`DELETE /:id`。
- **能力覆盖一致性**：需求中的创建、查询、更新、禁用、启用、绑定、fork、压缩、抽取、来源追溯等动词，必须在 API、数据和验证中都有落点。
- **页面覆盖一致性**：需求有页面或用户入口时，UI 契约、API 使用和验证必须逐项覆盖；不能只写“前端适配”。
- **验证引用一致性**：验证步骤不能引用方案中没有定义的 API、表、字段、事件或任务。
- **测试资产一致性**：验证契约中的命令、脚本、smoke 或审查清单必须有建议位置，并能交给计划阶段落地。
- **删除 vs 禁用**：需求写“禁用”时，不能只设计 `DELETE`；必须说明禁用语义、状态字段和恢复/过滤规则。
- **异步能力**：如果方案要求异步抽取或后台任务，必须设计触发方式、任务状态、失败记录和查询入口。
- **兼容接口**：如果保留 OpenAI chat 兼容接口，必须说明它与内部 run/event/log 接口的关系，不能只列一个兼容入口。

审计结果必须写入技术方案。任何“不一致”只能有三种处理：

- 修改方案使其一致。
- 明确列为待用户确认的设计变更。
- 标记 `not_verified` 风险或 `blocked`，说明缺少的输入。

不得把不一致留给实施计划阶段处理。

## 12. 契约完整性检查

对每条需求做动词提取，检查是否有完整契约。

示例：

- “查看、启用/禁用、绑定 Tool”需要列表 API、状态字段、启用/禁用 API、绑定关系、验证。
- “Agent 创建、更新、禁用”需要创建 API、更新 API、禁用状态字段、读取配置 API、验证。
- “撤回和压缩自动 fork”需要 branch 表/字段、fork API 或操作语义、压缩记录、查询 API、验证。
- “记忆异步抽取并追溯来源”需要抽取触发、任务状态、source conversation/branch/message 字段、查询 API、验证。

如果只能设计出一部分，必须把缺失部分写成风险或待决策，不得假装覆盖。

## 13. 字段级覆盖审计

不要只检查 `R-*` 是否“关联”到了 `C-*`。必须把需求和验收中的关键对象拆成字段、动作、状态和错误语义。

检查方法：

- 从每个 `R-*` 和 `A-*` 提取名词、动词、状态词和约束词。
- 名词必须能落到数据实体、API 字段、UI 状态或运行事件。
- 动词必须能落到 API、后台任务、用户操作或服务间调用。
- 状态词必须能落到字段、枚举、错误码、UI 状态或事件类型。
- 约束词必须能落到不变量、禁止责任、失败边界或验证契约。

典型阻塞问题：

- 需求说 Agent 配置包含 subagent 和压缩规则，但 create/update/config API 没有这些字段。
- 需求说 Knowledge 可追溯来源，但只有 memory_sources，没有 knowledge_sources 或通用 source 表。
- 需求说日志页按 run 查询，但 UI 契约没有日志页，API 也没有 run 级查询。
- 需求说兼容 OpenAI chat，但没有说明兼容路径、请求字段、响应流格式和内部 run/event 的映射。

## 14. 验证引用存在性

每个 `V-*` 交付前必须做引用检查：

- 引用的 API 必须在 API 契约中定义。
- 引用的字段必须在请求、响应或数据实体中定义。
- 引用的页面必须在 UI 契约中定义。
- 引用的事件、日志、任务状态必须有事实源和查询入口。
- 引用的命令或脚本必须有可执行位置，或明确标记 `not_verified` 风险。

如果验证步骤引用了方案中不存在的对象，不能进入计划阶段。
