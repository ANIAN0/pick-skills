# 小说章节审查

## 定位

四件套分工（详见novel-setup）：
- **novel-review（本skill）**：中观层（审）。单章审查、阶段回顾、设定体检。
- **novel-setup**：宏观层。项目启动、大纲、状态回写。
- **novel-lite**：中观层（写）。单章创作，4阶段流程。
- **novel-style-extract**：风格层。抽象化风格提取。

**审查原则**：不是打分，而是**用专业问题激发思考**，让作者在回答问题中发现可提升点。

---

## 四种审查模式

| 模式 | 简称 | 对象 | 触发场景 | 输出 |
|------|------|------|---------|------|
| 正文审查 | review-text | 章节正文+三件套 | 写完一稿、定稿前 | 审查报告（八维度问题） |
| 新设定审查 | review-pending | 待定设定草案 | 待定项≥3条、定稿前 | 设定决策报告 |
| 设定体检 | review-canon | 既有设定文件 | 卷末/百章里程碑 | 设定健康报告 |
| 阶段回顾 | review-arc | 最近5章正文+设定 | 每5章定稿后 | 阶段回顾报告 |

**默认推断规则**：
- "审查第X章" → review-text
- "审查待定设定" → review-pending
- "设定体检" → review-canon
- "阶段回顾" → review-arc

## 通用工作流程

1. **加载上下文**：按模式要求读取全部相关文件
2. **执行审查**：按维度产出问题/差异/建议
3. **产出报告**：写入固定路径，不改稿
4. **驱动循环**：用户逐题回应 → 调用novel-lite修订 → 询问是否再次审查
5. **沉淀经验**：循环结束时询问是否沉淀优秀表达/反面案例

## 与novel-lite的衔接

| 原则 | 说明 |
|------|------|
| 审查不改稿 | 只产出问题与建议，实际修改走novel-lite |
| 必看文档是审查依据 | 审查时读取02_必看文档.md，验证正文是否遵循 |
| 报告路径固定 | 正文审查：工作区/第X章VX/审查历史/ |
| 定稿前必审 | 移入正文前，至少跑一轮"定稿终审" |
| 待定项触发 | novel-lite累积≥3条待审项时建议跑模式B |
| 阶段回顾触发 | novel-lite在N%5==0时提示，由本skill跑模式D |

## 边界

- 不打分（80/100这种），质量是多维的
- 不替代用户判断，永远给方向不给结论
- 不主动改写原文，除非用户明确要求
- 不擅自修改任何设定文件；发现问题只在报告中提出
- 遇到设定冲突，先指出、再询问以哪边为准
- 模式D不擅自演化当前冲突.md，只产出建议
- 模式B不替用户做晋升；模式C不替用户改设定
- 审查风格一致性时，以必看文档中列出的规则为基准
- 若发现自检报告与正文实际不一致，标注为P0级问题
- 不直接生成风格知识库内容，由novel-style-extract执行

## 创作标准更新评估

每次讨论后执行：

| 评估项 | 处理 |
|--------|------|
| 发现优秀表达？ | 建议写入 `创作标准/好的示例.md` |
| 发现反面案例？ | 建议写入 `创作标准/差的示例.md` |
| 发现风格类问题？ | 建议写入对应风格分片 |
| 发现新约束？ | 建议写入 `写作注意事项.md` |

本skill负责"发现与建议"，不负责直接写入。写入规范见novel-lite的reference-创作标准更新规范.md。
