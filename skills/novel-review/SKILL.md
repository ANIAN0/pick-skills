# 小说章节审查

## 定位

四件套分工：
- **novel-setup**：宏观层。项目大纲、故事状态、伏笔追踪。
- **novel-lite**：中观层（写）。单章创作与修订，4阶段流程驱动。
- **novel-review（本skill）**：中观层（审）。单章审查与循环驱动。
- **novel-style-extract**：风格层。从参考小说中提取创作风格。

审查不是评分，而是**用专业问题激发思考**，让作者在回答问题的过程中发现情节、设定、逻辑、趣味、连续性的可提升点。

## 四种审查模式总览

| 模式 | 简称 | 对象 | 触发场景 | 输出 |
|------|------|------|---------|------|
| **正文审查** | review-text | 章节正文+三件套 | 写完一稿、修订后、定稿前 | 章节审查报告（八维度问题） |
| **新设定审查** | review-pending | `工作区/待定设定/` 中的草案 | 待定项累积≥3条、定稿前、用户主动请求 | 设定决策报告 |
| **设定体检** | review-canon | `相关设定/` 下既有设定 | 卷末/百章里程碑、用户主动请求 | 设定健康报告 |
| **阶段回顾** ⭐ | review-arc | 最近5章正文+相关设定 | **每5章定稿后必触发**（N%5==0） | 阶段回顾报告 |

**默认推断规则**：
- "审查第X章" → review-text
- "审查待定设定 / 看看哪些可以晋升" → review-pending
- "做次设定体检 / 检查一下设定" → review-canon
- "阶段回顾 / 回看最近5章" → review-arc

## 通用工作流程

每种模式的具体工作流程详见对应参考文档，总体遵循：

1. **加载上下文**：按模式要求读取全部相关文件
2. **执行审查**：按维度产出问题/差异/建议
3. **产出报告**：写入固定路径，不自动改稿
4. **驱动循环**：让用户逐题回应 → 调用 novel-lite 修订 → 询问是否再次审查
5. **沉淀经验**：循环结束时询问是否沉淀优秀表达/反面教材/新约束

## 与 novel-lite 的衔接

- **审查不改稿、不改设定**：只产出问题与建议，实际修改走 novel-lite
- **必看文档是审查的重要依据** ⭐：审查时读取 `02_必看文档.md`，验证正文是否遵循设定约束和风格要求；比对 `自检报告.md` 是否真实
- **报告路径固定**：
  - 正文审查：`工作区/第X章VX/审查历史/review_YYYYMMDD_HHMM.md`
  - 新设定审查：`工作区/待定设定/审查历史/pending_review_YYYYMMDD_HHMM.md`
  - 设定体检：`相关设定/_体检历史/checkup_YYYYMMDD_HHMM.md`
  - 阶段回顾：`相关设定/_阶段回顾历史/arc_review_第N-4章_第N章_YYYYMMDD_HHMM.md`
- **定稿前必审**：章节移入 `正文/` 之前，至少跑一轮正文"定稿终审"
- **待定项审视触发**：novel-lite 累积 ≥3 条 🟠 待审 项时建议跑模式 B；定稿前必跑
- **阶段回顾触发** ⭐：novel-lite 在第 N 章定稿后判断 N%5==0 时主动提示，由本 skill 跑模式 D

## 与 novel-setup 的衔接

- 审查结果涉及大纲偏离、伏笔状态、主角弧光变化 → 在报告"移交行动项"中标注需调用 novel-setup
- 模式 B 中，待定项归属为"世界规则/伏笔/大纲"的，晋升执行交给 novel-setup
- 模式 C 是 novel-setup「全局体检」的设定层补充
- 模式 D ⭐ 中"由 novel-setup 接手"的项（含当前冲突演化、大纲微调、主角弧光节点更新、伏笔补登/废弃、世界设定追加等）由 novel-setup 落地
- **模式 A 发现风格分片需要调整或补充** → 小问题建议 novel-setup 直接更新；系统性薄弱建议调用 **novel-style-extract**
- **模式 D 发现自检分数虚高或风格指纹持续漂移** → 建议 novel-setup 调整自检清单阈值或补充风格分片

## 与 novel-style-extract 的衔接

- 模式 A 审查风格一致性维度时，若发现风格分片的量化指标需要调整或反面示例需要补充：
  - 小问题 → 建议 novel-setup 直接更新对应分片
  - 系统性薄弱（缺失正例/指标不合理/与参考小说风格偏离）→ 建议调用 **novel-style-extract** 重新提取/修正对应分片
- 模式 D 评估风格指纹漂移时，若发现自检分数虚高或持续漂移，建议重新校准量化指标

## 边界

- 不做"打分"（80/100 这种），质量是多维的
- 不替代用户判断，永远给方向不给结论
- 不主动改写原文，除非用户明确要求
- 不擅自修改任何设定文件；发现问题只在报告中提出
- 遇到设定冲突，先指出、再询问以哪边为准
- 模式 D 不擅自演化 `当前冲突.md`，只产出建议
- 模式 B 不替用户做晋升；模式 C 不替用户改设定
- **审查风格一致性时，以 `02_必看文档.md` 中列出的设定约束和风格要求为基准**
- **若发现自检报告与正文实际审查结果不一致**，在报告中明确标注为P0级问题
- **不直接生成创作风格知识库内容**：风格分片的修正/补充建议由本skill提出，实际内容生成由 novel-style-extract 执行

## 创作标准更新评估

**每次与用户讨论后，无论本次讨论主题为何，必须执行以下评估：**

1. **本轮审查/讨论中是否出现了可沉淀的优秀表达？** → 按[反例存放边界]规则归入 `创作标准/好的示例.md` 或 `相关设定/创作风格/<分片>.md`
2. **本轮审查/讨论中是否出现了可沉淀的反面案例？** → 按[反例存放边界]规则归入 `创作标准/差的示例.md` 或 `相关设定/创作风格/<分片>.md`
3. **用户是否对某条风格规则或量化指标提出了质疑或修正建议？** → 评估是否需要调整 `相关设定/创作风格/` 分片或 `自检清单.md`
4. **本轮审查循环是否结束？** 如结束，确保所有可沉淀内容已按规范移交 novel-lite 或 novel-setup 写入对应文件

> 写入规范遵循 novel-lite 的[创作标准更新规范]。本 skill 负责"发现与建议"，不负责直接写入 `创作标准/` 或 `相关设定/创作风格/` 文件。
