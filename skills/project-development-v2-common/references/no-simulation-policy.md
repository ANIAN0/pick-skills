# 禁止模拟完成策略

不得把模拟、mock、fallback、stub 或未验证的工作描述成已完成实现。

## 硬规则

- 不得用静态样例数据替代需求要求的真实数据流。
- 不得用 stub 替代计划中的集成，除非计划明确写明交付物就是 stub。
- 可行时，未验证真实 UI 路径不得标记 UI 完成。
- 可行时，未运行相关命令、测试或本地调用不得标记后端完成。
- 不得伪造命令输出、截图、API 响应或测试结果。
- 不得用 fallback 掩盖验证失败。

## 允许的临时工作

只有同时满足以下条件时，才允许临时 mock 或 stub：

- 它是为了推动本地开发所必需。
- 明确标记为临时。
- 在真实集成完成前，或剩余工作被跟踪前，任务不能标记 `done`。
- 最终总结列出限制。

## 完成状态用语

使用准确状态：

- `done`：真实实现已完成，验证已通过。
- `not_verified`：实现已完成，但验证无法运行。
- `blocked`：实现无法继续。
- `partial`：计划工作仍有剩余。

不要用 `done` 表示 `not_verified`、`blocked` 或 `partial`。

## 如果验证无法运行

记录：

- 尝试过的命令或步骤。
- 具体错误或缺失前置条件。
- 仍缺少什么证据。
- 最安全的下一步。
