# 提问策略

少问，但要问关键问题。只有当答案会实质改变文档或实现路径时，才允许提问。

## 允许提问的条件

同时满足以下条件才提问：

- 答案无法从仓库、已有文档、历史对话或安全约定中获得。
- 错误假设会导致返工、用户可见错误、数据损坏、安全风险、兼容性破坏、明显成本或范围变化。
- 问题可以被用户具体回答。

## 不要提问

不要问：

- 不影响实现的偏好问题。
- 从上下文看答案明显的问题。
- 只是为了满足流程步骤的问题。
- “还有其他需求吗？”这类宽泛问题，除非已经发现明确缺口。
- 一次超过 3 个问题。

## 提问格式

提问时包含：

- 需要决策的点。
- 推荐默认方案。
- 选择不同方案的影响。

示例：

```markdown
在最终确定范围前，需要确认一个决策：

导入重复数据时，是覆盖已有记录还是跳过重复项？
推荐：跳过重复项，因为首版更安全。
影响：覆盖需要额外处理冲突和回滚行为。
```

## 记录假设

不提问时，明确记录假设：

```markdown
假设：沿用现有登录鉴权模型，因为相关路由目前都要求登录。
```
