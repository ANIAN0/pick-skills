# 交付契约

需求、方案、计划和执行阶段使用同一组 ID。这样可以保留关键可追踪性，同时避免在文档里维护大量重复映射表。

## ID

| 前缀 | 含义 | 创建阶段 |
|------|------|----------|
| `R-*` | 需求项 | requirements-workshop-v2 |
| `A-*` | 验收标准 | requirements-workshop-v2 |
| `D-*` | 技术决策 | tech-design-v2 |
| `C-*` | 代码或配置变更契约 | tech-design-v2 |
| `T-*` | 执行任务 | implementation-planning-v2 |
| `V-*` | 验证契约 | tech-design-v2 或 implementation-planning-v2 |
| `REV-*` | 审查发现 | project-development-review-v2 |

## 可追踪链路

维护这条链路：

```text
R-001 -> A-001 -> C-001 -> T-001 -> V-001
```

规则：

- 每个本期 `R-*` 必须至少有一个 `A-*`。
- 每个 `A-*` 必须至少有一个 `V-*`。
- 每个 `C-*` 必须关联至少一个 `R-*`，或标记为基础设施变更。
- 每个 `T-*` 必须关联它实现的 `C-*` 和 `V-*`。
- 每个阻塞级 `REV-*` 必须在进入下一阶段前解决、降级为已接受风险，或由用户明确确认跳过。
- 最终执行总结必须按 `T-*` 报告任务状态，按 `V-*` 报告验证状态。

## 版本目录

所有阶段写入工作产物前，必须把 `workplace/<version>/...` 解析为真实目录，不能把 `<version>`、`1.X` 或类似占位符写入持久化路径、任务命令或测试资产路径。

解析顺序：

1. 用户明确给出需求、技术方案、计划或测试目录路径时，沿用该路径中的版本目录。
2. 当前阶段有上游文档时，沿用上游文档所在的 `workplace/<version>/`。
3. 没有上游路径时，扫描 `workplace/` 下已有版本目录，选择用户当前讨论对应的版本；无法可靠判断时询问用户。
4. 新项目没有 `workplace/` 或版本目录时，创建新的具体版本目录，例如 `workplace/1/`，并在交接清单记录。

审查规则：

- 正式文档路径、计划路径、任务文件路径、测试资产路径和命令中出现 `<version>`、`1.X` 等占位符，默认是阻塞问题。
- 示例、模板或说明文字可以保留 `<version>`，但必须明确它不是实际路径。

## 状态值

计划和执行中使用这些状态：

- `todo`：未开始。
- `in_progress`：正在实现。
- `done`：已实现且已验证。
- `not_verified`：已实现但验证无法运行。
- `blocked`：缺少用户输入或外部条件，无法继续。
- `skipped`：明确跳过，必须说明原因和影响。

状态只能使用以上值。不要引入 `partial`、`NOT VERIFIED` 等替代状态。需要表达未验证风险时，使用 `not_verified` 状态，或在风险/待决策项中写“未验证风险”。

## 证据

证据必须具体：

- 执行的命令和关键输出。
- 测试名称和结果。
- 用户可见行为的浏览器或 UI 观察。
- 修改的文件或代码路径。
- 验证无法运行时的阻塞详情。

不要把“看起来没问题”或“应该可以”当作证据。
