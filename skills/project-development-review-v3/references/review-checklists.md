# 审查清单

## 通用审查

所有阶段都检查：

- 入口是否只有一份，内容是否足以支持用户判断，且没有把大量支撑正文塞进入口。
- 路径、编号、链接和状态是否真实、一致、无占位符，双向链接是否成立。
- 是否存在模糊描述、假设、矛盾、遗漏或未经证据支持的完成声明。
- `F-* -> C-* -> T-* -> V-*` 的相关链路是否覆盖实际范围。
- 调研任务是否已登记、报告是否回写、`completed` 结论是否由责任阶段吸收、`blocked` 是否保留恢复条件。
- 当前 skill 是否越权修改上游结论、自动确认或把未来能力当作现有能力。
- 阶段自检、独立审查和用户确认是否被混为一件事。
- 回退后受影响阶段状态、审查和用户确认是否已失效并重新完成，是否错误复用了修改前证据或确认。

## 需求

- 用户故事是否准确表达角色、场景、目标能力和结果。
- 功能点是否完整且每项用一至两句话说明是什么、做什么，没有实现设计。
- 每个 `F-*` 是否有可观察、覆盖关键失败或边界的验收要求。
- 涉及 UI 时是否有 UI 需求文档或入口链接；无 UI 时是否明确。
- 项目证据、核心机制、业务边界和用户强调约束是否在压缩入口时丢失。

## 方案

- 每个 `F-*` 和验收要求是否有 `C-*` 与 `V-*` 落点。
- 方案是否基于当前代码和必要调研，而不是凭经验指定模块、依赖、API 或表。
- 是否没有大段复述需求；需求内容应通过编号和链接追踪。
- 关键边界、事实源、数据、API、UI、失败、兼容和迁移是否一致。
- 每个 `C-*` 是否有真实位置、当前状态、目标修改和关联验证，足以拆任务。
- 每个 `V-*` 是否真实可执行并覆盖关键闭环。

## 任务

- 入口是否只保留覆盖、测试策略、任务顺序、依赖、闭环、风险和任务链接，没有塞入执行细节。
- 是否读取并审查入口链接的全部 `T-*`。
- 每个 `D-*`、`C-*` 和 `V-*` 是否有任务落点，没有额外范围。
- 每个任务是否完整自含，保留当前/目标状态、特别关注点、不做范围、真实文件、上下游契约和相邻接口摘要。
- 任务是否优先按可独立演示的纵向行为拆分；依赖是否无环且顺序合理。
- 测试场景是否包含输入、动作、预期、类型、资产和具体测试或检查内容；测试入口、mock 边界和路径是否符合项目事实。
- 每个循环是否先规划有效 RED/失败检查、最小 GREEN 和转绿后的 REFACTOR。
- 每个任务是否有必须消失、出现、传递、落盘的执行断言、失败判定和完成证据要求。
- UI、BUGFIX、真实依赖、失败路径、测试资产、特别关注点和最终闭环是否按适用范围进入任务。

## 执行

- 是否直接复用已确认任务入口和全部 `T-*`，没有创建独立执行入口。
- 入口任务状态和每个任务的执行记录是否与代码差异、命令结果和证据一致，规划契约是否保持未改写。
- `done` 是否有真实实现、真实依赖、成功及适用失败验证，而非 build/lint 或模拟实现。
- 计划要求的删除、出现、字段传递、持久化、UI 和测试资产是否真实落地。
- `not_verified`、`blocked`、失败重试和计划偏差是否被如实保留。
- 验收缺陷重开执行时是否有开放 `DEF-*`、双向链接、受影响任务回退、执行审查重置和重新确认。
- 全局与闭环验证是否运行，相关回归是否成立。

## 验收

- 每个 `F-*` 及其验收要求是否有独立验收结果和证据。
- 验收是否检查真实用户路径、关键失败、UI 需求、兼容和回归，而非复述执行报告。
- 缺陷是否有复现证据、影响和回退责任阶段；未解决问题是否阻止错误确认。
- 剩余风险是否明确由用户接受，不能由 Agent 默认接受。

## 写回边界

审查 skill 只创建或更新审查报告，并在入口增加或更新对应的 `review`、`plan_review` 或 `execution_review` 链接。禁止：

- 重写用户故事、技术决策、变更、任务、执行或验收结论。
- 修改阶段状态、任务状态或调研状态。
- 关闭责任阶段的问题或替用户确认。
- 为了通过审查降低完成标准。
