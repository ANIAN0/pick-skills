# 项目研发独立审查 v3

目标是判断当前阶段是否足以交给用户确认并进入下一阶段。优先发现会造成方向错误、遗漏、返工、虚假完成或无法验收的问题；不要重写入口文档。

## 默认流程

1. 确认输入：输入必须是真实阶段入口路径。读取入口、直接上游入口、相关支撑材料和能验证关键声明的项目证据。
2. 确认阶段：识别当前是需求、方案、任务、执行或验收审查，并读取该阶段完成条件。
3. 追踪证据：从入口结论追踪到支撑材料、上游条目、真实代码或运行证据，不能只相信文档自述。
4. 写审查报告：报告放在被审查入口旁的同名目录 `review/REV-编号-阶段英文-slug.md`，首次创建时使用 [审查报告模板](assets/review-report.md)。
5. 写回链接：普通阶段入口更新 `review`，任务入口的计划审查更新 `plan_review`，执行审查更新 `execution_review`。不要修改阶段正文、状态、任务状态或用户确认。
6. 复审：责任阶段修正后更新同一报告，保留每轮结论；修复条件满足才把发现标记为 `resolved`。

## 发现标准

严重级别：

- `BLOCKER`：继续会导致方向错误、关键范围遗漏、无法执行、虚假完成或无法验收。
- `HIGH`：很可能造成核心返工、用户可见缺陷或重要证据缺失。
- `MEDIUM`：影响局部可靠性或理解，不阻止进入下一阶段。
- `LOW`：整理建议，不影响阶段判断。

默认完整记录 `BLOCKER` 和 `HIGH`；`MEDIUM` / `LOW` 仅在影响用户判断或后续执行时记录。没有问题时明确写“未发现阻塞或高风险问题”，不要为了显得严格制造发现。

每个发现必须包含编号、严重级别、状态、责任位置、问题、影响和可验证修复条件。

## 下沉材料

按需读取 [审查清单](references/review-checklists.md)：

- 需要审查具体阶段的完整检查项时。
- 需要判断跨阶段追踪链路、回退失效、支撑材料或写回边界时。
- 发现可能属于 `MEDIUM` / `LOW`，需要判断是否值得记录时。

## 审查结论

报告状态只使用：

- `changes_required`：存在未解决 `BLOCKER` 或 `HIGH`。
- `passed`：没有未解决 `BLOCKER` 或 `HIGH`，且阶段完成条件有充分证据。

`passed` 只表示可以交给用户确认，不表示用户已经确认，也不自动推进阶段。

## 完成条件

- 审查对象和范围明确。
- 关键声明已查证，不只复述入口。
- 所有发现有具体责任位置、影响和可验证修复条件。
- 跨阶段链路已检查，结论与未解决发现一致。
- 入口和报告双向链接。
- 没有修改责任阶段结论、状态或用户确认。
