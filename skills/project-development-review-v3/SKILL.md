# 项目研发独立审查 v3

审查的目标是证明当前阶段是否足以交给用户判断和进入下一阶段，不是重写入口文档。优先发现会造成方向错误、遗漏、返工、虚假完成或无法验收的问题。

## 输入与报告

输入必须是一个真实阶段入口路径。执行审查使用已确认的任务入口，不要求独立执行入口。读取入口、直接上游入口、相关支撑材料和能够验证关键声明的项目证据；不能只相信文档自述。

报告放在被审查入口旁的同名目录中，使用 `review/REV-编号-阶段英文-slug.md`。首次审查创建一份报告并使用 [审查报告模板](assets/review-report.md)；修正后复审更新同一报告，保留每轮结论，不为每个发现或每轮复审创建新文档。

普通阶段入口 frontmatter 的 `review` 链接报告；任务入口的计划审查使用 `plan_review`，执行审查使用 `execution_review`。报告链接回入口。审查只允许更新对应审查链接，不得修改阶段正文、状态或用户确认。

## 审查方法

1. 确认审查阶段、入口状态、上游输入和本阶段完成条件。
2. 从入口结论追踪到支撑材料、上游条目、真实代码或运行证据。
3. 按当前阶段检查内容完整性、事实正确性、内部一致性、跨阶段覆盖、路径有效性和越界。
4. 先写发现，再写阶段结论。每个发现必须有编号、严重级别、责任位置、问题、影响、修复条件和状态。
5. 修正后重新检查责任位置和受影响链路；条件满足才把发现标记为 `resolved`。

严重级别：

- `BLOCKER`：继续会导致方向错误、关键范围遗漏、无法执行、虚假完成或无法验收。
- `HIGH`：很可能造成核心返工、用户可见缺陷或重要证据缺失。
- `MEDIUM`：影响局部可靠性或理解，但不阻止进入下一阶段。
- `LOW`：整理建议，不影响阶段判断。

报告默认完整记录 `BLOCKER` 和 `HIGH`；`MEDIUM`/`LOW` 仅在影响用户判断或后续执行时记录。没有问题时明确写“未发现阻塞或高风险问题”，不能为了显得严格制造发现。

## 通用审查

所有阶段都检查：

- 入口是否只有一份，内容是否足以支持用户判断且没有把大量支撑正文塞进入口。
- 路径、编号、链接和状态是否真实、一致、无占位符，双向链接是否成立。
- 模糊描述、假设、矛盾、遗漏和未经证据支持的完成声明是否存在。
- `F-* -> C-* -> T-* -> V-*` 的相关链路是否覆盖实际范围。
- 调研任务是否经审核、报告是否回写、结论是否由责任阶段吸收，未完成任务是否被错误关闭。
- 当前 skill 是否越权修改上游结论、自动确认或把未来能力当作现有能力。
- 阶段自检、独立审查和用户确认是否被混为一件事。
- 回退后受影响阶段状态、审查和用户确认是否已失效并重新完成，是否错误复用了修改前证据或确认。

## 阶段审查

### 需求

- 用户故事是否准确表达角色、问题和目标结果。
- 功能点是否完整且每项用一至两句话说明是什么、做什么，没有实现设计。
- 每个 `F-*` 是否有可观察、覆盖关键失败或边界的验收要求。
- 原型是否与功能和验收一致；无 UI 时是否明确。
- 项目证据、核心机制、业务边界和用户强调约束是否在压缩入口时丢失。

### 方案

- 每个 `F-*` 和验收要求是否有 `C-*` 与 `V-*` 落点。
- 方案是否基于当前代码和必要调研，而不是凭经验指定模块、依赖、API 或表。
- 关键边界、事实源、数据、API、UI、失败、兼容和迁移是否一致。
- 每个 `C-*` 是否有真实位置、当前状态、目标修改和关联验证，足以拆任务。
- 每个 `V-*` 是否真实可执行并覆盖关键闭环。

### 任务

- 入口是否只保留覆盖、测试策略、任务顺序、依赖、闭环、风险和任务链接，没有塞入执行细节。
- 是否读取并审查入口链接的全部 `T-*`；每个 `D-*`、`C-*` 和 `V-*` 是否有任务落点，没有额外范围。
- 每个任务是否完整自含，保留当前/目标状态、特别关注点、不做范围、真实文件、上下游契约和相邻接口摘要，执行者无需历史对话或重新设计。
- 任务是否优先按可独立演示的纵向行为拆分；依赖是否无环且顺序合理，基础设施或横向任务是否有真实依赖理由。
- 测试场景是否包含输入、动作、预期、类型、资产和具体测试或检查内容；测试入口、mock 边界和路径是否符合项目事实。
- 每个循环是否先规划有效 RED/失败检查、最小 GREEN 和转绿后的 REFACTOR；是否禁止批量先写测试。
- 每个任务是否有必须消失、出现、传递、落盘的执行断言、失败判定和完成证据要求。
- UI、BUGFIX、真实依赖、失败路径、测试资产、特别关注点和最终闭环是否按适用范围进入任务。

### 执行

- 是否直接复用已确认任务入口和全部 `T-*`，没有创建独立执行入口。
- 入口任务状态和每个任务的执行记录是否与代码差异、命令结果和证据一致，规划契约是否保持未改写。
- `done` 是否有真实实现、真实依赖、成功及适用失败验证，而非 build/lint 或模拟实现。
- 计划要求的删除、出现、字段传递、持久化、UI 和测试资产是否真实落地。
- `not_verified`、`blocked`、失败重试和计划偏差是否被如实保留。
- 验收缺陷重开执行时是否有开放 `DEF-*`、双向链接、受影响任务回退、执行审查重置和重新确认，没有直接复用历史确认。
- 全局与闭环验证是否运行，相关回归是否成立。

### 验收

- 每个 `F-*` 及其验收要求是否有独立验收结果和证据。
- 验收是否检查真实用户路径、关键失败、UI/原型、兼容和回归，而非复述执行报告。
- 缺陷是否有复现证据、影响和回退责任阶段；未解决问题是否阻止错误确认。
- 剩余风险是否明确由用户接受，不能由 Agent 默认接受。

## 写回边界

审查 skill 只创建或更新审查报告，并在入口增加或更新对应的 `review`、`plan_review` 或 `execution_review` 链接。禁止：

- 重写用户故事、技术决策、变更、任务、执行或验收结论。
- 修改阶段状态、任务状态或调研状态。
- 关闭责任阶段的问题或替用户确认。
- 为了通过审查降低完成标准。

责任阶段负责修正内容并请求复审。

## 审查结论

报告状态只使用：

- `changes_required`：存在未解决 `BLOCKER` 或 `HIGH`。
- `passed`：没有未解决 `BLOCKER` 或 `HIGH`，且阶段完成条件有充分证据。

`passed` 只表示可以交给用户确认，不表示用户已经确认，也不自动推进阶段。

## 完成自检

报告完成前检查：审查对象和范围明确；关键声明已查证；所有发现有具体责任位置、影响和可验证修复条件；跨阶段链路已检查；结论与未解决发现一致；入口和报告双向链接；没有修改责任阶段结论。

发现缺项先补充审查，不把审查完整性留给用户检查。
