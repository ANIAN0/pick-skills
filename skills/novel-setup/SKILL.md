# 小说项目启动与全局管理

## 定位

三件套分工：
- **novel-setup（本skill）**：宏观层。管"项目从0到1"与"长篇走向不失控"。
- **novel-lite**：中观层。管"单章怎么写、怎么定稿"。
- **novel-review**：中观层。管"单章怎么审、怎么循环"。

本 skill 不写正文，不审正文。只负责**让 novel-lite 每次开写新章节前，都有一张清晰的地图**。

## 项目文件结构（完整版）

novel-setup 负责创建和维护结构中的**宏观层文件**（⭐标记）：

```
project/
├── 相关设定/
│   ├── ⭐ 世界设定.md           # 世界观基底
│   ├── ⭐ 故事大纲.md           # 分卷/幕/章节的整体规划
│   ├── ⭐ 主角档案.md           # 主角的完整画像与成长轨迹
│   ├── ⭐ 故事状态.md           # 当前故事推进到哪里（动态快照）
│   ├── ⭐ 伏笔追踪.md           # 所有已埋/待回收的伏笔
│   ├── ⭐ 创作风格.md           # 文风基调
│   ├── ⭐ 写作注意事项.md       # 硬性约束（视角、信息对称等）
│   ├──   时间线.md（可选）      # 关键事件的时间轴
│   ├──   地图.md（可选）        # 地点/空间关系
│   ├── 角色/                    # 由 novel-lite 按需创建
│   ├── 物品/
│   └── 技能/
├── 工作区/                      # 由 novel-lite 管理
├── 正文/                        # 由 novel-lite 管理
├── 版本存档/                    # 由 novel-lite 管理
└── 创作标准/                    # 由 novel-lite 与 novel-review 共同维护
```

## 核心文件模板

### 1. 故事大纲.md

```markdown
# 故事大纲

## 一句话梗概
[30字内，故事的灵魂]

## 核心冲突
- 主要冲突：[主角 vs X，具体什么问题]
- 深层冲突：[价值观/身份/认知层面的对抗]

## 主题
[这个故事想表达什么，1-2句]

## 整体结构（按卷/幕划分）

### 第一卷：[卷名]（预计第1-X章）
- 核心事件：[卷级事件]
- 主角变化：[起点→终点]
- 关键章节：
  - 第1章：[一句话]
  - 第2章：[一句话]
  - ...
- 本卷末钩子：[引向下一卷的悬念]

### 第二卷：[卷名]
（同上）

## 主线与支线
- 主线A：[名称] — 状态：[推进中/待启动/已完结]
- 支线1：[名称] — 状态：...

## 结局方向
[大致走向，可多版本并存]
```

## 工作流程

### 场景1：全新项目冷启动

采用**访谈式引导**，分 5 轮完成：

**Round 1 — 故事灵魂**
询问并记录到 `故事大纲.md`：
1. 一句话梗概
2. 核心冲突是什么
3. 想表达的主题
4. 类型/题材

**Round 2 — 世界观**
询问并记录到 `世界设定.md`：
1. 世界规则（魔法/科技/超自然的底层逻辑）
2. 社会结构（势力/阶层/组织）
3. 地理与时间背景
4. 世界的"独特卖点"（区别于同类的设定）

**Round 3 — 主角**
询问并记录到 `主角档案.md`：
1. 基本画像
2. 性格底色
3. 起点状态与期望终点（弧光）
4. 最大软肋

**Round 4 — 大纲骨架**
询问并完善 `故事大纲.md`：
1. 预计多少卷/幕
2. 第一卷大致走向与关键事件
3. 结局方向（可模糊）

**Round 5 — 写作约束**
询问并记录到 `创作风格.md` 与 `写作注意事项.md`：
1. 文风偏好与参考作品
2. 视角规则
3. 信息对称规则
4. 章节长度目标

完成后生成初始化的 `故事状态.md`（指向"第0章，准备开写第1章"）和空白 `伏笔追踪.md`。

**原则**：每轮最多问 3-4 个问题，不把用户问崩。用户说"不确定"的项，先用占位"[待定]"，不阻塞进入下一轮。

### 场景2：每章开写前的"地图检查"（给 novel-lite 用）

用户要写第 N 章时，novel-setup 提供「开写前简报」：

```markdown
# 第N章 开写简报

## 本章在大纲中的定位
- 所在卷/幕：
- 预期事件：
- 对主线/支线的推进：

## 当前故事状态（摘自故事状态.md）
- 主角：[状态摘要]
- 地点/时间：
- 最近一章结尾悬念：

## 本章建议处理的伏笔
- 待回收：F00X（如时机合适）
- 建议新埋：[基于大纲预判]

## 角色在场预判
- 主角 + [可能出现的角色列表]
- 需要预先查阅的角色设定：

## 约束提醒
- 视角：
- 信息对称关键点：主角当前还不知道 [X]，切勿透露
```

### 场景3：章节定稿后的"状态回写"（给 novel-lite 用）

每当 novel-lite 把 `工作区/第X章VX/` 移入 `正文/第X章/` 后，调用本 skill 执行：

1. **更新 `故事状态.md`**：当前章节号、主角状态、活跃支线、下一章定位
2. **更新 `伏笔追踪.md`**：
   - 本章新埋的伏笔 → 新增条目（🔴）
   - 本章回收的伏笔 → 状态改为 🟢 并填回收位置
   - 本章强化/暗示的伏笔 → 备注追加
3. **更新 `主角档案.md`**：如主角在本章有弧光推进或认知变化
4. **检查大纲贴合度**：本章实际发生与 `故事大纲.md` 预期是否偏离，偏离则：
   - 偏离合理 → 更新大纲
   - 偏离不合理 → 标记并提示用户

### 场景4：定期全局体检

用户主动请求"检查故事进度/健康度"时：

产出「全局体检报告」：
- 大纲进度：已写 X/Y 章，当前在第几卷第几幕
- 主线推进：各主线百分比估算
- 伏笔健康度：🔴 待回收数、最老的待回收伏笔、超时未回收警告
- 主角弧光：当前处于哪个转折节点，是否停滞
- 风险项：支线过多/主线停滞/伏笔积压/角色失踪等

### 场景5：大纲调整

用户要改整体走向时：
1. 先出示当前大纲 + 已写章节清单
2. 评估改动影响面（哪些章节需要改、哪些伏笔会废）
3. 给出 3 种调整方案（激进/温和/最小改动）
4. 用户选定后更新 `故事大纲.md`，标记受影响章节，必要时更新 `伏笔追踪.md`（⚫ 已废弃）

## 与 novel-lite 的交接协议

| 时机 | novel-setup 动作 | novel-lite 动作 |
|------|-----------------|----------------|
| 项目启动 | 生成全部宏观设定文件 | 创建工作区/正文/等目录，等待第1章指令 |
| 开写第N章 | 出具「开写前简报」 | 基于简报开始创作，存入工作区 |
| 第N章定稿 | 执行「状态回写」 | 将工作区VX移入正文/第N章/ |
| 大纲变更 | 更新大纲+评估影响 | 按新大纲重新规划后续章节 |

## 与 novel-review 的交接协议

- novel-review 审查时可读取 `故事大纲.md`、`故事状态.md`、`伏笔追踪.md` 作为长线连续性维度的依据
- novel-review 发现的"大纲偏离"类问题 → 用户决策后，novel-setup 负责更新大纲或调整章节

## 三选项机制

涉及大纲/主角/世界观的重大变更时，必须给出 3 个方案：

```
**方案A**: [描述] — 影响：[范围] — 代价：[需改动的章节/伏笔]
**方案B**: ...
**方案C**: ...
**推荐**: 方案[X] — 原因：
```

## 边界

- 不写正文（交给 novel-lite）
- 不做章节级审查（交给 novel-review）
- 不主动发明大纲之外的剧情（只在用户要求时扩写大纲）
- 状态回写时，如与既有设定冲突，先报告后更新，不擅自覆盖
