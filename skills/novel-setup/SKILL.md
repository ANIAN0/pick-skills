# 小说项目启动与全局管理

## 定位

三件套分工：
- **novel-setup（本skill）**：宏观层。管"项目从0到1"与"长篇走向不失控"。
- **novel-lite**：中观层。管"单章怎么写、怎么定稿"。
- **novel-review**：中观层。管"单章怎么审、怎么循环"。

本 skill 不写正文，不审正文。只负责**让 novel-lite 每次开写新章节前，都有一张清晰的地图**。

## 项目文件结构（完整版）

novel-setup 负责创建和维护结构中的**宏观层文件**（⭐标记）：

```
project/
├── 相关设定/
│   ├── ⭐ 世界设定.md           # 世界观基底
│   ├── ⭐ 故事大纲.md           # 分卷/幕/章节的整体规划
│   ├── ⭐ 主角档案.md           # 主角的完整画像与成长轨迹
│   ├── ⭐ 故事状态.md           # 当前故事推进到哪里（动态快照）
│   ├── ⭐ 当前冲突.md           # 短期核心：眼下数章到一卷内的冲突靶心
│   ├── ⭐ 伏笔追踪.md           # 所有已埋/待回收的伏笔
│   ├── ⭐ 创作风格.md           # 文风基调
│   ├── ⭐ 写作注意事项.md       # 硬性约束（视角、信息对称等）
│   ├──   时间线.md（可选）      # 关键事件的时间轴
│   ├──   地图.md（可选）        # 地点/空间关系
│   ├── 角色/                    # 由 novel-lite 按需创建
│   ├── 物品/
│   └── 技能/
├── 工作区/                      # 由 novel-lite 管理（含"待定设定/"项目级共享池）
├── 正文/                        # 由 novel-lite 管理
├── 版本存档/                    # 由 novel-lite 管理
└── 创作标准/                    # 由 novel-lite 与 novel-review 共同维护
```

### 5. 当前冲突.md（短期核心，每章定稿后审视，每5章必更新）

> **定位**：`故事大纲.md` 是长期蓝图（卷/幕级），`故事状态.md` 是即时快照（当前章动态），`当前冲突.md` 是**两者之间的中期核心**——主角眼下数章到一卷内必须面对的"靶心"，是创作时的短期指南针。
>
> **为什么需要**：长期大纲太远，容易写飘；即时状态太碎，缺方向感。当前冲突回答："这几章主角到底在跟什么死磕？读者为什么要紧张？"
>
> **生命周期**：一个"当前冲突"通常持续 3-15 章，随关键节点（达成/失败/转化/升级）演化为下一个冲突。

```markdown
# 当前冲突

> 最后更新：第X章定稿后
> 本冲突起始于：第Y章
> 预计延续至：第Z章左右

## 一句话冲突
[主角 vs 什么 / 必须解决什么 / 否则会怎样，30字内]

## 冲突结构

### 表层冲突（眼下要解决的事）
- 对手/障碍：[具体的人/物/局面]
- 主角的目标：[要达成什么]
- 时限/压力：[多久内必须出结果，不解决会怎样]

### 深层冲突（驱动表层的内核）
- 价值观/身份层面：[主角内在的撕扯]
- 与主角弧光的关系：[本冲突如何推动主角的成长节点]

## 双方筹码与风险
- 主角持有：[资源、信息、关系、能力]
- 对手持有：[威胁、信息差、资源优势]
- 主角软肋（对手可能利用）：
- 对手软肋（主角可能利用）：

## 推进刻度（关键节拍）
> 用于判断每章是否在"啃这块骨头"，避免原地打转或跳跃推进

- [ ] 节拍 1：[事件/认知/对抗] — 预计第 Y+? 章
- [ ] 节拍 2：...
- [ ] 节拍 3：...
- [x] 已完成节拍：第 ? 章 [简述]

## 与其他文件的接口
- 关联伏笔（来自 `伏笔追踪.md`）：F00X、F00Y
- 涉及主角弧光节点（来自 `主角档案.md`）：[节点名]
- 涉及活跃支线（来自 `故事状态.md`）：[支线名]

## 演化预案
- 若主角达成 → 下一阶段冲突可能是：[简述]
- 若主角失败 → 下一阶段冲突可能是：[简述]
- 若中途转化（如对手变盟友、目标变质）→ 走向：[简述]

## 历史冲突链（按时间倒序）
- 第A章 — 第B章：[上一轮冲突简述] → 结局：[达成/失败/转化]
- ...
```

## 工作流程

### 场景1：全新项目冷启动

采用**访谈式引导**，分 5 轮完成：

**Round 1 — 故事灵魂**
询问并记录到 `故事大纲.md`：
1. 一句话梗概
2. 核心冲突是什么
3. 想表达的主题
4. 类型/题材

**Round 2 — 世界观**
询问并记录到 `世界设定.md`：
1. 世界规则（魔法/科技/超自然的底层逻辑）
2. 社会结构（势力/阶层/组织）
3. 地理与时间背景
4. 世界的"独特卖点"（区别于同类的设定）

**Round 3 — 主角**
询问并记录到 `主角档案.md`：
1. 基本画像
2. 性格底色
3. 起点状态与期望终点（弧光）
4. 最大软肋

**Round 4 — 大纲骨架**
询问并完善 `故事大纲.md`：
1. 预计多少卷/幕
2. 第一卷大致走向与关键事件
3. 结局方向（可模糊）

**Round 5 — 写作约束**
询问并记录到 `创作风格.md` 与 `写作注意事项.md`：
1. 文风偏好与参考作品
2. 视角规则
3. 信息对称规则
4. 章节长度目标

完成后生成初始化的 `故事状态.md`（指向"第0章，准备开写第1章"）、初始的 `当前冲突.md`（首段冲突，通常对应第一卷开篇 3-10 章的核心矛盾，可在 Round 4 大纲骨架成形后顺势提取）和空白 `伏笔追踪.md`。

**原则**：每轮最多问 3-4 个问题，不把用户问崩。用户说"不确定"的项，先用占位"[待定]"，不阻塞进入下一轮。

### 场景2：每章开写前的"地图检查"（给 novel-lite 用）

用户要写第 N 章时，novel-setup 提供「开写前简报」：

```markdown
# 第N章 开写简报

## 本章在大纲中的定位
- 所在卷/幕：
- 预期事件：
- 对主线/支线的推进：

## 当前故事状态（摘自故事状态.md）
- 主角：[状态摘要]
- 地点/时间：
- 最近一章结尾悬念：

## 当前冲突（摘自当前冲突.md）⭐
- 一句话冲突：
- 主角眼下的目标 / 时限：
- 本阶段已完成节拍：
- 本章应推进的下一节拍：[节拍编号 + 建议形态]

## 本章建议处理的伏笔
- 待回收：F00X（如时机合适）
- 建议新埋：[基于大纲预判]

## 角色在场预判
- 主角 + [可能出现的角色列表]
- 需要预先查阅的角色设定：

## 约束提醒
- 视角：
- 信息对称关键点：主角当前还不知道 [X]，切勿透露
```

### 场景3：章节定稿后的"状态回写"（给 novel-lite 用）

每当 novel-lite 把 `工作区/第X章VX/` 移入 `正文/第X章/` 后，调用本 skill 执行：

1. **更新 `故事状态.md`**：当前章节号、主角状态、活跃支线、下一章定位
2. **更新 `当前冲突.md`** ⭐：
   - 勾掉本章实际推进的节拍（`[ ]` → `[x]`），写入对应章号
   - 如本章触发冲突演化（达成/失败/转化），先把当前冲突卡片整体归入"历史冲突链"，再询问用户是否替换为下一段当前冲突，由用户确认后生成新冲突卡
   - 如已连续 3 章未推进任何节拍，提示用户"当前冲突疑似停滞，是否调整节拍或更换冲突"
3. **更新 `伏笔追踪.md`**：
   - 本章新埋的伏笔 → 新增条目（🔴）
   - 本章回收的伏笔 → 状态改为 🟢 并填回收位置
   - 本章强化/暗示的伏笔 → 备注追加
4. **更新 `主角档案.md`**：如主角在本章有弧光推进或认知变化
5. **检查大纲贴合度**：本章实际发生与 `故事大纲.md` 预期是否偏离，偏离则：
   - 偏离合理 → 更新大纲（同时检查 `当前冲突.md` 是否需要相应修订）
   - 偏离不合理 → 标记并提示用户
6. **接手待定项晋升**：处理 novel-lite 在定稿流程中标注的"待 novel-setup 处理"的待定项（按"待定设定接手协议"执行）
7. **阶段回顾触发判断** ⭐：若本章号 N 满足 N % 5 == 0，提示用户"建议跑一轮 novel-review 的阶段回顾（review-arc）"，由 review 输出的报告再驱动本 skill 对 `故事大纲.md` / `主角档案.md` / `当前冲突.md` / `伏笔追踪.md` 等宏观文件做进一步调整

### 场景4：定期全局体检

用户主动请求"检查故事进度/健康度"时：

产出「全局体检报告」：
- 大纲进度：已写 X/Y 章，当前在第几卷第几幕
- 主线推进：各主线百分比估算
- 伏笔健康度：🔴 待回收数、最老的待回收伏笔、超时未回收警告
- 主角弧光：当前处于哪个转折节点，是否停滞
- 待定设定池：规模、长期未处理项、是否建议触发 novel-review 模式 B
- 风险项：支线过多/主线停滞/伏笔积压/角色失踪/待定项积压等

**与 novel-review 模式 C 的分工**：
- 本场景看"进度与走向"——大纲贴合、弧光节奏、支线活跃、伏笔超期
- review-canon 看"设定文件本身"——一致性、时效性、完备性、文件健康度
- 重要节点（卷末、百章里程碑、定稿前）建议两者依次执行

### 场景5：大纲调整

用户要改整体走向时：
1. 先出示当前大纲 + 已写章节清单
2. 评估改动影响面（哪些章节需要改、哪些伏笔会废）
3. 给出 3 种调整方案（激进/温和/最小改动）
4. 用户选定后更新 `故事大纲.md`，标记受影响章节，必要时更新 `伏笔追踪.md`（⚫ 已废弃）

## 与 novel-lite 的交接协议

| 时机 | novel-setup 动作 | novel-lite 动作 |
|------|-----------------|----------------|
| 项目启动 | 生成全部宏观设定文件 | 创建工作区/正文/等目录，等待第1章指令 |
| 开写第N章 | 出具「开写前简报」 | 基于简报开始创作，存入工作区 |
| 第N章定稿 | 执行「状态回写」 | 将工作区VX移入正文/第N章/ |
| 大纲变更 | 更新大纲+评估影响 | 按新大纲重新规划后续章节 |

## 与 novel-review 的交接协议

- novel-review 审查时可读取 `故事大纲.md`、`故事状态.md`、`当前冲突.md`、`伏笔追踪.md` 作为长线连续性维度的依据
- novel-review 发现的"大纲偏离"类问题 → 用户决策后，novel-setup 负责更新大纲或调整章节
- novel-review **模式 B（新设定审查）** 报告中标注"由 novel-setup 接手"的条目（涉及世界规则/伏笔/大纲），由本 skill 执行晋升
- novel-review **模式 C（设定体检）** 报告中"由 novel-setup 接手"的整改项，由本 skill 执行修订
- novel-review **模式 D（阶段回顾）** ⭐ 报告中"由 novel-setup 接手"的整改项（大纲微调、当前冲突演化、主角弧光节点修订、伏笔重新规划等），由本 skill 执行；阶段回顾每 5 章触发一次，是宏观层与正文同步的主要机会

## 待定设定接手协议（与 novel-lite 共建）⭐ 新增

`工作区/待定设定/` 由 novel-lite 维护，但其中**部分类别**的晋升由本 skill 负责，避免 novel-lite 越界改宏观文件。

### novel-setup 必须接手的待定项类型

| 待定项归属 | 晋升目标 | 谁执行 |
|-----------|---------|--------|
| 角色（普通配角/路人）| `相关设定/角色/X.md` | novel-lite |
| 物品 | `相关设定/物品/X.md` | novel-lite |
| 技能 | `相关设定/技能/X.md` | novel-lite |
| **世界底层规则** | `相关设定/世界设定.md` | **novel-setup** |

### 接手晋升流程

收到 novel-review 报告或 novel-lite 转交时：

1. 读取待定项文件 `工作区/待定设定/<类别>/<名称>.md` 与决策日志
2. 比对宏观文件相关章节，确认无冲突；如有冲突，再次询问用户
3. 在目标宏观文件中合并新内容（保留原结构）
4. 在待定项的"决策日志"中追加：`YYYY-MM-DD：晋升至 [目标文件] § [章节]`
5. 由 novel-lite 从待定池删除该文件，更新 `_索引.md`（本 skill 不直接动 lite 的池）
6. 如涉及伏笔登记，同步更新 `故事状态.md` 中的"未解决悬念"

### 边界

- 本 skill **不直接读写** `工作区/待定设定/` 的文件结构（增删文件由 novel-lite 执行），只负责把内容合并进宏观文件后通知 lite 清理
- 接手前如发现待定项与既有宏观设定有不可调和冲突，回退到 novel-review 模式 B 重新评估，不擅自选边

## 三选项机制

涉及大纲/主角/世界观的重大变更时，必须给出 3 个方案：

```
**方案A**: [描述] — 影响：[范围] — 代价：[需改动的章节/伏笔]
**方案B**: ...
**方案C**: ...
**推荐**: 方案[X] — 原因：
```

## 边界

- 不写正文（交给 novel-lite）
- 不做章节级审查（交给 novel-review）
- 不主动发明大纲之外的剧情（只在用户要求时扩写大纲）
- 状态回写时，如与既有设定冲突，先报告后更新，不擅自覆盖
- 不直接读写 `工作区/待定设定/` 的文件结构（由 novel-lite 维护），只接手内容并通知 lite 清理
- 接手晋升前必须确认待定项已经过 novel-review 模式 B 审查，否则回退
