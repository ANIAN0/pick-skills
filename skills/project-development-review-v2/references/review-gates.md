# 审查门槛

审查不是润色文档，而是判断产物能否支撑下一阶段。

## 1. 阻塞判定

出现以下任一情况，默认标为 `BLOCKER`：

- 需求中的核心能力没有设计落点。
- 技术方案的验证引用了不存在的 API、字段、表、页面、事件或命令。
- 计划任务没有覆盖关键 `C-*` 或 `V-*`。
- 需求要求真实闭环，但方案或计划只提供 mock、stub、fallback、固定回复或硬编码 ID。
- 服务隔离、事实源、兼容接口、数据归属等关键约束在文档内互相冲突。
- 前端验收要求页面状态或日志定位，但 UI 契约和任务没有覆盖。

## 2. 字段级覆盖

审查时必须拆解需求句子：

- 名词：对象、页面、服务、数据、来源。
- 动词：创建、查询、更新、禁用、绑定、fork、压缩、抽取、追溯。
- 状态：loading、empty、error、success、enabled、disabled、failed、running。
- 约束：兼容、隔离、唯一事实源、异步、不阻塞、可回查。

只写“关联 R-002”不算覆盖。必须能指出 API 字段、数据字段、UI 状态、事件类型、验证步骤或任务文件。

## 3. 技术方案可实施性

技术方案必须让计划阶段能直接拆任务。以下情况说明不可实施：

- 只写服务名，没有物理代码落点。
- 只写 API 名，没有请求、响应、错误和字段。
- 只写数据表名，没有事实源、写入方、读取方和迁移规则。
- 只写 UI 页面名，没有状态、用户反馈和 API 使用。
- 只写验证目标，没有可执行命令、可观察证据或 `NOT VERIFIED` 标记。

## 4. 计划可执行性

计划必须让执行阶段不重新设计。以下情况说明计划过粗：

- 一个任务覆盖多个服务、页面和数据模型，且不能独立验证。
- 任务没有文件路径或路径模式。
- 任务没有上下游依赖和产物。
- 任务没有真实验证方式。
- 最终闭环缺失，或只重复单模块测试。

## 5. 审查结论

结论只能是：

- `可进入下一阶段`：没有阻塞问题，高风险已说明。
- `修正后可进入下一阶段`：存在 `HIGH` 或少量明确可修的问题。
- `不可进入下一阶段`：存在 `BLOCKER`。

不要给含糊结论。
