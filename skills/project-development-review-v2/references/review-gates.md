# 审查门槛

审查不是润色文档，而是判断产物能否支撑下一阶段。

## 1. 阻塞判定

出现以下任一情况，默认标为 `BLOCKER`：

- 需求中的核心能力没有设计落点。
- 需求文档的本期 `R-*` 没有用户可见闭环，缺入口、操作、成功可见结果、失败反馈或结果可见位置。
- 需求包含页面、入口、日志、状态或反馈，但没有页面/入口覆盖表；或无 UI 时没有可观察入口声明。
- 需求的 `A-*` 没有验收关注点，导致技术方案无法稳定拆出验证场景。
- 需求范围把本期必须做、后续候选和不做范围混在一起。
- Agent/AI 类需求没有说明任务类型、Agent/LLM 选择理由、角色边界、工具需求、自主度或失败风险。
- 技术方案的新技术、新模块或旧功能扩展没有技术依据，且没有列为风险、待决策或 `not_verified` 风险。
- 技术方案的 `C-*` 只有抽象模块名，没有分类、类型、当前状态、目标状态、具体修改和关联 `R/A/V`。
- 技术方案没有把 `A-*` 拆成可验证场景，导致 `V-*` 只是在复述验收标题。
- 技术方案的验证引用了不存在的 API、字段、表、页面、事件或命令。
- 技术方案有页面、用户入口、日志、状态或反馈要求，但缺少逐页 UI 契约或页面覆盖表。
- 技术方案的关键 `V-*` 没有测试资产建议位置，计划阶段无法稳定落地到 `workplace/<version>/test/`。
- 计划任务没有覆盖关键 `C-*` 或 `V-*`。
- 计划没有测试资产规划，或关键 `V-*` 没有测试文件、验证脚本、UI smoke 或可观察步骤落点。
- 计划中的关键任务没有测试场景表或具体验收内容。
- 执行结果把任务标为 `done`，但缺少 done 前硬门禁、真实依赖接入、负向验证、测试资产合规或独立审核证据。
- 执行结果的 `index.md` 状态、任务文件状态和证据日志互相矛盾。
- 技术方案包含 UI，但计划没有前端任务且未声明无 UI。
- 需求要求真实闭环，但方案或计划只提供 mock、stub、fallback、固定回复或硬编码 ID。
- 执行代码中保留 mock、stub、fallback、固定回复、硬编码测试 ID、空校验或内存替代持久化，并被用作完成路径。
- 执行结果新增测试资产不在计划指定测试目录，且没有计划例外。
- `not_verified`、`blocked` 或 `skipped` 被包装成整体完成。
- 服务隔离、事实源、兼容接口、数据归属等关键约束在文档内互相冲突。
- 前端验收要求页面状态或日志定位，但 UI 契约和任务没有覆盖。
- 需求特别关注点没有进入技术方案、计划或任务，导致执行阶段无法继承用户强调的约束。
- 需求文档没有记录用户明确强调的特别关注点，或特别关注点空泛到无法指导后续设计和执行。

## 2. 字段级覆盖

审查时必须拆解需求句子：

- 名词：对象、页面、服务、数据、来源。
- 动词：创建、查询、更新、禁用、绑定、fork、压缩、抽取、追溯。
- 状态：loading、empty、error、success、enabled、disabled、failed、running。
- 约束：兼容、隔离、唯一事实源、异步、不阻塞、可回查。

只写“关联 R-002”不算覆盖。必须能指出 API 字段、数据字段、UI 状态、事件类型、验证步骤或任务文件。

## 3. 需求文档可设计性

需求文档必须让技术方案阶段不重新做需求澄清。以下情况说明不可进入技术设计：

- 需求写成模块或服务清单，无法看出用户能力或系统能力。
- 本期 `R-*` 没有完整闭环，无法判断成功和失败如何被观察。
- 页面、入口、状态、日志或反馈是验收的一部分，但需求没有覆盖表。
- 验收标准只写“功能可用”“模块完成”，没有后端事实、前端可见性、失败异常、数据追溯、兼容或隔离关注点。
- 范围没有 MVP 分层，后续候选混进本期，导致技术方案被迫过度设计。
- Agent/AI 需求只写“用 AI 实现”，没有任务、角色、工具、自主度和失败风险。
- 特别关注点没有被记录，后续阶段无法继承用户强调的约束。

## 4. 技术方案可实施性

技术方案必须让计划阶段能直接拆任务。以下情况说明不可实施：

- 新技术、新模块或旧功能扩展没有依据来源。
- 只写服务名，没有物理代码落点。
- 只写 API 名，没有请求、响应、错误和字段。
- 只写数据表名，没有事实源、写入方、读取方和迁移规则。
- 只写 UI 页面名，没有状态、用户反馈和 API 使用。
- 只写变更目标，没有 `C-*` 分类、变更类型、当前状态、目标状态和具体修改。
- 只写验收编号，没有验收场景、输入、动作、预期结果和观测证据。
- 只写验证目标，没有可执行命令、可观察证据或 `not_verified` 风险标记。
- 只写验证契约，没有测试资产建议位置。

## 5. 计划可执行性

计划必须让执行阶段不重新设计。以下情况说明计划过粗：

- 一个任务覆盖多个服务、页面和数据模型，且不能独立验证。
- 任务没有文件路径或路径模式。
- 任务没有上下游依赖和产物。
- 任务没有真实验证方式。
- 最终闭环缺失，或只重复单模块测试。
- 没有统一测试资产目录，测试文件落点由执行者临时决定。
- 没有最终审核或独立任务审核要求，导致 `done` 只能依赖实现者自述。
- 任务文件依赖外部上下文才能执行，例如用“参见 index.md”“参见其他任务”省略目标、契约、测试场景或特别关注点。
- 没有相邻接口摘要，导致执行者无法判断上下游字段、事件或状态是否匹配。
- BUGFIX 计划没有关联根因、风险点、回退方案或回归验证任务。

## 6. 执行结果可信性

执行结果必须能证明计划任务真实完成。以下情况说明 `done` 不成立：

- 没有证据日志，或证据日志只写结论不写命令、关键输出、修改文件和观察结果。
- 没有 done 前硬门禁记录。
- 只跑了 build/typecheck/lint，却没有验证任务承诺的 API、数据、MCP、diagnostics、UI 或跨服务链路。
- 任务要求持久化、日志、事件或 diagnostics，但没有写入后查询证据。
- 任务要求失败隔离或错误处理，但只验证 happy path。
- 反模拟扫描没有执行，或命中项解释不能证明它不是完成捷径。
- 独立任务审核只复述实现报告，没有重新对照 `R/A/C/V` 和关键代码路径。
- 最终闭环缺失，或无法证明同一业务 ID、run ID、session ID 或 trace ID 贯穿关键模块。

## 7. 版本路径和状态

- 正式文档路径、计划路径、任务文件路径、测试资产路径或命令中出现 `<version>`、`1.X` 等占位符时，默认阻塞。
- 状态只能使用 `todo`、`in_progress`、`done`、`not_verified`、`blocked`、`skipped`。
- `done` 必须表示已实现且已验证；未验证只能是 `not_verified` 风险或状态。

## 8. 审查结论

结论只能是：

- `可进入下一阶段`：没有阻塞问题，高风险已说明。
- `修正后可进入下一阶段`：存在 `HIGH` 或少量明确可修的问题。
- `不可进入下一阶段`：存在 `BLOCKER`。

不要给含糊结论。
