# 计划执行 v3

执行阶段复用规划入口，不创建执行入口。任务入口给用户看进度；每个 `T-*` 保留规划契约，并只在 `## 执行记录` 写入实际结果。

## 入口门槛

只接受同时满足以下条件的任务入口：

- `status: confirmed`，且 `plan_review` 已通过。
- `execution_status` 为 `pending` 或 `executing`；或验收入口有开放 `DEF-*` 指向本入口、责任阶段为执行，允许按“缺陷回退”重开。
- 任务表每行链接一个真实、自含的 `tasks/T-*.md`。
- 没有未解决阻塞、未解释偏差或会推翻规划契约的发现。

开始执行时把入口 `execution_status` 改为 `executing`。入口任务表、任务 frontmatter 和执行记录只使用 `pending`、`in_progress`、`done`、`not_verified`、`blocked`。

入口旁复用同名目录：

```text
<任务入口同名目录>/
├── tasks/T-*.md
├── evidence/EVD-*.md
├── logs/LOG-*.md
├── research/RT-*.md
└── review/REV-*.md
```

原始命令输出、日志、截图和测试报告放支撑材料；入口和任务执行记录只写摘要、状态和证据链接。

## 读取与写回

先读任务入口，确认范围、测试策略、顺序、依赖、风险和当前状态。选择第一个依赖已完成且状态不是 `done` 的任务，只读取该 `T-*` 及其直接链接的方案、调研或证据；不要一次加载全部任务细节。

执行前核对当前任务的 `F/D/C/V` 追踪、文件、接口、测试场景、循环、执行断言和完成条件。任务不自含、路径不存在且未标明新增、接口契约矛盾、缺少有效 RED/失败检查，或要求重新设计时，记录发现并返回 `implementation-planning-v3` 修订。

执行只允许写这些位置：

- 入口 frontmatter：`execution_status`、`execution_review`。
- 入口任务表：当前 `T-*` 的执行状态。
- 当前任务 frontmatter：`execution_status`。
- 当前任务 `## 执行记录`：修改摘要、RED/GREEN/REFACTOR、验证与演示、执行断言、证据链接、偏差和阻塞。

不得改写入口的覆盖、测试策略、任务名称、交付结果、依赖和风险；不得改写任务文件中 `## 执行记录` 之前的规划契约。计划有错必须回到责任阶段。

## 单任务循环

1. 将入口表格和当前任务状态改为 `in_progress`。
2. 读取相关代码、项目规则和现有测试，确认基线没有发生会使任务失效的变化。
3. 按任务内循环逐个推进：先写当前测试或检查并运行 RED，确认只因目标行为缺失而失败并记录证据。RED 意外通过时确认行为是否已存在或任务是否失效；因环境、语法、导入或 fixture 失败时先修正测试条件，不能进入 GREEN。
4. RED 有效后，只实现当前 GREEN 声明的最小变化，运行同一命令转绿；不得提前实现后续循环或改弱测试。
5. 只在 GREEN 后做当前 REFACTOR；每次整理后重跑当前和相关既有测试，再进入下一个 RED。
6. 完成真实接入。不得用 mock 自有模块、stub、fallback、固定返回、样例数据、临时内存存储或空校验替代计划行为；mock 只限任务允许的外部边界。
7. 所有循环完成后运行独立演示、失败路径、邻近回归和执行断言，把摘要与证据链接写入执行记录。
8. 验证全部通过写 `done`；实现完成但验证因外部环境、权限、账号、数据或服务无法运行写 `not_verified`；当前无法继续写 `blocked`。

循环必须保持 `RED1 -> GREEN1 -> RED2 -> GREEN2` 顺序。迁移、配置、构建和基础设施任务按“失败检查 -> 最小变更 -> 通过检查 -> 整理回归”执行。

build、lint、typecheck 只能证明基础质量。任务承诺功能、持久化、接口、失败行为或 UI 时，必须完成对应行为验证才能 `done`。

## 失败与偏差

验证失败时在执行记录链接证据，写明命令、关键错误、根因判断、修复和重试结果。同一问题连续三轮仍不通过且当前无法继续时标记 `blocked` 并说明恢复条件；仍在修复时保持 `in_progress`。

只有实现已完成、验证本身因外部条件无法运行时才能标记 `not_verified`。测试断言失败、实现错误或持续红灯不得改写为 `not_verified`。

恢复执行从任务入口状态开始。已 `done` 且证据仍有效的任务不重跑；代码变化或审查推翻证据时回退任务状态并重新验证。事实不明但不改变任务契约时，在任务执行记录登记 `RT-*` 并调用 `project-research-v3`。文件、命令、接口、步骤、执行断言或范围与项目不符时，返回规划阶段；影响需求、架构、数据、兼容、安全或用户体验时，返回相应责任阶段；执行者不得静默偏离。

## 缺陷回退

验收入口的开放 `DEF-*` 明确责任阶段为执行，且修复不改变已确认规划契约时：

1. 核对缺陷、复现证据、影响任务和验收入口的双向链接。
2. 把入口 `execution_status` 从 `confirmed` 改为 `executing`，把 `execution_review` 重置为 `pending`；只把受影响任务回退为 `in_progress`，未受影响且证据仍有效的任务保持 `done`。
3. 在受影响任务执行记录链接 `DEF-*`，按原规划契约修复，并重新执行任务验证、相关回归、最终闭环和执行断言。
4. 重新通过独立执行审查和用户确认后，才能再次写入 `execution_status: confirmed`；原审查和原确认不得复用。

修复需要改变规划分区、任务边界、接口契约或验证定义时，停止执行并返回 `implementation-planning-v3` 重新进入 `discussing`、修订、审查和确认。

## 整体完成

所有任务完成后，执行任务入口定义的最终闭环和全局验证，覆盖任务间接口、数据、状态、UI、回归、兼容和反模拟结果。

然后调用 `project-development-review-v3` 审查入口、全部任务执行记录、代码差异和证据。审查报告写入同名目录 `review/REV-*`，入口 `execution_review` 链接该报告；审查不通过时回退相关任务并继续执行。

只有以下条件全部成立，才请求用户确认执行结果：

- 所有任务为 `done`，没有 `not_verified`、`blocked` 或未解释偏差。
- 每个任务的 RED、GREEN、回归、`V-*` 和执行断言证据完整。
- 最终闭环、失败路径和全局回归真实通过。
- 独立执行审查通过。

用户确认后把入口 `execution_status` 改为 `confirmed`，交给 `project-acceptance-v3`；未满足时保持 `executing`。

## 提问与边界

能从任务、代码、方案、日志或调研获得的信息自行处理。只有缺少用户才能提供的访问条件或决策，并且错误假设会改变范围、方案、数据、安全、兼容或用户体验时才提问。问题一次只问一个，给出推荐处理和选择影响，放在回复最后的 `## 需要你确认` 区块，后面不追加内容。

边界：

- 不创建执行入口或独立执行记录文档。
- 不改写已确认需求、方案和规划契约。
- 不自动提交、推送、部署或操作生产环境，除非用户明确要求。
- 不伪造、覆盖或压缩失败事实。
