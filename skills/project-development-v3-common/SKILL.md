# 项目研发 v3 公共协议

本 skill 只在维护或审查 V3 skill 套件时使用。正常执行某个研发阶段时，各阶段 SKILL.md 已直接包含运行所需规则，不要求额外读取本 skill，避免增加调用和 token 成本。

## 稳定流程

标准研发流程只有五个阶段：

```text
需求 → 方案 → 任务 → 执行 → 验收
```

需求、方案、任务和验收阶段各创建一份入口文档；执行阶段不创建新入口，直接复用已确认任务入口和 `T-*` 文档。当前阶段可以经过多轮讨论、调查、修改和复审；独立审查通过且用户明确确认后，才能进入下一阶段。

## 入口与支撑契约

**路径**

- 需求入口：`workplace/<版本>/requirements/<YYYY-MM-DD-功能名>.md`
- 方案入口：`workplace/<版本>/tech-design/<YYYY-MM-DD-功能名>.md`
- 任务入口：`workplace/<版本>/implementation-planning/<YYYY-MM-DD-功能名>.md`
- 验收入口：`workplace/<版本>/acceptance/<YYYY-MM-DD-功能名>.md`

用户给出路径时使用该文件；当前事项已有对应阶段入口时继续更新原文件；都没有时从上游入口标题推导文件名，输出完整路径后默认继续，不阻塞等待确认。版本号无法唯一确认时才询问。持久化文档不得包含 `<version>`、`1.X` 等占位路径。不同阶段的内容禁止写入同一文件。

**入口文档**

- 入口只保留用户判断本阶段是否完成所需的结论，不保存完整调查、讨论、日志或审查正文。
- 使用自然语言，不堆砌编号列表和技术细节；每个区块聚焦用户判断所需的结论。
- 禁止出现：函数名、字段名、测试代码、RED/GREEN/REFACTOR 细节、命令行输出；方案一级的变更清单中可用真实文件路径。
- 技术细节、实现步骤、调研过程全部放入支撑材料。

**编号追踪**

阶段结论使用稳定编号保持追踪：

- `F-*` 功能（来自需求）
- `D-*` 技术决策
- `C-*` 变更
- `T-*` 任务
- `V-*` 验证
- `RT-*` 调研
- `REV-*` 审查
- `DEF-*` 缺陷

不为单个编号创建入口文档；编号是入口正文或支撑报告中的条目。任务入口同时承载规划摘要和执行状态；详细 `T-*` 的规划契约与执行记录分区保存，执行不得改写规划分区。

**支撑材料**

支撑材料放在入口文档旁的同名目录，按需创建，不预建空目录：

```text
workplace/3/tech-design/2026-06-21-order-import.md
workplace/3/tech-design/2026-06-21-order-import/
├── research/RT-001-duplicate-handling.md
├── evidence/EVD-001-current-flow.md
└── review/REV-001-tech-design-review.md
```

- 新建材料使用"类型编号 + 简短英文 slug"。
- 入口在受影响条目链接支撑材料；Markdown 支撑材料链接回入口对应条目。
- 能直接链接真实代码、已有文档或既有报告时不复制内容。
- 完整复用已有材料时保留原路径并增加双向链接；增量更新时在原路径修改；范围实质不同时才新建。

## 调研契约

事实不明确且会阻止当前结论时，在入口受影响条目下登记 `RT-*`，至少写清问题、范围、不调查内容、证据类型、完成条件和受影响条目。登记后可直接调用 `project-research-v3`；字段缺失由调研阶段按自身契约回写 `blocked`。

调研开始前优先复用已有项目知识：如果 `personal-kb` skill 可用，按它的读取和查询规则检索；不可用时跳过知识复用并在调研报告记录能力限制。只有找不到有效答案时才进行新调研。

调研完成后必须按能力处理知识库沉淀：

- 结论可复用且 `personal-kb` 可用时，按 `personal-kb` 规则归档或更新。
- 一次性结论、`personal-kb` 不可用或 `blocked` 报告标注 `skipped`。
- 沉淀失败时记录 `error`，但调研问题本身证据充分时仍可标记为 `completed`。

## 审查契约

审查分两轮执行：

**第一轮**：扮演怀疑论者，针对核心逻辑提出 5-8 个问题，不针对格式。

**第二轮**：对每个问题给出四级评级——`BLOCKER`（文档无答案或自相矛盾）、`HIGH`（严重模糊）、`MEDIUM`（有歧义但不阻塞）、`PASS`（有明确答案）。

结论规则：有任何 `BLOCKER` 或 `HIGH` 则 `changes_required`，否则 `passed`。

两轮完成后补充核查四项：路径、文件合并、链路完整性、上游静默修改。发现问题升级为 `BLOCKER`。

- 当前阶段先自检并修正，再调用 `project-development-review-v3`。
- 审查报告使用入口同名目录下的 `review/REV-*`，并与入口双向链接。
- 审查只报告问题和复审结果，不替责任阶段重写结论或替用户确认。
- 阻塞问题未修复时保持阶段进行中；审查通过不等于用户确认。
- 用户确认只表示当前入口内容可以交给下一阶段，不能由 Agent 推断或自动记录。

## 状态契约

阶段入口和复用状态保持最小：

- 需求、方案、任务：`discussing` / `confirmed`。
- 执行复用任务入口的 `execution_status`：`pending` / `executing` / `confirmed`。
- 验收：`accepting` / `confirmed`。

调研、审查、任务和缺陷可以使用自身必要状态，但不得扩展阶段入口状态机。

## 回退与重新确认契约

- 验收发现实现缺陷且既有规划契约仍正确时，验收入口保持 `accepting` 并登记开放的 `DEF-*`。执行阶段可以据此把任务入口从 `execution_status: confirmed` 显式重开为 `executing`，把受影响 `T-*` 回退为 `in_progress`，并把 `execution_review` 重置为 `pending`。
- 修复后必须重新运行受影响任务验证、全局回归和独立执行审查，并再次取得用户对执行结果的确认；原确认不能自动复用。验收阶段随后重新执行受影响验收和回归。
- 缺陷需要改变规划契约时，由规划阶段把 `status` 改回 `discussing`，重置执行状态和审查，并重新完成计划审查与用户确认；影响方案或需求时继续逐级回退并使所有受影响下游确认失效。
- 任何 skill 都不能静默降级状态、保留已失效审查，或把历史确认当作修改后确认。

## 提问契约

只有答案无法从项目、文档、对话或可靠来源获得，且错误假设会实质改变范围、方案、任务、实现或验收时才提问。能调查就调查；能安全假设时写明依据和影响。

必须提问时，一次只问当前最有决策价值的问题，给出推荐方案和不同选择的影响。问题必须是回复最后的 `## 需要你确认` 区块，后面不追加其他内容。

## 证据与完成契约

- 事实结论必须能追溯到项目文件、运行结果或可靠来源。
- 不伪造命令输出、测试结果、截图、API 响应或用户确认。
- 不用 mock、stub、fallback、固定返回、样例数据或未验证实现冒充完成。
- build、lint、typecheck 只能证明其实际覆盖的质量，不能代替功能、失败路径、持久化、集成或 UI 验证。
- `done` 只用于真实实现且相关验证通过；不能验证时使用 `not_verified`，无法继续时使用 `blocked`。
- 报告完成前逐项自检；发现问题先修复并重新检查，不把基础缺陷留给用户。
