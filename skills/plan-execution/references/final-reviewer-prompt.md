# 最终审核 Subagent 提示词模板

委派最终审核 subagent 时使用此模板。

**目的**：从全局视角验证整体实现满足需求文档的功能点和验收标准。

**只在所有模块状态均为"完成"后委派。**

**传入方式**：将 `index.md` 的完整内容传入（含执行索引、特别关注点、功能点映射表），再加上从需求文档和技术方案中提取的功能点/测试点清单。

```
Agent tool (general-purpose):
  description: "全局最终审核：验证整体实现满足需求功能点"
  prompt: |
    你正在做最终审核。所有模块已完成并通过了各自的模块审核。
    你的任务是从全局视角检查：**整体实现是否满足需求文档的所有功能点**。

    ## 需求文档功能点清单

    {从需求文档提取所有功能，以及技术方案阶段拆分的功能点（FP-N）和测试点（TC-N）}

    ## 模块清单（全部完成）

    {复制 index.md 的执行索引表格和功能点-模块映射表，含对应功能点和测试点}

    ## 特别关注点

    {从 index.md 的"特别关注点"章节复制，若无则写"无"}

    ## 审核重点

    **1. 功能点全覆盖**
    - 需求文档的每个功能是否都有对应的功能点（FP-N）？
    - 技术方案中的每个功能点是否都被某个模块实现？
    - 是否有功能点被遗漏，未被任何模块覆盖？

    **2. 测试点全覆盖**
    - 每个功能点是否都有对应的测试点（TC-N）？
    - 每个测试点是否都有验收结果（后端脚本PASS / 前端审查通过）？
    - 是否有测试点没有验收结果？

    **3. 模块间集成（静态核对）**
    - 各模块对外接口/数据格式是否一致？
    - 前端调用的 API 字段名/类型是否与后端一致？
    - 是否存在"模块各自通过审核但组合后会出问题"的迹象？

    **4. 横切关注点**
    - 错误处理、边界情况、用户反馈是否完整？
    - **前端覆盖**：需求文档"页面清单"中的每个页面是否都已实现并可访问？

    **5. 降级逻辑全局扫查**
    - 是否存在跨模块的降级模式：某模块出错，下游模块静默接受并继续"成功"？
    - 是否有任何功能点的实现是"假成功"（返回成功但实际未执行核心逻辑）？
    - 全局搜索 `catch`, `fallback`, `default`, `mock`, `skip` 等关键词，确认无隐藏降级路径
    - 若存在任何降级逻辑残留，标记为❌问题（同功能点未覆盖一样严重）

    **6. 验收脚本/清单完整性**
    - `workplace/1.X/test/backend/unit/` 和 `workplace/1.X/test/backend/integration/` 下是否有所有后端模块的验收脚本？
    - `workplace/1.X/test/frontend/unit/` 和 `workplace/1.X/test/frontend/component/` 下是否有前端单元/组件测试？
    - `workplace/1.X/test/review/` 下是否有所有前端模块的审查清单？
    - `workplace/1.X/test/fixtures/` 下是否有共享测试数据？
    - `workplace/1.X/test/README.md` 是否存在且包含运行命令矩阵和覆盖率目标？
    - 每个脚本/清单是否都有执行结果记录？
    - **不应出现 `e2e/` 目录或端到端测试文件**——若有，标记为问题（本套 skill 不做 E2E）

    **7. 特别关注点合规**
    - index.md 中列出的特别关注点（用户强调的约束、风险、偏好）是否在实现中得到了落实？
    - 每个特别关注点是否可追溯至某个模块的实现或测试验证？

    **8. BUGFIX 计划验证**（仅当计划为修复计划时）
    - 每个模块是否包含"关联根因"、"风险点"、"回退方案"字段？
    - 最后一个模块是否为"回归验证"模块？该模块是否验证了原 bug 已修复且无回归？
    - 修复模块是否实际针对根因（而非表象）？

    **不要**：
    - 重复单模块的审核（已完成）
    - 重新审查代码质量细节（已完成）
    - 实际运行端到端流程或模拟完整用户路径（不在本范围）
    - 编写或要求新增端到端测试

    **要**：
    - 对照需求功能点清单逐项核查
    - 对照技术方案的功能点清单逐项核查
    - 通过阅读关键模块的对外接口/调用点，发现跨模块的集成问题
    - 检查测试目录结构完整性

    ## 校准原则

    **只标记会导致需求遗漏或集成失败的严重缺陷。**

    功能点未覆盖、模块间接口不一致、测试目录缺失——这些是问题。
    个别实现风格偏好、"某些模块可以更优雅"——这些不是问题。

    除非有严重缺陷会导致需求功能缺失或模块间集成失败，否则通过审核。

    报告：
    - ✅ 通过（功能点全覆盖，测试点全覆盖，接口一致，测试目录完整）
    - ❌ 发现问题（列出问题，按功能点归类，标记前端/后端/测试问题）

    如果发现问题，列出：
    - [功能点 FP-X]：[问题描述] → [建议修复方式]
    - [测试点 TC-Y]：[问题描述] → [建议修复方式]
```
