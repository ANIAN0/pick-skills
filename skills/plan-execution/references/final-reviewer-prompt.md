# 最终审核 Subagent 提示词模板

委派最终审核subagent时使用此模板。

**目的**：验证所有模块合在一起满足需求文档的验收标准（全局视角，弥补单模块审核的局部盲区）

**只在所有模块的状态均为"完成"后委派。**

```
Agent tool (general-purpose):
  description: "全局最终审核：验证整体实现满足需求验收标准"
  prompt: |
    你正在做最终审核。所有模块已完成并通过了各自的模块审核。
    你的任务是从全局视角检查：**整体实现是否满足需求文档的验收标准**。

    ## 需求文档验收标准

    {从需求文档的"验收标准"章节复制完整内容}

    ## 模块清单（所有模块已完成）

    {复制计划文件的执行索引表格}

    ## 审核重点

    **验收标准覆盖**：
    - 需求文档中每条验收标准是否都有对应的实现？
    - 是否有验收标准被遗漏，未被任何模块覆盖？
    - 功能是否可以端到端走通，还是存在断点？

    **模块间集成**：
    - 各模块的实现是否能正确衔接？（接口、数据格式、依赖关系）
    - 是否存在模块A和模块B各自通过审核，但组合后出现问题的情况？

    **端到端验证**：
    - 实际运行或追踪关键用户路径
    - 检查数据流是否完整：用户输入 → 前端 → API → 服务 → 数据库 → 返回

    **遗漏检测**：
    - 有没有"每个模块都没问题，但整体缺了一块"的情况？
    - 错误处理、边界情况、用户反馈等横切关注点是否完整？
    - **前端覆盖**：需求文档"页面清单"中的每个页面是否都已实现并可访问？
    - **测试整体性**：`workplace/1.X/test/` 下是否覆盖：后端单元、后端集成、前端单元/组件、E2E？是否有 README.md 列出运行命令？覆盖率是否达标？

    **不要**：
    - 重复单模块的审核（那已经完成了）
    - 重新审查代码质量细节（那也已经完成了）
    - 相信模块完成报告——独立验证端到端流程

    **要**：
    - 对照需求文档验收标准逐条核查
    - 实际运行或追踪关键用户路径
    - 发现跨模块的集成问题

    报告：
    - ✅ 通过（所有验收标准满足，集成无问题，测试套件完整）
    - ❌ 需修复（列出发现的问题，按模块归类，并标记是前端/后端/测试问题）

    如果需修复，列出：
    - [模块名]：[问题描述] → [建议修复方式]
```
