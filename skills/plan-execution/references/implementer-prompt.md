# 实现者 Subagent 提示词模板

委派实现subagent时使用此模板。

**目的**：让subagent实现单个任务

```
Agent tool (general-purpose):
  description: "实现 Task N: {任务名称}"
  prompt: |
    你正在实现 Task N: {任务名称}

    ## 任务描述

    {完整复制任务清单中的任务内容 - 不要让subagent读取文件}

    ## 上下文

    {说明此任务在整体中的位置、依赖关系、架构背景}

    ## 开始前确认

    如果对以下内容有疑问：
    - 需求或验收标准
    - 实现策略
    - 依赖或假设
    - 任务描述中不清楚的内容

    **现在提问。** 有疑问就提出来，不要猜测或假设。

    ## 你的工作

    确认需求后：
    1. 按任务规范实现
    2. 编写测试（若任务要求TDD）
    3. 验证实现功能正确
    4. 自检（见下方）
    5. 报告结果

    **工作目录**：{目录路径}

    **重要**：不要提交git，保持变更在工作目录中。

    ## 代码组织

    你能最好地理解一次就能装入上下文的代码，编辑也更可靠当文件聚焦时。记住：
    - 遵循计划定义的文件结构
    - 每个文件应有单一清晰职责和定义良好的接口
    - 如果创建的文件超出计划意图变大，停止并报告DONE_WITH_CONCERNS - 不要自行拆分
    - 如果修改的已有文件已经很大或混乱，小心处理并在报告中作为担忧说明
    - 在已有代码库中，遵循既有模式。像好开发者那样改进你接触的代码，但不要重构超出任务范围的内容

    ## 遇到困难时

    随时可以停止并报告困难。错误的实现比没有实现更糟糕。

    **停止并升级的情况**：
    - 需要做架构决策（多种方案可选）
    - 需要理解超出所提供范围的代码
    - 对实现方式不确定
    - 任务涉及超出计划预期的重构
    - 阅读大量文件仍无法理解系统

    **升级方式**：报告状态 BLOCKED 或 NEEDS_CONTEXT，说明具体卡住的地方、尝试过什么、需要什么帮助。控制器可以提供更多上下文、用更强模型重新委派、或拆分任务。

    ## 完成前：自检

    检查自己的工作：

    **完整性**：
    - 是否完全实现了规范要求？
    - 是否遗漏了任何要求？
    - 是否处理了边缘情况？

    **质量**：
    - 这是你最好的工作吗？
    - 名称是否清晰准确（匹配功能而非实现方式）？
    - 代码是否整洁可维护？

    **纪律**：
    - 是否避免了过度构建（YAGNI）？
    - 是否只构建了被请求的内容？
    - 是否遵循了代码库现有模式？

    **测试**：
    - 测试是否验证了实际行为（而非mock行为）？
    - 是否按要求遵循了TDD？
    - 测试是否全面？

    自检发现问题立即修复后再报告。

    ## 报告格式

    完成时报告：
    - **状态**：DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么（或尝试了什么，如果被阻塞）
    - 测试内容和结果
    - 变更的文件
    - 自检发现（如有）
    - 任何问题或担忧

    使用 DONE_WITH_CONCERNS 如果完成但有正确性疑问。
    使用 BLOCKED 如果无法完成任务。
    使用 NEEDS_CONTEXT 如果需要未提供的信息。
    绝不要默默产出不确定的工作。
```

## 处理Subagent状态

| 状态 | 处理方式 |
|------|----------|
| **DONE** | 进入审核阶段 |
| **DONE_WITH_CONCERNS** | 先读担忧内容，评估后再决定是否进入审核 |
| **NEEDS_CONTEXT** | 提供缺失上下文，重新委派 |
| **BLOCKED** | 评估阻塞原因：提供上下文/换更强模型/拆分任务/升级给用户 |

**绝不**忽略升级请求或强行让同一模型重试。Subagent说卡住就是真的需要改变。
