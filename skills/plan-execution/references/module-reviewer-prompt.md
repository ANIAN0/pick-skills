# 模块审核 Subagent 提示词模板

委派模块审核 subagent 时使用此模板。

**目的**：一次性验证模块实现是否符合规范、质量达标、边界正确、测试目录合规。

**只在实现者完成模块后委派。**

```
Agent tool (general-purpose):
  description: "审核 模块N 的实现"
  prompt: |
    你正在审核模块N 的实现是否符合规范、质量达标、边界接口正确、测试目录合规。

    ## 模块要求

    {完整复制模块详情：目标、子步骤、接口契约}

    ## 实现者声称构建的内容

    {从实现者报告复制}

    ## 相邻模块接口契约（用于边界检查）

    {上一个模块的输出契约 / 下一个模块的输入契约}

    ## 关键：不要相信报告

    实现者可能完成得异常快。报告可能不完整、不准确或过于乐观。
    你必须独立验证，但不需要逐行审查所有代码——聚焦模块级别的关键问题。

    **不要**：
    - 相信他们说的实现了什么
    - 相信他们声称的完整性
    - 逐行审查每个文件

    **要**：
    - 检查每个子步骤是否实际完成
    - 阅读关键接口和主要实现文件
    - 验证对外接口是否符合契约
    - 检查声称实现但缺失的部分

    ## 审核维度（四合一）

    ### 1. Spec 合规
    **子步骤完成度**：每个子步骤完成？（是/否/部分）
    **缺失的要求**：是否跳过或遗漏？是否声称功能工作但实际未实现？
    **额外/不需要的工作**：是否构建了未请求的内容？是否过度工程化？

    ### 2. 代码质量
    **文件组织**：文件单一清晰职责？模块内结构合理？
    **关键质量**：核心接口命名清晰？无明显设计问题？遵循代码库现有模式？
    **不检查**：非关键代码逐行；格式细节（交给 linter）

    ### 3. 模块边界
    **输入匹配**：本模块是否正确消费前置模块的接口/数据？
    **输出匹配**：本模块暴露的接口/数据是否符合契约？下一个模块能否正确使用？
    **数据一致性**：字段名、类型是否匹配契约？

    ### 4. 测试目录合规
    **位置正确性**：
    - 所有新增测试文件位于 `workplace/1.X/test/` 下？（X 为当前版本数字）
    - 源码目录中是否残留测试文件（*.test.* / *.spec.* / test_*.py 等）？发现即不通过

    **分类正确**：
    - 后端单元在 backend/unit、集成在 backend/integration
    - 前端单元在 frontend/unit、组件在 frontend/component
    - **未出现 e2e/ 目录或端到端测试文件**——若有，标记为问题（本套 skill 不做 E2E，需移除）

    **前端模块**：若本模块属前端层，至少含一类前端测试（单元/组件）

    **fixtures**：声称使用的 mock/fixture 是否在 `workplace/1.X/test/fixtures/`？

    **测试可执行**：实现者声称的测试命令是否能跑通？（必要时本地复现一遍命令）

    ## 报告格式

    ```
    ## 审核结果：模块N

    ### Spec 合规
    - 子步骤完成：[全部/部分/未开始]
    - 缺失项：[列表或"无"]
    - 多余项：[列表或"无"]

    ### 代码质量
    - 优点：[做得好的地方]
    - 问题（Critical/Important/Minor）：
      - [Critical: 必须修复]
      - [Important: 建议修复]
      - [Minor: 可选修复]

    ### 模块边界
    - 输入匹配：[✅/❌] [说明]
    - 输出匹配：[✅/❌] [说明]

    ### 测试目录合规
    - 位置正确：[✅/❌] [越界文件列表]
    - 分类正确：[✅/❌]
    - 无 E2E 残留：[✅/❌]
    - 前端测试覆盖（仅前端模块）：[✅/❌/不适用]
    - 测试命令可执行：[✅/❌] [说明]

    ### 结论
    - [ ] 通过（无 Critical 问题，子步骤全部完成，边界匹配，测试目录合规）
    - [ ] 需修复（列出 Critical 和 Important 问题）
    - 修复后是否需要重新审核：[是/否]
    ```

    如果结论为"需修复"，列出具体问题清单。实现者修复后，根据修复范围决定是否需重新审核。
```
