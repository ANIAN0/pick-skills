# 模块审核 Subagent 提示词模板

委派模块审核 subagent 时使用此模板。

**目的**：一次性验证模块实现是否符合功能点要求、验收标准是否实际通过、测试目录合规。

**只在实现者完成模块后委派。**

```
Agent tool (general-purpose):
  description: "审核 模块N 的实现"
  prompt: |
    你正在审核模块N 的实现是否符合功能点要求、验收标准是否实际通过。

    ## 模块要求

    {完整复制模块详情：目标、对应功能点、对应测试点、子步骤、接口契约}

    ## 验收标准

    {完整复制模块验收标准：
    - 如果是后端脚本：复制完整的py脚本内容
    - 如果是前端审查：复制完整的审查清单内容
    }

    ## 实现者声称构建的内容

    {从实现者报告复制}

    ## 相邻模块接口契约（用于边界检查）

    {上一个模块的输出契约 / 下一个模块的输入契约}

    ## 关键：不要相信报告

    实现者可能完成得异常快。报告可能不完整、不准确或过于乐观。
    你必须独立验证，但不需要逐行审查所有代码——聚焦模块级别的关键问题。

    **不要**：
    - 相信他们说的实现了什么
    - 相信他们声称的完整性
    - 相信他们声称的验收通过
    - 逐行审查每个文件

    **要**：
    - 对照功能点清单逐项验证是否实现
    - 实际运行验收脚本（后端模块）或按清单逐项检查（前端模块）
    - 阅读关键接口和主要实现文件
    - 验证对外接口是否符合契约
    - 检查声称实现但缺失的部分

    ## 审核维度（四合一）

    ### 1. 功能点覆盖（核心）
    对照模块的"对应功能点"列表，逐项检查：
    - FP-X：是否实现？验收是否通过？
    - FP-Y：是否实现？验收是否通过？
    ...

    **规则**：任一功能点未实现或验收未通过 → 不通过

    ### 2. 验收标准实际执行（核心）

    **后端模块**：
    - 找到计划阶段定义的验收py脚本
    - **实际运行脚本**，记录完整输出
    - 检查是否ALL PASS
    - 如果实现者报告说通过了但你不相信，**重新运行一遍**

    **前端模块**：
    - 找到计划阶段定义的审查清单
    - **逐项验证**（如果可能，实际查看页面效果）
    - 检查是否全部勾选
    - 如果实现者报告说全部勾选了但你不相信，**重新检查一遍**

    ### 3. 代码质量
    **文件组织**：文件单一清晰职责？模块内结构合理？
    **关键质量**：核心接口命名清晰？无明显设计问题？遵循代码库现有模式？
    **不检查**：非关键代码逐行；格式细节（交给 linter）

    ### 4. 模块边界
    **输入匹配**：本模块是否正确消费前置模块的接口/数据？
    **输出匹配**：本模块暴露的接口/数据是否符合契约？下一个模块能否正确使用？
    **数据一致性**：字段名、类型是否匹配契约？

    ### 5. 测试目录合规
    **位置正确性**：
    - 所有新增测试文件位于 `workplace/1.X/test/` 下？（X 为当前版本数字）
    - 源码目录中是否残留测试文件（*.test.* / *.spec.* / test_*.py 等）？发现即不通过

    **分类正确**：
    - 后端验收脚本在 `workplace/1.X/test/backend/`
    - 前端审查清单在 `workplace/1.X/test/review/`
    - **未出现 e2e/ 目录或端到端测试文件**——若有，标记为问题（本套 skill 不做 E2E，需移除）

    **fixtures**：声称使用的 mock/fixture 是否在 `workplace/1.X/test/fixtures/`？

    ## 报告格式

    ```
    ## 审核结果：模块N

    ### 功能点覆盖
    - FP-X：{✅已实现且验收通过 / ❌未实现 / ⚠️实现但验收未通过} {说明}
    - FP-Y：{✅已实现且验收通过 / ❌未实现 / ⚠️实现但验收未通过} {说明}
    ...
    覆盖结论：{全部覆盖 / 部分遗漏}

    ### 验收标准执行
    **后端脚本**：
    - 脚本路径：{路径}
    - 实际运行输出：
      ```
      {粘贴实际运行输出}
      ```
    - 结果：{ALL PASS / 部分失败}

    **前端审查清单**：
    - 清单路径：{路径}
    - 逐项验证：
      - [ ] TC-X：{通过/未通过} {说明}
      - [ ] TC-Y：{通过/未通过} {说明}
    - 结果：{全部通过 / 部分未通过}

    ### 代码质量
    - 优点：[做得好的地方]
    - 问题（Critical/Important/Minor）：
      - [Critical: 必须修复]
      - [Important: 建议修复]
      - [Minor: 可选修复]

    ### 模块边界
    - 输入匹配：[✅/❌] [说明]
    - 输出匹配：[✅/❌] [说明]

    ### 测试目录合规
    - 位置正确：[✅/❌] [越界文件列表]
    - 分类正确：[✅/❌]
    - 无 E2E 残留：[✅/❌]

    ### 结论
    - [ ] 通过（所有功能点覆盖且验收通过，无 Critical 问题，边界匹配，测试目录合规）
    - [ ] 需修复（列出 Critical 和 Important 问题，以及未通过的功能点/验收项）
    - 修复后是否需要重新审核：[是/否]
    ```

    如果结论为"需修复"，列出具体问题清单。实现者修复后，根据修复范围决定是否需重新审核。
    **如果功能点未覆盖或验收未通过，必须标记为需修复。**
```
