# 验证工作流

## 同步定义

开发开始前，能力与同父级验证在需求阶段同步创建；验证通过 `verifies` 指向能力。验收消费能力和验证，不为匹配实现而重新定义它们。

## 验收范围

每个 Acceptance 记录能力 ID、验证 ID、完整用户路径、前置条件、正向及必要负向步骤、预期观察、环境和完成边界，并以 `depends-on` 关联实际执行验证的 Verification/Task。Acceptance done 要求这些目标均有通过 Evidence。

## E2E 执行

对跨能力发布关键旅程执行真实 E2E。需要 UI 时使用真实浏览器；需要服务、数据库或外部工具时连接实际测试环境。保存命令、退出码、截图、日志、追踪和其他可读取产物。

## 失败归因

失败后保留原始证据并定位最早责任节点，三类归因互斥：`test-asset` 表示测试资产错误；`environment-data` 表示环境或数据阻塞；`product-code` 表示产品实现缺陷。只有 test-asset 允许修改测试资产；其他情况不得改变期望或弱化断言。实现缺陷创建 Root Cause，上游契约问题创建 Finding，环境缺失记录 blocked/not_verified。只重开局部影响范围，不全流程重跑。

## 回归交接

稳定通过且适合自动化的路径交给测试套件适配器。记录来源验证、覆盖场景、脚本路径、运行命令和最近通过证据。生成但未执行的脚本不是回归证据；自动化脚本本身必须真实运行后才能作为回归资产。
