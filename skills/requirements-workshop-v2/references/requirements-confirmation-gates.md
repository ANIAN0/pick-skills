# 需求确认闸门

需求阶段必须固定确认关键环节。不要因为“少问问题”而跳过这些闸门。

## 总规则

- 每个闸门先给出 agent 的候选理解，再请求用户确认。
- 用户确认后才能进入下一闸门。
- 如果用户要求跳过确认，必须记录“已跳过”和风险。
- 每个闸门最多列 3 个需要用户判断的点；其余作为假设或风险记录。

## G1：问题、目标和核心机制确认

在需求推导后执行。

必须确认：

- 当前问题是否判断准确。
- 不做的后果是否成立。
- 核心机制是否抓住了系统真正如何运转。

输出格式：

```markdown
### G1 问题和机制确认
我理解当前问题是：...

核心机制是：...

不做的后果是：
- ...

请确认：这个问题判断和核心机制是否准确？有没有我理解偏的地方？
```

## G2：用户路径、关键对象和边界确认

在写范围前执行。

必须确认：

- 主路径和关键分支路径是否完整。
- 每个本期需求的入口、操作、成功结果、失败反馈和结果可见性是否完整。
- 页面/入口覆盖是否完整，或无 UI 声明是否成立。
- 关键对象是否识别正确。
- 每个对象的拥有者、生命周期和边界是否准确。
- Agent/AI 节点的任务、角色边界、自主度和失败风险是否准确，如适用。

输出格式：

```markdown
### G2 路径和边界确认
关键用户路径：
- ...

页面/入口覆盖：
- ...

关键对象和边界：
- ...

AI 节点需求（如适用）：
- ...

请确认：是否遗漏了关键路径、页面入口、对象边界或 AI 节点约束？
```

## G3：范围、验收标准和待决策项确认

在保存需求简报前执行。

必须确认：

- 本期范围。
- 后续候选。
- 不做范围。
- 验收标准是否代表真实成功。
- 每条验收标准的关注点是否完整，能覆盖后端事实、前端可见性、失败/异常、数据追溯、兼容或隔离。
- 特别关注点是否完整记录。
- 待决策项是否需要现在确认。

输出格式：

```markdown
### G3 范围和验收确认
本期范围：
- R-001 ...

后续候选：
- ...

不做范围：
- ...

验收口径：
- A-001 ...（关注点：后端事实/前端可见性/失败异常/数据追溯/兼容隔离）

特别关注点：
- ...

请确认：这些范围和验收标准是否可以作为技术方案输入？
```

## 跳过规则

用户可以要求跳过某个闸门，但文档必须记录：

- 跳过哪个闸门。
- 跳过原因。
- 可能导致的风险。
- 后续哪个阶段补确认。
