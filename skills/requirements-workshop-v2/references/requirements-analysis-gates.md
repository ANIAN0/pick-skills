# 需求推导门槛

需求阶段不是把用户话术整理成清单，而是把“为什么要做、系统要承担什么责任、哪些决策会影响后续实现”推导清楚。

## 1. 问题诊断

写需求前先回答：

- 当前系统或流程哪里不满足目标？
- 这个问题影响谁？
- 不做会有什么后果？
- 现有替代方案为什么不够？
- 本期成功后，用户或系统状态会发生什么变化？

如果这些问题答不出来，不要直接写需求列表，先补充上下文或提出关键问题。

## 2. 核心机制

中大型需求必须说明核心机制。核心机制不是模块名，而是系统如何运转。

示例：

- “Agent 平台服务化重构”的核心机制不是“Tool、Agent、Session、Memory 八个模块”，而是“用户消息进入路由后，配置、工具、上下文、运行引擎、事件、历史和长期记忆如何协同形成一次可追踪运行”。

至少说明：

- 哪些对象在系统里流转。
- 谁创建、读取、更新这些对象。
- 哪些对象是事实源。
- 哪些能力只是消费事实，不拥有事实。
- 哪些失败必须隔离，哪些失败可以阻断主流程。

## 3. 关键对象和边界

识别真正被管理或追踪的对象，例如：

- Agent 配置。
- Tool 能力。
- Run。
- Session。
- Branch。
- Message。
- Event。
- Memory。
- Knowledge。

每个对象要说明：

- 含义。
- 所属边界或拥有者。
- 生命周期。
- 与其他对象的关系。
- 不属于它的责任。

## 4. 需求不是模块清单

不要把“实现 X Service”直接写成需求。需求应描述用户能力或系统能力：

- 差：实现 Agent Config Service。
- 好：用户能创建、更新、禁用 Agent 配置，运行服务能读取同一份有效配置。

模块可以作为组织方式，但每个模块下的需求必须回答“提供了什么能力、服务谁、成功如何判断”。

## 5. 决策记录

记录真正影响后续设计的决策：

- 范围：本期做什么、不做什么。
- 数据：是否迁移旧数据，谁是事实源。
- 兼容：是否保留旧接口或旧项目。
- 交互：用户在哪里完成关键操作。
- 风险：失败时隔离还是阻断。

每个决策写明结论、理由、影响和状态。状态只能是：

- `已确认`
- `待确认`
- `假设`

## 6. 验收口径

验收标准必须证明需求，不是证明模块存在。

优先写可观察结果：

- 用户能完成什么操作。
- 系统能查询到什么历史或状态。
- 某个失败不会影响什么链路。
- 某个对象能追溯到来源。

对于跨模块需求，验收必须覆盖真实闭环，而不是只覆盖单模块返回成功。
