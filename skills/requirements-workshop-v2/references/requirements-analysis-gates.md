# 需求推导门槛

需求阶段不是把用户话术整理成清单，而是把“为什么要做、系统要承担什么责任、哪些决策会影响后续实现”推导清楚。

## 1. 问题诊断

写需求前先回答：

- 当前系统或流程哪里不满足目标？
- 这个问题影响谁？
- 不做会有什么后果？
- 现有替代方案为什么不够？
- 本期成功后，用户或系统状态会发生什么变化？

如果这些问题答不出来，不要直接写需求列表，先补充上下文或提出关键问题。

## 2. 核心机制

中大型需求必须说明核心机制。核心机制不是模块名，而是系统如何运转。

满足以下任一条件时，必须展开核心机制和关键决策：

- 新子系统、新模块或跨模块协作。
- 历史、日志、配置、记忆、知识、权限等事实源发生变化。
- 涉及 Agent、LLM、MCP、异步任务、队列或外部系统。
- 三条以上用户路径或多个角色协作。
- 影响持久化数据、兼容接口、权限或失败隔离。
- 用户明确提到“重构”“服务化”“平台”“架构”“扩展”。

示例：

- “Agent 平台服务化重构”的核心机制不是“Tool、Agent、Session、Memory 八个模块”，而是“用户消息进入路由后，配置、工具、上下文、运行引擎、事件、历史和长期记忆如何协同形成一次可追踪运行”。

至少说明：

- 哪些对象在系统里流转。
- 谁创建、读取、更新这些对象。
- 哪些对象是事实源。
- 哪些能力只是消费事实，不拥有事实。
- 哪些失败必须隔离，哪些失败可以阻断主流程。

## 3. MVP 范围分层

需求阶段必须把范围分成三类：

- 本期必须做：没有它就不能形成完整闭环。
- 后续候选：有价值，但不影响本期闭环或验收。
- 不做范围：明确排除，并说明原因。

不要把“以后可能做”“看情况”“顺便支持”写入本期需求。辅助能力只有在用户确认本期必须交付时才进入 `R-*`。

范围分层必须能回答：

- 本期最小闭环是什么？
- 哪些能力只是增强体验或扩展效率？
- 哪些能力会显著增加技术风险、迁移成本或验证成本？
- 如果删掉某项，本期验收是否仍成立？

## 4. 用户可见闭环和前端覆盖

每个本期 `R-*` 必须能描述完整闭环：

- 入口：用户、系统或外部调用从哪里触发。
- 操作：用户或系统做什么。
- 成功结果：用户或调用方能看到什么结果。
- 失败反馈：失败、无权限、空数据、异常或超时时如何反馈。
- 结果可见性：结果、状态、日志或数据在哪里可查询或展示。

需求包含页面、入口、配置、日志、列表、详情、状态或用户反馈时，必须输出页面/入口覆盖表。无 UI 需求时写明“无 UI”，并说明可观察入口，例如 API 响应、日志、事件、查询接口或后台状态。

## 5. 关键对象和边界

识别真正被管理或追踪的对象，例如：

- Agent 配置。
- Tool 能力。
- Run。
- Session。
- Branch。
- Message。
- Event。
- Memory。
- Knowledge。

每个对象要说明：

- 含义。
- 所属边界或拥有者。
- 生命周期。
- 与其他对象的关系。
- 不属于它的责任。

## 6. 需求不是模块清单

不要把“实现 X Service”直接写成需求。需求应描述用户能力或系统能力：

- 差：实现 Agent Config Service。
- 好：用户能创建、更新、禁用 Agent 配置，运行服务能读取同一份有效配置。

模块可以作为组织方式，但每个模块下的需求必须回答“提供了什么能力、服务谁、成功如何判断”。

## 7. Agent/AI 节点需求

涉及 Agent、LLM、MCP、自动分析、自动生成、自动审核或智能路由时，需求阶段必须说明 AI 节点的业务需求，不设计具体实现。

每个 AI 节点至少说明：

- 任务类型：分类、生成、审核、分析、规划、路由或诊断。
- 选择倾向：Agent 或 LLM；默认倾向 Agent，除非输入固定、上下文极少且不需要工具。
- 角色边界：它应该判断什么，不应该越权做什么。
- 工具或上下文需求：需要访问哪些业务对象、日志、配置、知识或外部能力。
- 自主度：哪些判断可自动完成，哪些必须用户确认。
- 失败风险：错误输出会导致误报、漏报、阻断、数据损坏还是安全风险。

如果这些信息会影响范围或验收，必须进入确认闸门；如果只是技术实现细节，交给技术阶段确认。

## 8. 决策记录

记录真正影响后续设计的决策：

- 范围：本期做什么、不做什么。
- 数据：是否迁移旧数据，谁是事实源。
- 兼容：是否保留旧接口或旧项目。
- 交互：用户在哪里完成关键操作。
- 风险：失败时隔离还是阻断。
- AI：Agent/LLM 倾向、角色边界、自主度。

每个决策写明结论、理由、影响和状态。状态只能是：

- `已确认`
- `待确认`
- `假设`

## 9. 验收口径

验收标准必须证明需求，不是证明模块存在。

优先写可观察结果：

- 用户能完成什么操作。
- 系统能查询到什么历史或状态。
- 某个失败不会影响什么链路。
- 某个对象能追溯到来源。

对于跨模块需求，验收必须覆盖真实闭环，而不是只覆盖单模块返回成功。

每条 `A-*` 必须标注验收关注点，至少从以下类型选择：

- 后端事实：数据写入、状态变化、权限、校验、日志、事件、来源追溯。
- 前端可见性：页面渲染、状态转换、提示、错误、跳转、列表/详情更新。
- 失败/异常：无权限、无数据、超时、外部服务失败、部分失败隔离。
- 兼容和隔离：旧接口兼容、事实源一致、失败不阻断主链路。
- 数据追溯：来源、分支、历史、撤回、压缩、重试或审计。

## 10. 特别关注点

特别关注点只记录用户明确强调的约束、风险或偏好，不要把全部需求复制进去。

典型来源：

- 用户反复强调“不能 mock”“必须真实串联”“不要降级实现”。
- 对兼容、迁移、页面、性能、权限、失败隔离有明确偏好。
- 对某些目录、技术边界、上线范围或验收方式有强约束。

特别关注点必须交接给 `tech-design-v2`、`implementation-planning-v2` 和执行阶段。

## 11. 技术阶段交接问题

需求阶段必须输出“技术阶段必须确认的问题”。这些问题不是普通闲聊，而是会影响方案结构、数据契约或验收方式的决策。

优先识别：

- 兼容接口：是否必须保留旧接口、OpenAI 兼容接口或外部调用协议。
- 事实源：历史、日志、配置、记忆、知识分别由谁拥有。
- 关键字段：需求列出的对象字段是否完整，例如 subagent、压缩规则、来源引用。
- 失败隔离：哪些失败必须不影响主链路，隔离等级是模块级、进程级还是部署级。
- 用户页面：哪些页面、状态和错误反馈是验收的一部分。
- 异步链路：是否需要任务状态、失败记录、重试和查询入口。

如果这些问题无法从上下文安全推断，必须在确认闸门中提出；如果可以推断，写入假设并说明影响。
