# 项目知识库质量评估标准

本标准用于评估 `project-kb/`，重点是 `project-kb/code/` 下的代码文件说明。

## 单文件说明基本要求

| 维度 | 标准 | 不达标表现 |
|---|---|---|
| 路径一致 | 文档路径与源码相对路径一致 | `src/foo.ts` 被写成 `foo.md` |
| 功能点清晰 | 列出该文件承担的可验证功能 | 只写“工具函数”“页面组件” |
| 相关文件 | 使用 Obsidian 双向链接列出强关联文件 | 只写纯文本路径或没有反向链接 |
| 重要逻辑 | 记录业务规则、边界条件、兼容约束 | 只复述代码结构 |
| 测试映射 | 列出相关测试文件或说明缺失 | 修改后不知道运行哪些测试 |
| 修改注意 | 说明修改风险和验证建议 | 没有影响范围提示 |
| 时效性 | 最近更新记录与代码变更相符 | 文档长期不更新且与代码冲突 |

## 质量等级

| 等级 | 标准 |
|---|---|
| A | 基本要求全部满足，相关文件双链完整，测试映射可直接指导验证 |
| B | 基本要求满足，但重要逻辑或测试说明仍可补充 |
| C | 有明显缺项，但仍能提供部分修改参考 |
| D | 路径错误、内容过时或无法指导修改 |

C/D 级条目应优先在当前任务涉及时补齐。

## 知识库健康检查

| 维度 | 检查方法 | 良好表现 |
|---|---|---|
| 核心覆盖 | 对照 `PROJECT_RULES.md` 的重要文件 | 重要文件均有说明 |
| 代码镜像 | 抽查源码路径和文档路径 | 路径一一对应 |
| 双链完整 | 检查 `相关文件` 链接 | 强关联文件互相可追踪 |
| 测试可用 | 抽查 `测试文件` 字段 | 能选出定向测试 |
| 内容准确 | 对照当前代码 | 无过时描述 |
| 入口清晰 | 从 `PROJECT_RULES.md` 和 README 进入 | 可快速定位核心条目 |

## 更新触发条件

必须更新知识库：
- 修改已有代码文件。
- 新增、删除或重命名代码文件。
- 新增或调整测试文件。
- 发现功能点、重要逻辑或相关文件描述过时。
- `PROJECT_RULES.md` 新增重要文件入口。

可以延后但需记录待办：
- 非核心文件缺少说明。
- 相关文件链接还不完整。
- 测试映射需要人工确认。

## 评审问题

更新完成后自查：

- 修改这个文件前，后来者能否从条目中知道原有功能点？
- 条目是否说明了本次修改可能影响的其他文件？
- 是否能根据 `测试文件` 字段选择定向测试？
- 重要逻辑是否解释了“为什么”，而不仅是“是什么”？
- 新增的双向链接是否真的代表强关联？

## 操作日志质量

`project-kb/changelog.md` 每条记录应包含：

- 触发原因。
- 创建或更新的条目。
- 对应源码文件。
- 测试映射变化。
- 待跟进事项。
