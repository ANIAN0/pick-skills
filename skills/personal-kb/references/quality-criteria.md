# 项目知识库质量标准

只在迁移、健康检查或大范围更新后读取。

## 格式与导航

| 维度 | 合格标准 |
|---|---|
| OKF 最低契约 | 每个普通概念有可解析 frontmatter 和非空 `type` |
| 入口 | 根 `index.md` 声明 `okf_version: "0.1"` 并能进入所有已有分类 |
| 渐进导航 | 每个已有分类的 `index.md` 列出实际概念和一句话描述 |
| 链接 | 使用可解析的标准 Markdown 链接，不使用 wiki link |
| 日志 | `log.md` 按日期倒序记录有意义的知识变化 |
| 宽容消费 | 未知 `type` 和扩展字段被保留，不因无法识别而删除 |

## 内容质量

| 维度 | 合格标准 |
|---|---|
| 证据 | 关键结论可追溯到代码、测试、阶段文档、报告或外部来源 |
| 持久价值 | 内容能支持后续判断，不是临时任务状态或过程副本 |
| 单一概念 | 同一结论维护在稳定路径，不因不同阶段重复创建 |
| 准确性 | 关键内容与当前项目和原始证据一致 |
| 边界 | 说明结论适用条件、限制或失效条件 |
| 关联 | 链接周围文字解释关系，强关联按需建立反向链接 |

## 分类检查

- `domain/` 是否保存业务和领域知识，而不是代码实现细节。
- `architecture/` 是否表达跨模块结构和契约，而不是逐文件复述。
- `code/` 的 `source_path` 是否与真实源码相对路径一致。
- `decisions/` 是否只记录重要且不能从当前事实直接推导的选择。
- `workflows/` 是否覆盖参与概念、顺序、关键分支、失败和验证。

## 代码概念检查

- 能否据此理解文件承担的可验证能力。
- 是否记录容易被破坏的规则、边界和兼容约束。
- 强关联文件是否说明原因并在必要时反向链接。
- 是否能选择相关测试和修改后验证。
- 源码删除或重命名后，条目和引用是否同步处理。

## 过程文档沉淀检查

- 是否只提取长期结论，没有复制完整入口、调研或审查报告。
- 是否链接原始来源，并在可修改时由来源链接回来。
- 同一结论是否先搜索并更新已有概念。
- 来源与当前项目冲突时是否暂停写入，而不是选择方便的版本。

## 处理顺序

1. 修复会导致错误读取的格式、断链和错误路径。
2. 修复与当前项目冲突的事实。
3. 合并重复概念并更新索引。
4. 补充当前工作需要的证据、边界和测试映射。
5. 不为提高表面覆盖率批量创建低价值条目。
