# 技术方案细则

## 文档和支撑材料

支撑材料放在入口旁的同名目录中，按需创建，不预建空目录或空文件。

```text
workplace/3/tech-design/2026-06-21-order-import.md
workplace/3/tech-design/2026-06-21-order-import/
├── research/RT-001-api-boundary.md
├── evidence/EVD-001-current-flow.md
└── review/REV-001-tech-design-review.md
```

编号使用类型加递增数字：`RT-*` 调研、`EVD-*` 证据汇总、`REV-*` 审查。项目证据能直接链接真实代码或已有文档时，不复制证据文件；只有多个来源需要汇总结论时才创建 `EVD-*`。

入口在相关方案摘要、`D-*`、`C-*` 或 `V-*` 下链接支撑材料；每份 Markdown 支撑材料开头链接回入口的对应标题或编号。

## 调研

证据不足以支撑当前方案时创建 `research/RT-*`，写清单一问题、范围、不调查内容、所需证据、完成条件和受影响条目，然后调用 `project-research-v3`。入口只保留任务和结果链接，格式：

```markdown
#### RT-001 <调研问题短标题>

- 调研任务：[RT-001 <调研问题短标题>](<research/RT-001-slug.md>)
- 调研结果：pending / [completed](<research/RT-001-slug.md>) / [blocked](<research/RT-001-slug.md>)
```

报告回写后，责任阶段必须把结论吸收进方案内容。不能用调研报告替代方案结论；`blocked` 时不得确认方案。

常见触发：新增依赖或外部服务、新模块边界、复用或替换现有能力、数据迁移、兼容性、安全或性能关键假设、无法从代码或文档确认的框架能力。

## 内容规则

方案摘要用三段以内讲清核心机制：请求或数据如何流转，谁拥有事实，关键状态如何变化，失败在哪里隔离，用户结果如何被观察。

`D-*` 只在存在真实取舍时创建。每条写选择、主要替代、理由和影响；不写通用技术原则。

`C-*` 必须包含分类、类型、真实文件或模块、当前状态、目标修改、关联 `F-*`、具体验收要求和 `V-*`。禁止使用“优化”“完善”“适配”等无落点描述。尚不存在的路径必须明确标注“新增”。

`V-*` 必须包含要证明的行为、真实可执行的验证方式和可观察证据。多模块或前后端协作时，至少一条 `V-*` 覆盖从真实入口到结果的闭环。

## 覆盖检查

每轮讨论后检查：

- 每个 `F-*` 和验收要求都有 `C-*` 与 `V-*` 落点。
- 新增与修改位置都是真实项目路径或显式“新增”。
- 关键取舍有依据，且没有未解释的替代方案。
- API、数据、UI、失败模型互相一致。
- 用户特别关注点和不做范围没有丢失。

## 回退

执行、任务或验收回退并证明方案需要改变时，把原入口 `status` 改回 `discussing`、`review` 重置为 `pending`，并列出失效的下游计划、执行和验收确认。修订后重新审查和用户确认，不能复用原确认。
