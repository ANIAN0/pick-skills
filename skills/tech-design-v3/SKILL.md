# 技术方案 v3

目标是把已确认需求转成可拆任务的技术契约。方案文档引用需求编号和验收要求，不重写需求正文。

## 默认流程

1. 确认入口路径：只在 `workplace/<版本>/tech-design/` 下创建或更新一份方案入口。优先使用用户路径，其次复用已有相关入口，否则按 `YYYY-MM-DD-功能名.md` 命名。输出完整路径并继续。
2. 校验输入：只接受 `status: confirmed` 的需求入口。读取需求三项核心内容、UI 需求链接、支撑材料和审查结论；缺少确认需求时返回 `requirements-workshop-v3`。
3. 调查真实项目：检查相关代码、配置、依赖、接口、数据、页面、测试、部署和项目知识，确认当前能力、可复用内容、真实差距和未知项。
4. 写入或更新入口：新建时使用 [技术方案模板](assets/tech-design.md)。入口只保留方案摘要、技术决策、变更清单、验证方案；证据、调研和审查报告放同名支撑目录。
5. 讨论缺口：只问会改变主要实现路径、数据归属、兼容、安全、用户体验、范围或验证可行性的问题。相关技术问题一次成组提问，通常 2-5 个，并给推荐方案和影响。
6. 自检、独立审查、用户确认：自检通过后调用 `project-development-review-v3`。审查通过后请用户确认四项方案内容，全部确认后把 `status` 改为 `confirmed`。

## 方案内容

- 方案摘要：说明核心实现机制、事实归属、关键状态、失败隔离和用户结果如何被观察。不要复述需求背景。
- 技术决策 `D-*`：只记录真实取舍。没有取舍写“无”，不要创建占位 `D-*`。
- 变更清单 `C-*`：给任务阶段的契约，包含真实位置或显式“新增”、当前状态、目标修改、关联 `F-*` / 验收 / `V-*`。
- 验证方案 `V-*`：引用需求验收要求，但不复述验收；写真实可执行的方法、命令或操作，以及可观察证据。

每个 `F-*` 和验收要求都必须有 `C-*` 与 `V-*` 落点。用户特别关注的限制、风险、兼容、UI 和不做范围必须落入四项内容之一，不新增第五项。

## 下沉材料

按需读取 [方案细则](references/tech-design-rules.md)：

- 需要创建支撑材料、调研项或证据汇总时。
- 方案涉及 API、数据归属、权限、迁移、兼容、异步、AI/Agent、UI 实现或外部依赖时。
- 需要判断 `D-*`、`C-*`、`V-*` 写法和覆盖是否合格时。
- 下游回退导致已确认方案需要重开时。

## 完成条件

- 入口路径真实，位于 `workplace/`，且一个事项只有一份方案入口。
- 方案基于真实项目证据，必要调研已完成并回写。
- `F-* -> C-* -> V-*` 覆盖完整，`D-*` / `C-*` / `V-*` 互相一致。
- 每个 `C-*` 足以直接拆任务，每个 `V-*` 真实可执行并可观察。
- 独立审查通过，用户明确确认方案内容。

任一项不成立时保持 `status: discussing`。

## 边界

- 不修改生产代码、迁移、测试或项目配置。
- 不重新定义已确认需求；方案无法满足需求时返回需求阶段。
- 不自动批准调研任务、技术取舍、审查结果或阶段完成。
