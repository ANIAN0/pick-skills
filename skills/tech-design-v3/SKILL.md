# 技术设计 v3

只设计当前请求对应的责任子图。产出可被计划阶段直接消费的小型技术决策和变更契约节点，不创建阶段级长篇技术文档。

## 必读内容

- 读取公共 Schema、关系、生命周期、确认和证据协议。
- 消费上游节点或创建设计前，读取 [设计工作流](references/design-workflow.md)。
- 记录实施 Delta 前，读取 [变更契约](references/change-contract.md)。
- 框架、依赖、标准、代码库、安全或运维事实不明确时，调用 `project-research-v3`。

## 工作流

1. 校验图谱，选择已确认的责任需求、能力和验证节点。
2. 重新计算上游语义哈希；有效确认直接复用，不重复询问。
3. 检查实际代码、配置和权威技术来源。
4. 每个独立取舍创建一个技术决策。
5. 每个可独立计划的 Delta 创建一个变更契约，并通过已注册关系绑定定义和验证 ID。
6. 运行图校验和局部影响分析，把就绪契约交给 `implementation-planning-v3`。

## 硬规则

- 上游哈希不匹配时停止；聚焦重新确认前，相关技术节点视为 stale。
- 记录决策使用的代码路径和来源证据，不声称未经证明的 API、字段、表、事件、包或运行时行为。
- 技术事实未知时从责任节点创建调研任务，保持决策 blocked，不编造临时结论。
- 变更契约必须自包含并精确到字段；模块名或“实现功能”不构成 Delta。
- 变更契约以 `implements` 关联需求/能力，以 `depends-on` 关联验证，以 `uses-knowledge` 关联复用报告。
- 不写生产代码、不创建任务节点、不复制完整需求树。

## 输出

报告新增或更新的技术决策与变更契约 ID、输入哈希、调研阻塞、受影响节点，以及可进入计划阶段的契约。
