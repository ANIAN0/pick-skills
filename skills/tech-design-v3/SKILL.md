# 技术方案 v3

从已确认需求开始，先理解真实项目和需要调查的问题，再逐步形成方案。未完成独立审查和用户确认前，不进入任务阶段。

## 文档

一个研发事项只能有一份技术方案入口文档。

- 用户给出方案路径时使用该文件。
- 当前事项已有方案入口时继续更新原文件。
- 两者都没有时询问保存位置，不猜测版本目录。

创建新文档时使用 [技术方案入口模板](assets/tech-design.md)。入口只保留用户会判断的四项：方案摘要、技术决策、变更清单、验证方案。状态只使用：

- `discussing`：仍在调查、讨论、修改或审查。
- `confirmed`：独立审查通过且用户明确确认。

入口旁同名目录按需保存 `research/RT-*`、`evidence/EVD-*`、`decisions/DEC-*` 和 `review/REV-*`。入口在相关 `D-*`、`C-*` 或 `V-*` 链接材料，材料链接回入口对应条目。不预建空目录，不复制可以直接链接的代码或已有报告。

## 输入门槛

只接受状态为 `confirmed` 的需求入口。开始前读取需求四项、相关支撑材料和审查结论，确认每个 `F-*` 及其验收要求；缺少已确认需求时返回 `requirements-workshop-v3`，不自行补写需求。

## 方案调查

先检查与需求直接相关的代码、配置、依赖、接口、数据、页面、测试、部署和项目知识。说明当前能力、可复用部分、差距和未知项，不能只把功能点翻译成模块、接口或数据表。

遇到下列情况且现有证据不足时，先登记技术调研任务：新依赖或外部服务、新模块边界、现有能力复用或替换、数据迁移、兼容性、安全或性能关键假设、无法确认的框架能力。

每个 `RT-*` 写在受影响方案内容下，包含单一问题、范围、不调查内容、所需证据、完成条件和受影响条目。用户审核后调用 `project-research-v3`；报告回写并消除疑问后才继续确定相关方案。

## 深入设计

对中大型方案先用自己的话说明核心实现机制：请求或数据如何流转，谁拥有事实，关键状态如何变化，失败在哪里隔离，用户结果如何被观察。根据实际需求检查：

- 服务、模块和进程边界是否有必要，责任和禁止承担的责任是否清楚。
- API、事件、数据结构和状态是否覆盖需求中的关键对象、动作、错误和来源追踪。
- 现有数据由谁拥有，新增写入、读取、迁移、回滚和兼容如何处理。
- UI 页面、状态、反馈和真实 API 是否形成闭环；无 UI 变更时明确说明。
- 异步、Agent、外部工具或后台任务的超时、重试、幂等、权限和失败影响是否明确。
- 新增与修改位置是否使用真实项目路径；尚不存在的位置明确标注“新增”。
- 验证是否能证明成功、关键失败、边界、集成、持久化和用户路径，而不只是 build 或 lint。

只有存在真实取舍时才创建 `D-*`。每个决策写选择、主要替代方案、理由和影响，不写通用技术原则。

每个 `C-*` 是可交给任务阶段的变更契约，写明分类、类型、真实文件或模块、当前状态、目标状态、具体修改以及关联的 `F-*` 和 `V-*`。不能使用“优化”“完善”“适配”等没有落点的描述。

每个 `V-*` 写清要证明的行为、验证方式和可观察证据。多个模块或前后端协作时，至少包含一条从真实入口到结果的闭环验证。

## 逐项讨论与提问

按决策依赖依次讨论方案摘要、技术决策、变更清单和验证方案。每解决一个问题立即更新同一入口或支撑材料；已经由需求、项目证据或调研解决的问题不重复询问。

只有答案会改变主要实现路径、数据归属、兼容、安全、用户体验、范围或验证可行性，且无法自行调查、安全假设会造成明显风险时才提问。能安全假设时写明依据和影响。

必须提问时，一次只问当前最关键的问题，给出推荐方案和选择影响；放在回复最后的 `## 需要你确认` 区块，后面不追加其他内容。

## 审查与确认

执行、验收或规划回退并证明方案需要改变时，先把原入口 `status` 改回 `discussing`、把 `review` 重置为 `pending`，并明确列出失效的下游计划、执行和验收确认。修订后重新审查和请求用户确认，原确认不能复用。

四项内容形成后先自检并直接修正，直到：所有 `F-*` 有方案落点、所有变更有真实位置、关键取舍有依据、API/数据/UI/失败模型一致、所有验收要求有 `V-*`、没有未处理调研任务或模糊描述。

然后调用 `project-development-review-v3`。审查发现必须回到当前方案修正并复审。审查通过后，请用户确认四项内容；全部确认后把状态改为 `confirmed`，交给 `implementation-planning-v3`。

## 完成条件

- 需求中的每个 `F-*` 和验收要求都有明确方案与验证落点。
- 方案基于真实项目证据，必要技术调研已经完成并双向链接。
- 技术决策、变更清单和验证方案互相一致，没有未解释取舍或占位路径。
- 每个 `C-*` 足以直接拆任务，每个 `V-*` 可真实执行和观察。
- 独立审查通过，用户明确确认方案摘要、技术决策、变更清单和验证方案。

向用户报告方案完成前重新对照本节自检。任一项不成立时保持 `discussing`。

## 边界

- 不修改生产代码、迁移、测试或项目配置。
- 不重新定义已确认需求；方案无法满足需求时返回需求阶段讨论。
- 不为单个决策、变更或验证创建入口文档。
- 不自动批准调研任务、技术取舍、审查结果或阶段完成。
